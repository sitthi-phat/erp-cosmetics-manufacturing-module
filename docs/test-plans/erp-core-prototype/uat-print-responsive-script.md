- **slug**: `erp-core-prototype`
- **สำหรับ**: ปอนด์ (ใช้ตอน Gate 2 verify — ส่วนที่ automate ไม่ได้ตาม tasks.md Q11)
- **เขียนโดย**: QA — Gate 2 Round 2 (เขียนคู่ขนานกับ Engineer, ก่อนโค้ดเสร็จ)
- **อ้างอิง**: ECP-042 AC1 (print fidelity), ECP-043 (UI consistency), ECP-044 (responsive),
  pond-gate2-feedback.md (ตัวอย่างเอกสารที่แนบมา), ADR-009

## ทำไมต้องทำ manual (ไม่ automate)

3 เรื่องนี้เป็นการตัดสินใจเชิงสายตา/ความรู้สึกของมนุษย์ที่ automated assertion ไม่สามารถแทนได้จริง:
1. **Print fidelity** (ECP-042 AC1): "ตรงกับตัวอย่างที่แนบมา 100%" เป็นการเทียบด้วยสายตาของปอนด์เอง
   กับเอกสารตัวอย่างจริง (font, การจัดวาง, ระยะห่าง) — QA เขียน automated test ยืนยันได้แค่ว่า
   "ข้อความ/ตัวเลขที่ควรมีครบทุกส่วนปรากฏอยู่จริง" (ดู `tests/integration/invoiceDocument.spec.ts`)
   แต่ไม่สามารถยืนยัน "หน้าตาเหมือนตัวอย่างเป๊ะ" ได้ด้วยโค้ด
2. **UI consistency** (ECP-043): "ดูเป็นมืออาชีพขึ้น" เป็นการประเมินเชิงสุนทรียะ ไม่มีตัวเลขที่ assert ได้ตรงๆ
3. **Responsive บนอุปกรณ์จริง**: Playwright จำลอง viewport ขนาดได้ (ดู `tests/e2e/responsiveGate2.spec.ts`
   ซึ่งเช็ค "ไม่มี horizontal scroll/ปุ่มหาย" แล้ว) แต่ "กดด้วยนิ้วบนแท็บเล็ตจริงสะดวกจริงไหม" ต้องใช้มือจริงกด

## สิ่งที่ต้องเตรียมก่อนเริ่ม
- [ ] Backend + Frontend รันอยู่ (`npm run dev` หรือเทียบเท่า), DB reseed แล้ว (`npm run db:seed`)
- [ ] เอกสารตัวอย่างใบแจ้งหนี้/ใบกำกับภาษีที่ปอนด์เคยแนบไว้ (พิมพ์ออกมาหรือเปิดคู่กันบนจอ)
- [ ] อุปกรณ์: คอมพิวเตอร์จอปกติ (desktop), แท็บเล็ตหรือจำลองด้วย browser resize เป็น 768-1024px
- [ ] Login เป็น Admin (สำหรับตั้งค่า CompanyProfile), Finance (สำหรับออก/พิมพ์ invoice)

## ส่วนที่ 1 — Print Fidelity (ECP-042 AC1, สำคัญที่สุด)

### ขั้นตอน
1. Login เป็น Admin → ไปหน้า "ตั้งค่าข้อมูลบริษัท" → กรอกชื่อบริษัท/ที่อยู่/เลขผู้เสียภาษี 13 หลัก/โทรศัพท์/โลโก้
   ให้ครบ แล้วบันทึก
2. Login เป็น Sales/CS → สร้างลูกค้าใหม่ที่มีเลขประจำตัวผู้เสียภาษีครบ (13 หลัก) + ที่อยู่จดทะเบียน
3. สร้าง PO → ยืนยัน → ให้ผลิต → QC ผ่าน → จัดส่ง (หรือใช้ PO สาธิตที่ seed มาให้แล้วถ้ามี)
4. Login เป็น Finance → ออก invoice จาก PO นั้น โดยใส่ส่วนลด (discount) เป็นตัวเลขที่ไม่ใช่ 0 (เช่น 2,000 บาท)
5. เปิดหน้ารายละเอียด invoice → กด "พิมพ์/ดาวน์โหลด"

### สิ่งที่ต้องตรวจ (เทียบกับเอกสารตัวอย่างที่แนบ ทีละส่วน)
- [ ] หัวเอกสาร: มีข้อความ "ใบแจ้งหนี้ / ใบกำกับภาษี | Invoice / Tax Invoice" + มีพื้นที่โลโก้จริง
- [ ] ข้อมูลผู้ออก: ชื่อบริษัท, ที่อยู่เต็ม, เลขผู้เสียภาษี 13 หลัก, โทรศัพท์ — ตรงกับที่ตั้งค่าไว้ในข้อ 1
- [ ] ข้อมูลลูกค้า: ชื่อบริษัท, ที่อยู่, เลขผู้เสียภาษี, โทรศัพท์ — ตรงกับข้อ 2
- [ ] เลขที่เอกสาร + วันที่ (รูปแบบ dd/mm/yy) แสดงถูกต้อง
- [ ] เงื่อนไขชำระ/เครดิต แสดงอยู่
- [ ] ตารางรายการ: คอลัมน์ ลำดับ | รายการ | จำนวน | ราคา | ราคารวม ครบทุกบรรทัด อ่านง่าย
- [ ] ส่วนสรุปยอด: รวมเป็นเงิน (subtotal) / หักส่วนลด / จำนวนเงินหลังหักส่วนลด / VAT 7% / จำนวนเงินทั้งสิ้น
      — ตัวเลขคำนวณถูกต้องตามที่กรอก (ลองคำนวณมือเทียบดูสัก 1 ใบ)
- [ ] **ตัวหนังสือไทยของยอดเงิน** ปรากฏอยู่ และอ่านถูกต้องตามหลักภาษาไทย (เช่น ลองใบที่ยอด 51,360.00 บาท
      ต้องอ่านว่า "ห้าหมื่นหนึ่งพันสามร้อยหกสิบบาทถ้วน")
- [ ] ช่องลายเซ็น 2 ช่อง: "ผู้รับใบแจ้งหนี้" และ "ผู้ออกใบแจ้งหนี้" พร้อมวงเล็บชื่อ — มีครบทั้งคู่
- [ ] มีพื้นที่สำหรับตรายาง
- [ ] **ภาพรวม**: หน้าตา/การจัดวาง/ระยะห่าง "ตรงกับตัวอย่างที่แนบมา" ในระดับที่ยอมรับได้หรือไม่
      (ผลการตัดสินเป็นของปอนด์ 100% — QA เขียนโค้ดยืนยันได้แค่ว่าข้อมูลครบ ไม่ได้ยืนยันเรื่องความสวยงาม)

### เคสเสริมที่ควรลองด้วย
- [ ] ออก invoice โดยไม่ใส่ส่วนลดเลย (default 0) → เอกสารต้องยังแสดงบรรทัด "หักส่วนลด: 0.00 บาท" อยู่
      (ไม่ซ่อนบรรทัดนี้ไปเฉยๆ)
- [ ] แก้ไข (revise) invoice ที่เคยพิมพ์ไปแล้ว 1 ครั้ง แล้วลองพิมพ์ version เก่า (Superseded) อีกครั้ง
      → ต้องเห็นลายน้ำ/ป้าย "ยกเลิกแล้ว — ใช้ version ล่าสุดแทน" ชัดเจน
- [ ] ลองพิมพ์ invoice ของลูกค้าที่ไม่มีเลขผู้เสียภาษี → ต้องถูกบล็อกพร้อมข้อความแจ้งให้ไปแก้ข้อมูลลูกค้าก่อน
- [ ] หลังพิมพ์ไปแล้ว 1 ใบ ลองกลับไปแก้ที่อยู่บริษัทที่หน้า Admin แล้วเปิดพิมพ์ invoice ใบเดิม (ใบเก่า) อีกครั้ง
      → ที่อยู่ที่แสดงต้องเป็นที่อยู่ ณ ตอนที่พิมพ์ครั้งแรก ไม่ใช่ที่อยู่ใหม่ (สำคัญมาก - เป็นกฎ snapshot)

### ผลการตรวจ (ปอนด์กรอก)
- ผ่าน 100% ตรงตัวอย่าง: ☐ ใช่ ☐ ไม่ (ถ้าไม่ ระบุจุดที่ต่าง): ______________________

## ส่วนที่ 2 — Responsive บนอุปกรณ์จริง (ECP-044)

### ขั้นตอน (ทำบน 2 อุปกรณ์: คอมพิวเตอร์ปกติ + แท็บเล็ตจริงถ้ามี ไม่มีก็ resize browser แทน)
1. เปิดหน้า "บันทึกการผลิต" (Production) และหน้า "ตรวจรับวัตถุดิบขาเข้า" (QC incoming) บนแท็บเล็ต
   ทั้งแนวตั้ง (portrait) และแนวนอน (landscape)
2. ลองกรอกฟอร์มจริงด้วยนิ้ว (ไม่ใช่เมาส์) — เลือก lot, กรอกจำนวน, กดยืนยัน

### สิ่งที่ต้องตรวจ
- [ ] ปุ่ม/ช่องกรอกมีขนาดใหญ่พอกดด้วยนิ้วได้จริง ไม่เล็กจนพลาดบ่อย
- [ ] ไม่ต้อง scroll แนวนอนเพื่อดูเนื้อหา (เลื่อนขึ้นลงได้ปกติ)
- [ ] ลองกรอกฟอร์มค้างไว้ครึ่งหนึ่ง แล้วหมุนจอจาก portrait เป็น landscape กลางคัน
      → ข้อมูลที่กรอกไว้ต้องไม่หายไป
- [ ] ทำ task หลักของหน้านั้น (เช่น บันทึกผลิตสำเร็จ) จบได้ครบเหมือนบน desktop ไม่มีปุ่มสำคัญหายไป

### ผลการตรวจ (ปอนด์กรอก)
- ใช้งานสะดวกบนแท็บเล็ตจริง: ☐ ใช่ ☐ ไม่ (ถ้าไม่ ระบุปัญหา): ______________________

## ส่วนที่ 3 — UI Consistency ภาพรวม (ECP-043, เชิงสุนทรียะ)

### ขั้นตอน
เปิด 8 หน้าที่ถูกแก้ไข/สร้างใหม่รอบนี้เรียงกัน (เปิดแต่ละหน้าไล่ดู ไม่ต้องพร้อมกัน):
PO create, PO detail, stock, trace, production, QC incoming, BOM management, ฟอร์มลูกค้า,
invoice detail+print

### สิ่งที่ต้องตรวจ
- [ ] ระยะห่าง (spacing) ระหว่าง element ดูสม่ำเสมอกันทั้ง 8 หน้า ไม่มีหน้าไหนแน่น/หลวมผิดปกติ
- [ ] ปุ่มหลัก (primary) กับปุ่มรอง (secondary) ใช้สีเดียวกันสม่ำเสมอทุกหน้า
- [ ] หน้าที่ไม่มีข้อมูล (empty state) หน้าตาเหมือนกันทุกหน้า (ไอคอน+ข้อความ+ปุ่ม ถ้ามี)
- [ ] ข้อความ error ที่เกิดขึ้น (ถ้าลองทำให้ error) อยู่ตำแหน่ง/สี/ไอคอนเดียวกันทุกหน้า
- [ ] ภาพรวม "ดูเป็นมืออาชีพขึ้นชัดเจน" เมื่อเทียบกับตอนทดสอบรอบแรก (feedback ข้อ 7)

### ผลการตรวจ (ปอนด์กรอก)
- ดูเป็นมืออาชีพขึ้นชัดเจน: ☐ ใช่ ☐ ไม่ (ถ้าไม่ ระบุจุดที่ยังรู้สึกไม่โอเค): ______________________
