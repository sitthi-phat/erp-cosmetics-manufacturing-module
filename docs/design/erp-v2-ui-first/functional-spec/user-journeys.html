User Journeys — 6 สายชีวิตจริงของคนทำงาน

สะพานให้ปอนด์อ่านว่า requirement (stories/AC) ตรงกับงานจริงไหม · เล่าเป็นเรื่องต่อเนื่องตั้งแต่ต้นจนจบ ใช้ dataset เดิม (เดอร์มา แคร์ PO-181, บิวตี้ เอเชีย PO-176 ฯลฯ) · source of truth = entity-status-map.md (ไม่มีการสร้างกติกาใหม่)

อ่านยังไง: แต่ละสายมี (1) persona + เป้าหมาย (2) เส้นเดินหลัก ทีละขั้น: ทำอะไร → หน้าไหน (ลิงก์ mockup) → ระบบตอบ → สถานะเปลี่ยน → ใครได้ noti (3) ทางแยกสำคัญ (4) จุดเชื่อม journey อื่น (5) ตารางอ้างอิง step ↔ story ID ให้ BA cross-review

ภาพร้อย 6 สายเข้าด้วยกัน:
① Customer (ลูกค้า→ยืนยันออเดอร์) ─▶ ② PO (ยืนยัน→คิวผลิต) ─▶ ③ การผลิต (รับงาน→Batch→ส่งตรวจ) ─▶ ④ QC (ผ่าน/ตก) ─▶ ⑤ จัดส่ง (รอบ+DN→ส่งถึง) ─▶ ⑥ Billing (วางบิล→เก็บเงิน)
วนกลับ: QC ตก→③ · DN ปฏิเสธ→⑤/① · Hold→②/① · Overdue→①

① Journey: Customer — ลูกค้า & งานขาย

Persona: สมชาย พนักงานขาย/CS — รับลูกค้าใหม่ทางโทรศัพท์ ดูแลลูกค้าประจำ และติดตามงานที่ค้างที่ลูกค้า · เป้าหมาย: เปลี่ยนผู้สนใจให้เป็นลูกค้าประจำ และไม่ปล่อยลูกค้าที่ต้องติดตามให้หลุด

เส้นเดินหลัก (happy path)

  1. ลูกค้าใหม่ "ร้านสวยใส" โทรมา → สมชายสร้างลูกค้าที่ customer-create (ชื่อ + เลขผู้เสียภาษี 13 หลัก + ที่อยู่) → ระบบสร้าง CUS-xxxxxx ผู้สนใจ (Lead)
  2. เพิ่มผู้ติดต่อหลายคนที่ contact-create + ตั้งผู้ติดต่อหลัก (ไม่จำกัดจำนวน)
  3. สมชายเปิด PO ใบแรกให้ (ไป Journey ② PO) → เมื่อ ยืนยัน PO ใบแรก ระบบเลื่อนลูกค้า Lead → ลูกค้าประจำ (Active) อัตโนมัติ (C1) → เห็นใน dashboard tile "ลูกค้าประจำ"
  4. ดูแลต่อ: ลูกค้ามี order สม่ำเสมอ = คงสถานะ Active · customer-detail เห็น timeline + ประวัติ PO

ทางแยกสำคัญ

งานลูกค้าติด Hold (จาก Journey ③): เดอร์มา แคร์ ขอแก้สูตร PO-188 → ฝ่ายผลิต Hold + raise Sale → สมชายตั้งลูกค้า ต้องติดตาม (Follow-up) + comment บังคับ "โทรยืนยันแก้สูตร PO-188" (C8) → โผล่ tile "ต้องติดตาม" ใน Sale dashboard + Home task-inbox → จบเรื่อง/มี order ใหม่ = กลับ Active
ลูกค้าเงียบหาย: ไม่มี order เกินรอบ (default 3 เดือน) → scheduler เลื่อนเป็น ห่างหาย (Inactive) อัตโนมัติ · ความเสี่ยง: เช็คเด้ง 2 ครั้ง → Sale Manager (สมหญิง) ตั้ง บัญชีดำ (Blacklist) + comment → ลูกค้าหายจาก dropdown เปิด PO ใหม่ · ปลด Blacklist = Admin เท่านั้น · reassign ผู้ดูแล = Sale Manager/Admin (2 dashboard + trace)

จุดเชื่อม journey อื่น

▶ ยืนยัน PO ใบแรก = จุดเริ่ม Journey ② (และทำ Lead→Active) · ◀ Follow-up ถูกจุดชนวนจาก Journey ③ (Hold) และ ⑤ (DN ปฏิเสธ→raise Sale) และ ⑥ (Overdue→Sale ตาม)

อ้างอิง step ↔ story

step/ทางแยกstory
สร้างลูกค้า + contact + เลขภาษีUS-CUS-01
Lead→Active (ยืนยันใบแรก) / Active↔Inactive (scheduler)US-CUS-02 + US-PO-02
ตั้ง Follow-up + comment บังคับUS-CUS-03
Blacklist/Disabled + reassign + ปลด=AdminUS-CUS-04
งานลูกค้าค้างโผล่หน้าแรกUS-HOME-01

② Journey: PO — ใบสั่งซื้อ

Persona: อารดา พนักงานขาย — รับออเดอร์เดอร์มา แคร์ สั่ง 2 สินค้า (โลชั่น + โฟม) · เป้าหมาย: เปิดออเดอร์ให้เร็ว ไม่ติดข้อจำกัดราคา และส่งงานเข้าสายผลิตครบถ้วน

เส้นเดินหลัก (happy path)

  1. ที่ po-create เลือกลูกค้า "เดอร์มา แคร์" + เพิ่ม 2 บรรทัด (โลชั่น BOM, โฟม BOM) → ระบบ suggest ราคาขาย default (จาก BOM) + เช็ควัตถุดิบพอไหม (ΣBOM×จำนวน เทียบ on_hand) → แก้ราคาได้ (0 ได้)
  2. กด "บันทึกร่าง" → ระบบออกเลข PO-202607-000181 ร่าง (Draft) · ราง billing ยังไม่วางบิล
  3. กด "ยืนยัน PO" ที่ po-detail → ยืนยันแล้ว (Confirmed) → แต่ละ line เข้าคิวฝ่ายผลิต "รอรับงาน" (ยังไม่เกิดเลข PRD) (C1) → noti ฝ่ายผลิต · ลูกค้า Lead→Active (ถ้าใบแรก)

ทางแยกสำคัญ

วัตถุดิบขาดตอนเปิด PO: เตือน "วัตถุดิบอาจไม่พอ" แต่ไม่บล็อก → ระบบเสนอสร้าง Purchase Request เฉพาะส่วนที่ขาด (C15) → noti Stock + Production (ไป Journey วัตถุดิบ) · PO เดินหน้าต่อได้
ต้องยกเลิก/แก้: ยกเลิก PO ได้ทุกขั้น (comment บังคับ) → เปิดใหม่ คงเลข PO เดิม (gapless) (C19) · เคส Hold: Sale แก้ PO ได้ทุกช่อง (จำนวน/ราคา/วันส่ง) + trace ราย field แล้วส่งกลับผลิต

จุดเชื่อม journey อื่น

◀ เริ่มจาก Journey ① (ลูกค้ายืนยันออเดอร์) · ▶ Confirmed = จุดเริ่ม Journey ③ การผลิต (คิวรอรับงาน) · ▶ วัตถุดิบขาด = แตกไป PR/GR (เส้นวัตถุดิบ) · ▶ ราง billing แยกไป Journey ⑥

อ้างอิง step ↔ story

step/ทางแยกstory
สร้างร่าง + ราคา 0 ได้US-PO-01
suggest ราคา + เช็ควัตถุดิบ (ΣBOM×qty vs on_hand)US-PO-07
ยืนยัน → คิว "รอรับงาน" + Lead→ActiveUS-PO-02
วัตถุดิบขาด → auto PR ส่วนที่ขาดUS-PR-01
ยกเลิก + เปิดใหม่คงเลขUS-PO-04
แก้ PO เคส Hold + trace ราย fieldUS-PO-05

③ Journey: การผลิต — PRD & Batch

Persona: วิชัย หัวหน้าฝ่ายผลิต — รับงานจากคิว จัดลำดับ ผลิตจริง และส่งตรวจคุณภาพ · เป้าหมาย: ผลิตให้ทันกำหนดส่ง โดยสายผลิตไม่สะดุดแม้วัตถุดิบยังไม่เข้าระบบ

เส้นเดินหลัก (happy path)

  1. ที่ production แท็บ "รอรับงาน" เห็น PO-181 มี 2 line (ยังไม่มีเลข PRD) → วิชัยกด "รับงาน" line1 (โลชั่น) → ระบบออกเลข PRD-202607-000091 รับงาน (C2)
  2. กด "เริ่มผลิต" → ระบบ gen B-PO-202607-000181-1-1 กำลังผลิต + ตัด stock วัตถุดิบตาม BOM แบบ FIFO → PO กำลังผลิต (C3)
  3. ผลิตเสร็จ กด "ส่งตรวจ QC" → Batch เดิมเปลี่ยนเป็น รอ QC (ไม่ gen ใหม่) → โผล่คิว QC (C4) → noti QC
  4. รอผล QC (Journey ④) → ถ้าผ่าน PRD line1 พร้อมส่งมอบ · เมื่อ ทุก PRD ของ PO ผ่าน → PO พร้อมจัดส่ง (C5)

ทางแยกสำคัญ

วัตถุดิบไม่พอ (แอลกอฮอล์ PO-185): กดรับงาน/เริ่มผลิตได้ ไม่บล็อก → เตือน "จะตัด stock ติดลบ ~30 หน่วย" → เริ่มผลิตตัด stock จนติดลบ (C16) → stock โชว์เลขแดง + badge "ติดลบ (รอรับเข้า)" → พอทำ Goods Receipt ระบบ บวกกลับ + ผูก Lot ย้อน FIFO (คง GMP) + notice "ชดเชยยอดติดลบ" (C17)
QC ตีกลับ (line2 โฟม): PRD line2 Rework + เห็น feedback จาก QC → กด "ผลิตซ้ำ" → gen B-...-181-2-2 (run ถัดไป) → กลับคิว QC (C6/C7) · line1 เดินต่อไม่สะดุด
ติดปัญหา: กด "พักงาน (Hold)" + เหตุผล (บังคับ) + raise Sale หรือ Stock (C8) → noti Sale/Stock

จุดเชื่อม journey อื่น

◀ เริ่มจาก Journey ② (PO Confirmed) · ▶ ส่งตรวจ = ส่งเข้า Journey ④ QC · ◀ QC ตก = วนกลับมาที่นี่ (rework) · ▶ ทุก PRD ผ่าน = ส่งต่อ Journey ⑤ จัดส่ง · Hold เหตุลูกค้า = จุดชนวน Follow-up Journey ①

อ้างอิง step ↔ story

step/ทางแยกstory
รับงาน → gen PRD (+ เตือนวัตถุดิบอาจไม่พร้อม)US-PRD-01
เริ่มผลิต → gen Batch + ตัด stock ติดลบได้ + FIFOUS-PRD-02 + US-STK-03
ส่งตรวจ QC (Batch เดิมเปลี่ยนสถานะ)US-PRD-03
Rework เฉพาะ line ที่ตก + gen run ถัดไปUS-PRD-04
Hold + raise Sale/StockUS-PRD-05

④ Journey: QC — ตรวจคุณภาพ

Persona: อรุณี เจ้าหน้าที่ QC — ตัดสินคุณภาพ Batch ที่ผลิตเสร็จ และตรวจรับวัตถุดิบ Lot ขาเข้า · เป้าหมาย: ปล่อยเฉพาะของที่ผ่านมาตรฐาน + ให้ฝ่ายผลิตรู้ชัดว่าต้องแก้อะไร (GMP)

เส้นเดินหลัก (happy path)

  1. ที่ qc แท็บ "ตรวจ Batch" เห็น B-PO-202607-000181-1-1 (โลชั่น) รอ QC → ดู GMP chain (Lot→Batch→PO) → กด "✓ ผ่าน" → Batch QC ผ่าน → PRD line1 พร้อมส่งมอบ
  2. Batch B-...-181-2-1 (โฟม) ตรวจเจอปัญหา → กด "✕ ไม่ผ่าน" + feedback บังคับ "เนื้อโฟมเป็นก้อน ปั๊มไม่ออก" → Batch ไม่ผ่าน · PRD line2 Rework (C6) → noti ฝ่ายผลิต (deep link ไป line ที่ต้องแก้)
  3. เมื่อทุก line ของ PO ผ่าน → PO พร้อมจัดส่ง (C5) → noti ฝ่ายจัดส่ง

ทางแยกสำคัญ

ตรวจซ้ำหลัง rework: run2 (B-...-181-2-2) โผล่คิว QC เป็นรายการใหม่ + เห็นประวัติ run1 ไม่ผ่าน + feedback เดิม → ตรวจแล้วผ่าน (เช่น PO-170 run1 fail→run2 pass)
Lot วัตถุดิบขาเข้าไม่ผ่าน: แท็บ "ตรวจรับ Lot" → ตัดสิน "ไม่ผ่าน" → Lot ระงับ → เปิดใบคืนของ supplier (C18 → Journey Return)

จุดเชื่อม journey อื่น

◀ รับงานจาก Journey ③ (ส่งตรวจ) · ▶ ผ่าน = ส่ง PO เข้า Journey ⑤ · ◀▶ ไม่ผ่าน = ตีกลับ Journey ③ (rework loop) · ▶ Lot ตก = Journey Return (คืน supplier)

อ้างอิง step ↔ story

step/ทางแยกstory
ตรวจ Batch ผ่าน → PRD/PO cascadeUS-QC-01
ไม่ผ่าน + feedback บังคับ + ประวัติ runUS-QC-02
ตรวจรับ Lot ขาเข้า ผ่าน/ไม่ผ่าน→คืนUS-QC-03 + US-RET-01

⑤ Journey: จัดส่ง — Shipment & DN

Persona: ก้อง ฝ่ายจัดส่ง — จัดรอบรถ เลือก PO ที่พร้อม และอัปเดตผลการส่งราย DN · เป้าหมาย: ส่งของครบถูกใบ ลูกค้าเซ็นรับ และปิดรอบให้ตรงความจริง

เส้นเดินหลัก (happy path)

  1. ที่ shipping เห็นคิว PO "พร้อมจัดส่ง" → สร้างรอบ SHP-20260708-0044 เลือก PO-176 (บิวตี้ เอเชีย) + กรอกคนขับ/เบอร์/route/รถกระบะ → ระบบ gen DN-20260708-00119 (1 DN = 1 PO) → PO กำลังจัดส่ง (C9)
  2. ที่ delivery-note พิมพ์ DN ทีละใบ → ลูกค้ารับ + เซ็น → กด "ส่งถึงแล้ว" → DN ส่งถึงแล้ว · PO ส่งถึงแล้ว + เริ่มนับเครดิต/overdue (C10) → noti Finance + Sale
  3. รอบจบเมื่อ DN ทุกใบถึงสถานะสุดท้าย → Shipment จบรอบ

ทางแยกสำคัญ (รอบรวมหลาย PO — SHP-0046)

ลูกค้าปฏิเสธ: DN-00123 (PO-175) → กด "ถูกปฏิเสธ" → PO พร้อมจัดส่ง (กลับคิว) + raise Sale (C11) → noti Sale (ไป Journey ①) → รอสร้าง DN รอบใหม่
ลูกค้าขอเลื่อน: DN-00124 (PO-178) → "เลื่อนส่ง" + วันที่ 10/07 → PO พร้อมจัดส่ง + flag "กันจัดส่ง Postpone 10/07" ค้างคิว (C12) → รอบรวม = ส่งบางส่วน (Partially) + breakdown 3 DN

จุดเชื่อม journey อื่น

◀ รับ PO จาก Journey ④ (พร้อมจัดส่ง) · ▶ ส่งถึง = จุดเริ่มนับเครดิต Journey ⑥ · ▶ ปฏิเสธ = raise Sale Journey ① (อาจตั้ง Follow-up)

อ้างอิง step ↔ story

step/ทางแยกstory
สร้างรอบ + DN ราย PO + ข้อมูลรถUS-SHP-01
Delivered / Rejected / PostponedUS-SHP-02
reconcile รอบ (จบรอบ/Partially) + printUS-SHP-03

⑥ Journey: Billing — วางบิล & เก็บเงิน

Persona: นภา ฝ่ายการเงิน — ออกใบแจ้งหนี้/ใบกำกับภาษี และติดตามหนี้ค้าง · เป้าหมาย: วางบิลถูกต้องตามสรรพากร และเก็บเงินตรงกำหนดเครดิต

เส้นเดินหลัก (happy path)

  1. ที่ invoices เห็น PO-176 ส่งถึงแล้ว → กด "ออกใบแจ้งหนี้" → ระบบ gen INV-2026-000135 รอชำระ · ราง billing ของ PO วางบิลแล้ว (C13) · เครดิต 30 วัน (จากระดับลูกค้า)
  2. ที่ invoice-print พิมพ์ใบกำกับภาษีไทย: ผู้ออก + เลขผู้เสียภาษี 13 หลัก, VAT 7% ตาม effective date ณ วันออกใบ (01/01/2569), ตัวหนังสือไทย, ลายเซ็น 2 ช่อง
  3. ลูกค้าจ่ายครบ → บันทึกรับชำระ → ชำระแล้ว (Paid)

ทางแยกสำคัญ

ออกบิลล่วงหน้า: ออกใบได้ตั้งแต่ PO ยืนยันแล้ว แต่ใบต้องโชว์ stage จริงของ PO เสมอ
เลยกำหนด: ส่งของแล้ว + เลยเครดิต (ระดับลูกค้า) + ยังไม่จ่าย → scheduler เลื่อนเป็น เกินกำหนด (Overdue) + จำนวนวันค้าง (C14) → noti Finance + Sale (Sale ไปตาม = Journey ①)
ต้องแก้/ยกเลิกใบ: ใช้ versioning (เก็บประวัติ) และ "ยกเลิก (void)" ไม่ลบ — เลขคงอยู่ (gapless)

จุดเชื่อม journey อื่น

◀ เริ่มนับเครดิตจาก Journey ⑤ (DN Delivered) · ▶ Overdue = ส่ง Sale ตาม Journey ① · อัตรา VAT ตั้งที่ Settings (US-SET-03)

อ้างอิง step ↔ story

step/ทางแยกstory
ออกใบตั้งแต่ Confirmed + โชว์ stage จริงUS-INV-01
ใบกำกับภาษีไทย + VAT ตาม invoice dateUS-INV-02 + US-SET-03
Overdue + ติดตามหนี้US-INV-03
versioning + void (ไม่ลบ/gapless)US-INV-04
