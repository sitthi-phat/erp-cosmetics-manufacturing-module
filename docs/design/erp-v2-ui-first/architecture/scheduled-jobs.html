Stage 2 · Architecture · ตอบคำถามปอนด์

งานที่ระบบทำเองอัตโนมัติเบื้องหลัง

หน้านี้รวบรวม "งานที่ทำงานเองตามเวลา/เบื้องหลัง" ทั้งหมดของระบบ (ผู้ใช้ไม่ต้องกด) — อธิบายว่าแต่ละงานทำงานเมื่อไหร่ ทำอะไรทีละขั้น กระทบข้อมูลอะไร แจ้งเตือนใครไหม และถ้าเครื่องปิด/งานพลาดจะกู้อย่างไร. ท้ายหน้าแยกชัดว่าอะไร "ไม่ใช่งานตั้งเวลา" (เป็นการคำนวณตอนเปิดดู / เป็น event / เป็นฝั่งเบราว์เซอร์). ลิงก์กลับ Architecture.

สรุปรวม — มีงานตั้งเวลา 7 งาน

ระบบมีงานเบื้องหลัง 7 งาน: 1 งานทำถี่ต่อเนื่อง (ส่งแจ้งเตือน), 4 งานทำวันละครั้ง (เตะออก 06:00 / ลูกค้าห่างหาย / invoice เกินกำหนด / เสี่ยงล่าช้า), 1 งานกลางคืน (ตรวจสต็อกให้ตรง), 1 งานโครงสร้างพื้นฐาน (สำรองข้อมูล — DevOps ดูแล). ทุกงานออกแบบให้ "ทำซ้ำได้ปลอดภัย (idempotent)" — ถ้าเครื่องปิดแล้วพลาดรอบ พอเปิดมางานจะไล่เก็บงานที่ค้างให้เอง ไม่เกิดข้อมูลซ้ำ/เพี้ยน.
งานความถี่ทำอะไรสั้น ๆแจ้งเตือน
J1 เตะออก 06:00ทุกวัน 06:00บังคับทุกคน login ใหม่—
J2 ลูกค้าห่างหายทุกวัน (เช้ามืด)ลูกค้าไม่สั่งเกินรอบ → "ห่างหาย"✅ Sale
J3 Invoice เกินกำหนดทุกวัน (เช้ามืด)เลยเครดิต+ยังไม่จ่าย → "เกินกำหนด"✅ Finance+Sale
J4 เสี่ยงล่าช้าทุกวัน (หรือทุกชั่วโมง)งานผลิตใกล้ไม่ทันส่ง → badge เตือน✅ Sale+Stock
J5 ส่งแจ้งเตือน (outbox)ต่อเนื่อง ~1 วินาทีส่ง noti ที่ค้างในคิวไปหาผู้รับ✅ (ตัวส่งเอง)
J6 ตรวจสต็อกให้ตรงทุกคืนเทียบยอดคงเหลือกับบัญชีเดินสะพัด— (เตือน admin ถ้าเจอไม่ตรง)
J7 สำรองข้อมูลทุกวันbackup ฐานข้อมูล (DevOps/Cloud SQL)—
J1 · บังคับ Login ใหม่ทุกเช้า (Session reset)ทุกวัน 06:00
ทำอะไร (ทีละขั้น)เวลา 06:00 ทุกวัน ระบบเลื่อนค่า session_epoch เป็นเวลาปัจจุบัน → token (บัตรผ่าน) ที่ออกก่อนหน้านี้ทั้งหมดใช้ไม่ได้ทันที → ทุกคนต้องล็อกอินใหม่. นอกจากนี้ session ยังหมดอายุเองใน 24 ชม.
กระทบอะไรไม่แก้ข้อมูลธุรกิจ — แค่ทำให้ session หมดอายุ (ความปลอดภัย)
ถ้าเครื่องปิด/พลาดตอนเปิดเครื่องมา ถ้าเลย 06:00 ของวันแล้วแต่ session_epoch ยังเป็นของเมื่อวาน → เลื่อนเป็น 06:00 ของวันนี้ทันที idempotent (ตั้งค่าเป็น timestamp เดิมซ้ำได้ไม่มีผลข้างเคียง)
ผูก story/ACADR-007 · api-platform › session · มี "คำเตือนก่อนหมดอายุ" ให้ผู้ใช้เซฟงานก่อนถูกเตะออก
idempotentไม่มีกะกลางคืน — เตะออก 06:00 คงเดิม (ปอนด์เคาะ)
J2 · ตั้งลูกค้า "ห่างหาย" อัตโนมัติ (Inactivity sweep)ทุกวัน เช้ามืด
ทำอะไร (ทีละขั้น)ไล่ดูลูกค้าสถานะ "ลูกค้าประจำ (Active)" ทุกราย → เช็ควันสั่งซื้อล่าสุด → ถ้าไม่มี order เกินรอบที่ตั้งไว้ต่อลูกค้า (inactive_after_months เลือกได้ {1, 3, 6, 8} เดือน, ค่าเริ่ม 3) → เปลี่ยนเป็น "ห่างหาย (Inactive)" + เขียนโน้ตอัตโนมัติ + แจ้ง Sale
กระทบ entity/สถานะcustomer: Active → Inactive (cascade C2) · แสดงใน Sale Dashboard tile "ห่างหาย" (ชนิด event)
แจ้งเตือน✅ แจ้ง Sale เจ้าของลูกค้า
ถ้าเครื่องปิด/พลาดงานคำนวณจาก "วันสั่งล่าสุดเทียบวันนี้" → รอบถัดไปที่รันจะจับลูกค้าที่ครบเกณฑ์ทั้งหมดได้เอง idempotent (ลูกค้าที่เป็น Inactive อยู่แล้วถูกข้าม) — พลาดรอบ = สถานะช้าไปวันเดียว ไม่เสียหาย
ผูก story/ACentity-status-map §1.1 + cascade C2 · functional-spec/customer · dashboard tile ห่างหาย
ตรงข้าม (Inactive → Active) ไม่ใช่งานตั้งเวลา — เกิดทันทีเป็น event เมื่อมี order ใหม่ (cascade). Follow-up → Inactive ตอนปิดติดตามโดยไม่มี order ก็ทำได้ทั้ง manual และ sweep นี้.
J3 · ติดธง Invoice "เกินกำหนด" (Overdue sweep)ทุกวัน เช้ามืด
ทำอะไร (ทีละขั้น)ไล่ดูใบแจ้งหนี้ที่ยังไม่จ่าย → เงื่อนไข: ส่งของแล้ว (PO=ส่งถึงแล้ว) + เลยจำนวนวันเครดิตของลูกค้า (credit_days) + ยังไม่ชำระ → เปลี่ยนสถานะ billing เป็น "เกินกำหนด (Overdue)" + คำนวณจำนวนวันค้าง
กระทบ entity/สถานะinvoice/PO billing → Overdue (cascade C11/C14) · โชว์ใน Finance Dashboard + Sale (จำนวนวันค้าง)
แจ้งเตือน✅ แจ้ง Finance + Sale
ถ้าเครื่องปิด/พลาดคำนวณจากวันครบกำหนดเทียบวันนี้ → รอบถัดไปจับครบ idempotent (ใบที่ Overdue แล้วข้าม; ถ้าจ่ายแล้วไม่ติดธง)
ผูก story/ACentity-status-map §1.3 + C11/C14 · api-invoice · dashboard tile เกินกำหนด
J4 · ประเมิน "เสี่ยงล่าช้า" (Potential Delay)ทุกวัน (หรือทุกชั่วโมง)
ทำอะไร (ทีละขั้น)ไล่ดูใบสั่งผลิต (PRD) ที่ยังไม่เสร็จ → ถ้าเวลาที่เหลือถึงวันต้องส่ง น้อยกว่าเกณฑ์ (2 วันผลิต + 1 วันจัดส่ง) → ติด overlay badge "เสี่ยงล่าช้า" · เมื่อไม่เข้าเกณฑ์แล้ว → ปลด badge
กระทบ entity/สถานะPRD: potential_delay=true/false (เป็น overlay ไม่ใช่สถานะแยก — cascade C7b) · Production Dashboard tile "เสี่ยงล่าช้า"
แจ้งเตือน✅ แจ้ง Sale + Stock
ถ้าเครื่องปิด/พลาดเป็นการ "ประเมินใหม่จากสถานะปัจจุบัน" ทุกรอบ idempotent — พลาดรอบ = badge อัปเดตช้า พอรันใหม่ก็ถูกต้อง
ผูก story/ACentity-status-map §1.4 + C7b · functional-spec/production · dashboard tile เสี่ยงล่าช้า
J5 · ตัวส่งแจ้งเตือน (Notification Outbox Dispatcher)ต่อเนื่อง ~1 วินาที
ทำอะไร (ทีละขั้น)ทุกครั้งที่มีการเปลี่ยนสถานะข้าม module ระบบเขียน "event" ลงคิว outbox (ในทรานแซกชันเดียวกับการเปลี่ยนสถานะ กันหลุด). งานนี้วนอ่าน outbox แถวที่ยังไม่ส่ง ~ทุก 1 วินาที → สร้าง notification ให้ทุกคนที่มีสิทธิ์ Read ของ module ปลายทาง → mark ว่าส่งแล้ว
กระทบ entityสร้าง notification (bell + deep link) — ไม่แก้ข้อมูลธุรกิจ
แจ้งเตือน✅ เป็นตัวสร้างแจ้งเตือนทั้งหมดของระบบ (14 เหตุการณ์)
ถ้าเครื่องปิด/พลาดจุดแข็งของดีไซน์นี้: event ถูกเก็บถาวรใน outbox ตั้งแต่ตอนเปลี่ยนสถานะ → เครื่องปิดแล้ว event ไม่หาย → เปิดมา dispatcher ส่งต่อได้เลย idempotent (mark "ส่งแล้ว" กันส่งซ้ำ)
ผูก story/ACADR-005 · api-notification (แผนที่ noti 14 เหตุการณ์)
เบราว์เซอร์ฝั่งผู้ใช้ดึง noti มาโชว์ที่ bell ~ทุก 15 วินาที — อันนั้นเป็นการ poll ฝั่งเบราว์เซอร์ ไม่ใช่งาน scheduler ของ server (ดู "ไม่ใช่งานตั้งเวลา").
J6 · ตรวจสต็อกให้ตรง (Stock Ledger Integrity)ทุกคืน
ทำอะไร (ทีละขั้น)เทียบ 2 cache: (1) stock_balance กับ Σ stock_movement (physical/on_hand) และ (2) reserved_balance กับ Σ reservation ที่ active ต่อวัตถุดิบ/lot → ถ้าไม่ตรง (ไม่ควรเกิด) → คำนวณใหม่จากแหล่งจริง + บันทึก log
กระทบ entityแก้เฉพาะ cache stock_balance + reserved_balance ให้ตรงกับความจริง (ledger + reservation เป็นความจริงเสมอ)
แจ้งเตือน— (แจ้ง Admin เฉพาะกรณีเจอไม่ตรง)
ถ้าเครื่องปิด/พลาดเป็นงานบำรุงรักษา — พลาดได้ไม่กระทบผู้ใช้ (ยอดคำนวณจาก ledger ได้เสมอ) idempotent
ผูก story/ACADR-001 (stock ledger) · รองรับ negative stock + retro-alloc
J7 · สำรองข้อมูล (Database Backup)ทุกวัน · DevOps
ทำอะไร (ทีละขั้น)เฟส 2 (PC): DevOps ตั้ง cron ทำ dump ฐานข้อมูล + โฟลเดอร์ไฟล์แนบ (STORAGE_ROOT) รายวัน · เฟส 3 (GCP): ใช้ backup อัตโนมัติของ Cloud SQL ทุกวัน (ปอนด์ยืนยัน NFR)
กระทบ entityไม่แก้ข้อมูล — คัดลอกเพื่อกู้คืน
ถ้าเครื่องปิด/พลาดรอบถัดไป backup ใหม่ idempotent · เป็นงานระดับ infra ไม่ผูกกับ business logic
ผูก story/ACNFR backup (ปอนด์) · เป็นงานของ DevOps (Stage 4) — flag ให้ทีม DevOps
