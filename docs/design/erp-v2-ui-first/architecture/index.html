ภาพรวมสถาปัตยกรรม (Architecture Overview)

หน้านี้คือ "แผนที่ช่างเทคนิค" ของระบบ ESSENCE Hub — อธิบายเป็นภาษาคนว่าระบบประกอบด้วยอะไร ต่อกันอย่างไร ทำงานบนเครื่องอะไร (เฟส 2 = PC ในโรงงาน → เฟส 3 = Google Cloud) และการตัดสินใจสำคัญแต่ละข้อ (ADR) มีเหตุผลอะไร. ทุกกล่องในไดอะแกรมมีคำอธิบายไทยกำกับ. ลิงก์ไป โครงสร้างฐานข้อมูล และ API ราย module ด้านล่าง.

ในหน้านี้

1. ระบบวางบนอะไร (System) 2. บริการ/โมดูล (Service) 3. เส้นทางข้อมูล 1 คำสั่ง 3b. งานตั้งเวลา (scheduled) 4. การตัดสินใจ (ADR ทั้ง 9) 5. API ราย module 6. NFR ที่ปอนด์กำหนด 7. เช็คครบทุก case

1. ระบบวางบนอะไร — System Diagram (เฟส 2 → เฟส 3)

อ่านซ้าย→ขวา: ผู้ใช้เปิดเว็บ (React) → คุยกับ Backend (Node.js/NestJS) → เก็บข้อมูลใน MySQL. ไฟล์แนบ/ไฟล์ archive เก็บผ่าน "ตัวกลางเก็บไฟล์" ที่สลับได้ระหว่างดิสก์ในเครื่อง (เฟส 2) กับ Google Cloud Storage (เฟส 3). โค้ดชุดเดียวกันย้ายขึ้น cloud ได้โดยไม่ต้องเขียนใหม่ — เปลี่ยนแค่ค่า config (หลักการที่ทำให้เส้นทาง PC→GCP ไม่ถูกปิด ตาม charter).

เส้นทึบ = การต่อในเฟส 2 · เส้นประน้ำเงิน = การยกขึ้น GCP ในเฟส 3 (เปลี่ยน environment ไม่เปลี่ยนโค้ดธุรกิจ — ดู ADR-006).

2. บริการ/โมดูลภายใน — Service Diagram

Backend แบ่งเป็น โมดูลตามแผนกงาน (ขาย/คลัง/ผลิต/QC/จัดส่ง/การเงิน/ตั้งค่า) ล้อมรอบด้วย "ชั้นกลาง" 6 ตัว ที่ทำงานให้ทุกโมดูลเหมือนกันอัตโนมัติ — ตรวจสิทธิ์, บันทึก audit, ห่อ response, ยิงแจ้งเตือน, ออกเลขเอกสาร, เก็บไฟล์. นี่คือเหตุผลที่ requirement ยาก ๆ (audit ทุก field, ลบ soft, เลขห้ามข้าม) ทำครั้งเดียวใช้ทั้งระบบ ไม่ต้องเขียนซ้ำรายโมดูล.

3. เส้นทางข้อมูลของ 1 คำสั่ง (เช่น QC กด "ผ่าน")

ตัวอย่างให้เห็นว่า "การกดปุ่มเดียว" ทำอะไรบ้างเบื้องหลัง และทำไมทุกอย่างต้องสำเร็จพร้อมกันหรือไม่สำเร็จทั้งหมด (1 transaction) — กันข้อมูลหลุด/สถานะเพี้ยน.

3b. งานตั้งเวลา / เบื้องหลัง (Scheduled & Background Jobs)

ระบบมีงานที่ทำเองอัตโนมัติ 7 งาน (เตะออก 06:00 · ลูกค้าห่างหาย · invoice เกินกำหนด · เสี่ยงล่าช้า · ตัวส่งแจ้งเตือน · ตรวจสต็อกตรง · สำรองข้อมูล) — ทุกงาน idempotent + มีตาราง scheduled_job_run ไว้ติดตาม/ไล่เก็บงานค้างหลังเปิดเครื่อง. รายละเอียดเต็ม step-by-step + การกู้คืน อยู่ที่หน้าแยก.

→ เปิดดู: งานตั้งเวลาทั้งหมด (scheduled-jobs.html) · แยกชัดว่าอะไร manual (archive trace) / on-read (badge ราคาทุนล้าสมัย) / event (ปิดรอบจัดส่ง) / client (refresh 15 วิ)

4. การตัดสินใจสำคัญ — ADR ทั้ง 9 (อธิบายไทย)

แต่ละ ADR = การตัดสินใจ 1 เรื่อง มีเหตุผล + ทางเลือกที่ทิ้ง + ผลที่ตามมา. คลิกอ่านเต็มได้. เรียงตามความสำคัญ.

ADR-001 · Stock Ledger + สต็อกติดลบ หัวใจ

สต็อกเป็นบัญชีเดินสะพัด (append-only) ยอดคงเหลือคำนวณจากประวัติ. อนุญาตติดลบระหว่างผลิต, GR บวกกลับ+ผูก lot FIFO ย้อนหลัง. แก้บั๊ก DEF-09 ไปในตัว.

ADR-002 · Framework/ORM

NestJS + Prisma + React/Vite + Ant Design + TypeScript. เลือกให้ทีมทำเหมือนกัน + ย้าย GCP ได้ + มี guard/interceptor รองรับ audit/สิทธิ์.

ADR-003 · Audit ระดับ field จุดยาก

บันทึก "ใครแก้อะไร จากอะไรเป็นอะไร" อัตโนมัติจากชั้น ORM — ไม่เขียนรายตาราง. เก็บ 1 ปี → Super User archive.

ADR-004 · Action endpoint + envelope

ทุกการเปลี่ยนสถานะเป็น endpoint เฉพาะ (เช่น /po/confirm) เซิร์ฟเวอร์คุม state machine + สิทธิ์ + comment + cascade ในทรานแซกชันเดียว. response รูปแบบเดียว + pagination.

ADR-005 · แจ้งเตือน (outbox)

event เขียนคู่กับการเปลี่ยนสถานะ (ไม่หลุด). ผู้เห็น = คนมีสิทธิ์ Read ของ module. กดอ่าน = ack (badge ลด). ไม่ต้องมี broker ในเฟส 2.

ADR-006 · เก็บไฟล์ (GCS-ready)

ไฟล์แนบ/archive ผ่านตัวกลางตัวเดียว — ดิสก์ในเครื่อง (เฟส2) → GCS (เฟส3) โดยไม่แก้โค้ดธุรกิจ.

ADR-007 · Auth + RUCDAA

login local+Google, สิทธิ์ RUCDAA 6 บิตราย module, บิต Admin = สิทธิ์พิเศษ. session 24 ชม. + เตะออกทุก 06:00.

ADR-008 · เลขเอกสารห้ามข้าม

PO/PRD/INV/... เรียงต่อเนื่องตามสรรพากร ออกจากตัวนับที่ล็อกในทรานแซกชัน. PO reopen ใช้เลขเดิม.

ADR-009 · ลบแบบ soft + guard

ไม่ลบถาวร ติดธง "ถูกลบ" ยังค้นเจอแต่ผูกใหม่ไม่ได้. เอกสารการค้าใช้ "ยกเลิก" คงเลข. กู้คืน = Admin.
