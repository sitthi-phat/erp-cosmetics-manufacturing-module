หน้าจอ: settings.html (Role & สิทธิ์, ผู้ใช้) · API: api-settings.html รอสร้าง

ตารางกลาง (cross-cutting) — audit / noti / เลข / ไฟล์

ตารางชุดนี้ทำงาน "เบื้องหลัง" ให้ทุก module โดยไม่ต้องเขียนซ้ำรายตาราง — เป็นหัวใจที่ทำให้ audit ระดับ field, การแจ้งเตือน, เลขเอกสารห้ามข้าม และไฟล์แนบ ทำงานเหมือนกันทั้งระบบ.

audit_log จุดยาก

1 แถว = การเปลี่ยน 1 field: entity_type, entity_id, field, old_value, new_value, action, reason, actor_id, at, request_id. เขียนอัตโนมัติจากชั้น ORM (ADR-003). ป้อนหน้า trace + Settings→Audit log. IDX (entity_type,entity_id,at) และ (at). เก็บ online 1 ปี → Super User purge/archive.

status_history (มุมมองของ audit)

ประวัติเปลี่ยนสถานะ = filter action='transition' ของ audit_log — ไม่แยกตาราง กัน "2 ความจริง". แสดงบน timeline ทุกหน้า detail.

outbox + notification + notification_ack

outbox: event ที่เขียนใน transaction เดียวกับการเปลี่ยนสถานะ (กันหลุด). ตัว dispatcher แปลงเป็น notification (type, entity_ref, deep_link, module). ผู้เห็น = คนมีสิทธิ์ r ของ module ลบด้วยแถวใน notification_ack (นับ badge ราย user). ADR-005.

sequence จุดยาก

(name, period_key, current_value) — ออกเลขห้ามข้ามด้วย SELECT..FOR UPDATE ในทรานแซกชันสร้างเอกสาร. รองรับ PO/PRD/INV/PR/GR/SHP/DN. ADR-008.

file_asset

metadata ของไฟล์ (เอกสารใบรับ supplier, ไฟล์ archive trace): key, filename, mime, size, sha256, uploaded_by, entity_ref. ไฟล์จริงอยู่บน disk (เฟส2)/GCS (เฟส3) ผ่าน StorageService — ADR-006. ไม่เก็บไฟล์ใน DB.

login_event ใหม่

ใบสั่งซื้อ PO — 2 ราง (ผลิต/จัดส่ง + วางบิล)

หน้าจอ: po-create · po-detail · po-list · API: api-po.html · Functional: functional-spec/po BA ✓
PO มี 2 ราง reconcile กัน: ราง fulfilment (ร่าง→ยืนยัน→กำลังผลิต→พร้อมจัดส่ง→กำลังจัดส่ง→ส่งถึง→ยกเลิก) และราง billing (ยังไม่วางบิล→วางบิล→ชำระ/เกินกำหนด). ยกเลิกได้ทุกขั้น + reopen คงเลข PO เดิม (ADR-008). สินค้าแต่ละบรรทัดขายได้ทั้ง BOM และวัตถุดิบตรง ราคา 0 ได้. ไม่มีปุ่มลบ — ใช้ยกเลิก (ADR-009).
ตารางfield สำคัญอธิบาย
poid(PO-{YYYYMM}-{6}), customer_id FK, sale_owner_id, fulfil_status, billing_status, created_date, want_date, actual_ship_date, postpone_flag, postpone_date, version3 วันที่ = ค้นได้ 3 แบบ (IDX ทั้ง 3). postpone_* = flag "กันจัดส่ง Postpone" ระดับ PO. reopen คงเลขเดิม → lifecycle event ใน audit.
po_linepo_id FK, item_type(bom|material), bom_id|material_id, qty, unit_price(≥0 allow 0), bom_snapshot_jsonตอนยืนยัน PO → snapshot สูตร BOM ลง line (ลบ BOM ทีหลังไม่กระทบ). 1 line = 1 งานผลิต (1 PRD).
Cascade: PO "ยืนยัน" → สร้าง production_queue_item ต่อ line (สถานะ "รอรับงาน") ยังไม่เกิด PRD (entity-status-map §2 cascade#1). ลูกค้า Lead→Active ถ้าเป็น PO ใบแรก.

วัตถุดิบ + Lot + Stock Ledger (สต็อกติดลบได้)

หน้าจอ: stock · goods-receipt · API: api-stock.html · Functional: functional-spec/stock BA ✓
จุดยากที่สุดของระบบ — สต็อกเป็น "บัญชีเดินสะพัด" (append-only ledger) ไม่ใช่ตัวเลขเดียว. การผลิตตัด stock ได้แม้ของไม่พอ → ยอดติดลบได้ (ไม่บล็อกสายผลิต). พอทำ Goods Receipt ระบบบวกกลับ/ชดเชยติดลบ + ผูก lot ย้อนหลังแบบ FIFO เพื่อคง trace GMP. วิธีนี้ยังแก้บั๊ก DEF-09 (สต็อกชนกันตอนตัดพร้อมกัน) ไปในตัว. อ่านเต็ม ADR-001.
ตารางfield สำคัญอธิบาย
material softid, name, uom, low_stock_threshold(null=ไม่เตือน), threshold_type(pct|qty), default_buy_price, default_sell_price(0 ได้)เพิ่มวัตถุดิบใหม่ + UOM ได้. เกณฑ์ใกล้หมด ไม่มี default (null = เงียบจนตั้งค่า).
lot ใหม่id({prefix}{YYMM}), material_id FK, supplier_id, gr_id, qc_status(รอตรวจ/พร้อมใช้/ระงับ), received_qty, buy_price, received_atlot prefix ต่อ supplier. QC ขาเข้าตัดสินสถานะ lot. GMP: raw lot → batch. buy_price = ราคาซื้อของ lot (ใช้คิดมูลค่าสต็อก = on_hand × buy_price ของ lot ล่าสุด). IDX (material_id, received_at desc) เพื่อหา "lot ล่าสุด" เร็ว.
stock_movement จุดยากmaterial_id, lot_id(null ได้=ยังไม่ผูก lot), qty(±), reason(production_consume/goods_receipt/return_out/adjust/retro_alloc), ref_type, ref_id, caused_negative, created_by, at, noteappend-only — ห้าม UPDATE/DELETE. ตัดผลิต = แถว qty ลบ (ไม่ต้องเช็คก่อน → ไม่มี race). GR = แถวบวก + แถว retro_alloc ผูก consumption เก่าเข้า lot ใหม่ (FIFO).
stock_balance (cache)material_id, lot_id, qty_on_hand, updated_atยอดคงเหลือ = ผลรวม movement (อัปเดตแบบ qty=qty+delta atomic). rebuild ได้จาก ledger เสมอ. ติดลบ = แสดงสีแดง + badge "ติดลบ (รอรับเข้า)".

ผู้จำหน่าย + ตารางราคา (price matrix)

หน้าจอ: supplier · Functional: functional-spec/supplier BA ✓
ผูก "วัตถุดิบ 1 ตัว × supplier ได้หลายราย" พร้อมราคารับซื้อต่อคู่ = ตาราง matrix. supplier มี Active/Inactive. ราคานี้ใช้คิด "ราคาทุน BOM = สูงสุดของ supplier ที่ active". ไม่มีฟังก์ชันรับเข้าคลังในหน้านี้ (อยู่ที่ goods-receipt).
ตารางfield สำคัญอธิบาย
supplier ใหม่ softid, name, lot_prefix, is_active, contact...inactive แนะนำใช้แทนลบ. lot_prefix ใช้ gen เลข lot.
supplier_material_price จุดยากsupplier_id FK, material_id FK, buy_pricePK รวม (supplier_id, material_id) = matrix. คิดราคาทุน BOM ใช้เฉพาะ supplier ที่ is_active.

สูตรการผลิต BOM + snapshot ราคาทุน

หน้าจอ: bom · bom-create · Functional: functional-spec/bom BA ✓
ราคาทุน = สูงสุดของ supplier ที่ active คิด ณ ตอนสร้าง/แก้ + user แก้ทับได้ + snapshot ตอนบันทึก (ไม่คิดใหม่อัตโนมัติ) + badge เตือน "ราคาทุนอาจล้าสมัย" เมื่อราคา supplier ปัจจุบันต่างจาก snapshot. ราคาขาย บังคับกรอก. (entity-status-map §11 / ADR-001 §4).
ตารางfield สำคัญอธิบาย
bom softid, name, cost_snapshot, cost_overridden(bool), sell_price(บังคับ), snapshot_atbadge ล้าสมัย = เทียบ cost_snapshot กับ max(active supplier price) ปัจจุบัน (คิดตอนอ่าน).
bom_linebom_id FK, material_id FK, qtyส่วนผสม. ใช้คิดวัตถุดิบที่ต้องใช้ตอนเปิด PO/ผลิต (auto-calc + lot FIFO).

คำขอสั่งซื้อ PR + ใบรับเข้า GR

หน้าจอ: purchase-request · pr-create · goods-receipt
PR เกิด 2 ทาง: auto จาก PO วัตถุดิบขาด หรือ Stock สร้างตรง. "ของเข้าครบ" = auto จาก GR (ผูก lot). "รับทราบ/ปิดคำขอ" = manual. GR เป็น multi-line (1 supplier หลายบรรทัด) แต่ละบรรทัด gen lot + อ้าง PR รายบรรทัด → ปิด PR อัตโนมัติ + บวก stock กลับ (ชดเชยติดลบ). PR/GR ยกเลิก (void) ไม่ลบ.
ตารางfieldอธิบาย
purchase_request / _line ใหม่id(PR-{6}), status(เปิด/รับทราบ/รับบางส่วน/ของเข้าครบ/ปิด/ยกเลิก), source(auto|manual), po_id?, line(material, qty)รับบางส่วน → เสนอสร้าง PR ใหม่ (user review).
goods_receipt / _line ใหม่id(GR-{YYYYMMDD}-{3}), supplier_id, doc_no, file_asset_id, line(material, qty, buy_price, sell_price, lot_id gen, pr_id?)บันทึกรับเข้า → gen lot + ปิด PR + บวก stock/retro-alloc. โชว์ notice "ชดเชยยอดติดลบ X".

ใบสั่งผลิต PRD + รุ่นผลิต Batch

หน้าจอ: production · po-detail · API: api-production.html · Functional: functional-spec/production BA ✓
1 PO : N PRD (N=line) : M Batch (M≥N, +1 ทุก rework). PO ยืนยัน → คิว "รอรับงาน" (ยังไม่มีเลข). ฝ่ายผลิตกด "รับงาน" → เกิดเลข PRD (ไม่ auto). กด "เริ่มผลิต" → เกิด Batch run แรก + ตัด stock (ติดลบได้). QC ผ่านทุก PRD → PO พร้อมจัดส่ง. (entity-status-map §1.4/§1.5).
ตารางfield สำคัญอธิบาย
production_queue_item ใหม่po_id, po_line_id, status(=รอรับงาน)เกิดตอน PO ยืนยัน. ยังไม่ใช่ PRD/ไม่มีเลข. กดรับงาน → gen PRD แล้ว mark accepted.
production_order (PRD) ใหม่id(PRD-{YYYYMM}-{6}), po_line_id FK, status(รับงาน/กำลังผลิต/รอ QC/พร้อมส่งมอบ/Rework/Hold), hold_reason, potential_delay(bool overlay), accepted_by, accepted_atHold บังคับ comment + raise Sale/Stock. Potential Delay = overlay auto (2 วันผลิต+1 วันส่ง) ไม่ใช่ state แยก.
batch ใหม่id(B-{PO}-{line}-{run}), prd_id FK, run_no, status(กำลังผลิต/รอ QC/QC ผ่าน/QC ไม่ผ่าน)เกิดตอน "เริ่มผลิต". rework → run+1. หลักฐาน GMP ห้ามลบ.
batch_material_consumptionbatch_id FK, material_id, lot_id(null ช่วงติดลบ), qty, movement_id FKผูก Batch ↔ Lot ↔ movement = สาย trace GMP. retro-alloc เติม lot_id ย้อนหลังตอน GR.

หน้าจอ: qc · Functional: functional-spec/qc BA ✓

Shipment (รอบจัดส่ง) 1 รอบ = หลาย DN · DN 1 ใบ = 1 PO เสมอ (ลูกค้าเซ็นรายใบ). รอบจบเมื่อ DN ทุกใบถึงสถานะสุดท้าย. DN Reject → PO กลับพร้อมจัดส่ง + raise Sale. DN Postpone → PO พร้อมจัดส่ง + flag Postpone+วันที่ ค้างคิว. (entity-status-map §1.9/§1.10).

delivery_note ใหม่id(DN-{YYYYMMDD}-{5}), shipment_id FK, po_id FK (1:1), status(กำลังนำส่ง/ส่งถึง/ปฏิเสธ/เลื่อน)print ทีละใบ. ส่งถึง→PO Delivered เริ่มนับ overdue.
