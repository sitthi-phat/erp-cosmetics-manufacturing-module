API Spec — Purchase Order (PO)

รวม endpoint ของใบสั่งซื้อ ผูกกับหน้าจอและ use case จริง. PO มี 2 ราง (ผลิต/จัดส่ง + วางบิล) ยกเลิก/เปิดใหม่ได้ทุกขั้น (คงเลขเดิม) ราคา 0 ได้ ค้นได้ 3 แบบวันที่. การเปลี่ยนสถานะทุกครั้งเป็น "action endpoint" ที่เซิร์ฟเวอร์คุมกติกา + เก็บ trace + ยิงแจ้งเตือน.

มาตรฐานร่วม (ใช้ทุก endpoint)

ทุก response ห่อด้วย "ซอง" มาตรฐานเดียว: data (ข้อมูล), meta (เลขหน้า), error (ถ้าพลาด เป็นข้อความไทยพร้อมโชว์). ไม่มี enum ดิบหลุดถึง UI. สิทธิ์ตรวจด้วย RUCDAA (ADR-007) แนวคิด action endpoint ดู ADR-004.
// สำเร็จ (list)
{ "data": [ {...} ], "meta": { "page":1, "pageSize":20, "total":1204 }, "error": null }
// พลาด
{ "data": null, "error": { "code":"PO_INVALID_TRANSITION",
    "message":"ไม่สามารถยืนยันใบสั่งซื้อที่ยกเลิกแล้ว", "details":{...} } }

อ่าน / ค้นหา

GET/api/po
รายการ PO + pagination. ค้น 3 แบบวันที่: ?dateType=created|actual_ship|want&from&to · q (เลข PO/ชื่อลูกค้า) · fulfilStatus/billingStatus · saleOwnerId. สิทธิ์: Read po · หน้า po-list
GET/api/po/{id}
รายละเอียด PO 1 ใบ: 2 ราง (fulfilment+billing) · po_line + snapshot BOM · sale ที่ดูแล · PRD/Batch ต่อ line + reconcile · timeline (จาก audit) · flag Postpone. หน้า po-detail
GET/api/po/{id}/trace
ประวัติระดับ field ของ PO นี้ (ใครแก้อะไร จากอะไรเป็นอะไร) รวม lifecycle cancel→reopen. อ่านจาก audit_log

สร้าง / แก้ไข

POST/api/po
สร้าง PO (สถานะ ร่าง) → ออกเลข PO ทันที (gapless, ADR-008). body: customer_id, lines[{item_type, bom_id|material_id, qty, unit_price≥0 (0 ได้)}], want_date. ตอบ suggest วัตถุดิบ + เตือนถ้าขาด (ไม่บล็อก). สิทธิ์: Create po หน้า po-create
PATCH/api/po/{id}
แก้ header/line (จำนวน/สินค้า/ราคา/วันส่ง) — ได้เสมอตามสถานะที่อนุญาต รวมเคส Hold ที่ Sale แก้ได้ทั้งหมด. ทุก field ที่แก้ลง audit. ต้องส่ง version (กันแก้ชนกัน → 409). สิทธิ์: Update po
GET/api/po/material-check
เช็ควัตถุดิบตาม BOM+จำนวน (ใช้ตอนสร้าง/แก้): คืน "ครบ/ขาดอะไรกี่หน่วย" — ขาด = เตือน + เสนอสร้าง PR (ไม่บล็อก).

เปลี่ยนสถานะ (action endpoints)

แต่ละปุ่มบนหน้าจอ = 1 endpoint. เซิร์ฟเวอร์ตรวจว่าเปลี่ยนสถานะนี้ถูกต้องไหม + สิทธิ์พอไหม + comment ครบไหม แล้วทำ cascade + audit + แจ้งเตือน ในทรานแซกชันเดียว.
Endpointทำอะไร (cascade)สิทธิ์ / comment
POST /po/{id}/confirmร่าง→ยืนยัน · สร้างคิว "รอรับงาน" ต่อ line (ยังไม่เกิด PRD) · ลูกค้า Lead→Active · ยิง noti Production · เช็ควัตถุดิบขาด→สร้าง PRApprove po
POST /po/{id}/cancel→ยกเลิก (ได้ทุกขั้น) · คิว/PRD/Batch ที่เกิดถูกยกเลิกตาม (trace)Delete po · comment บังคับ
POST /po/{id}/reopenยกเลิก→ร่าง · คงเลข PO เดิม · lifecycle event ใน auditAdmin po · comment บังคับ
POST /po/{id}/override-statusข้ามลำดับสถานะ (force) — เฉพาะกรณีจำเป็นAdmin po · comment บังคับ
สถานะ "กำลังผลิต / พร้อมจัดส่ง / กำลังจัดส่ง / ส่งถึง" ของ PO เป็น auto จาก cascade ของ Production/Shipping (ไม่ใช่ endpoint ตรงของ PO) — ดู api-production และ shipping.

รหัส error (ตัวอย่าง)

codeเมื่อไร (ข้อความไทยที่ผู้ใช้เห็น)
PO_INVALID_TRANSITIONเปลี่ยนสถานะไม่ถูกลำดับ (เช่น ยืนยันใบที่ยกเลิกแล้ว)
PO_REASON_REQUIREDยกเลิก/override โดยไม่ใส่เหตุผล
CONFLICT_STALEมีคนแก้ก่อนหน้า (version ไม่ตรง) → โหลดใหม่
REFERENCE_DELETEDอ้างลูกค้า/วัตถุดิบ/BOM ที่ถูกลบ (ADR-009)
FORBIDDEN_RUCDAAสิทธิ์ไม่พอสำหรับ action นี้
