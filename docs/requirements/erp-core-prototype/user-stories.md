## Epic 1: Customer Management (Sales/CS)

### ECP-001: สร้างข้อมูลลูกค้าใหม่
**Priority:** Must

ในฐานะ Sales/CS ฉันต้องการสร้างข้อมูลลูกค้าใหม่ในระบบ เพื่อให้เปิด PO ให้ลูกค้ารายนั้นได้ และเก็บ
ข้อมูลที่จำเป็นต่อการออกใบกำกับภาษีในอนาคต

- AC1 (Happy): Given ฉันอยู่หน้า "สร้างลูกค้าใหม่" (ฟอร์มไม่มีช่องให้กรอกรหัสลูกค้าเอง) และกรอกชื่อลูกค้า, ที่อยู่, เบอร์โทร, อีเมล ครบทุกช่องที่บังคับ, When ฉันกดบันทึก, Then ระบบสร้างลูกค้าใหม่สำเร็จ พร้อมกำหนดรหัสลูกค้า (Customer ID) ให้อัตโนมัติแบบไม่ซ้ำกัน (รูปแบบจริงกำหนดโดย Tech-Lead ใน ADR-006) แสดงสถานะ "Active" และแสดงในหน้ารายการลูกค้าทันทีโดยไม่ต้อง refresh หน้า
- AC2 (Edge): Given ฉันกรอกชื่อลูกค้าที่ซ้ำกับชื่อลูกค้าที่มีอยู่แล้วในระบบ (ชื่อตรงกันทุกตัวอักษร), When ฉันกดบันทึก, Then ระบบยังคงบันทึกลูกค้าใหม่ให้สำเร็จแต่แสดงข้อความเตือน "พบชื่อลูกค้าที่คล้ายกันในระบบ กรุณาตรวจสอบก่อนดำเนินการต่อ" (ไม่ block การบันทึก)
- AC3 (Error): Given ฉันไม่กรอกชื่อลูกค้า (ปล่อยว่าง) แล้วกดบันทึก, When ระบบตรวจสอบความถูกต้องของฟอร์ม, Then ระบบไม่บันทึกข้อมูลและแสดงข้อความ "กรุณากรอกชื่อลูกค้า" ใต้ช่องชื่อลูกค้าทันทีโดยไม่ต้องโหลดหน้าใหม่
- AC4 (Edge): Given มีการส่งคำขอสร้างลูกค้าที่พยายามระบุค่า customer_id มาเองจากฝั่ง client, When ระบบรับคำขอสร้างลูกค้านั้น, Then ระบบเพิกเฉยค่า customer_id ที่ส่งมาทั้งหมดและ generate รหัสใหม่ให้เองเสมอ
- AC5 (Happy) **[เพิ่มรอบ Gate 2 — customer tax fields, feedback ข้อ 9]**: Given ฉันกรอกข้อมูลลูกค้าใหม่ครบทุกช่องบังคับเดิม และเพิ่มกรอกเลขประจำตัวผู้เสียภาษี 13 หลัก "0105558000001" พร้อมที่อยู่จดทะเบียน (registered_address) สำหรับใช้พิมพ์บนใบกำกับภาษี, When ฉันกดบันทึก, Then ระบบบันทึกลูกค้าสำเร็จพร้อมฟิลด์ tax_id และ registered_address ครบถ้วน พร้อมใช้ในการออกใบกำกับภาษี (ECP-042) ได้ทันที
- AC6 (Error) **[เพิ่มรอบ Gate 2]**: Given ฉันกรอกเลขประจำตัวผู้เสียภาษีไม่ครบ 13 หลัก (เช่น 10 หลัก) หรือมีตัวอักษรที่ไม่ใช่ตัวเลขปนอยู่, When ฉันกดบันทึก, Then ระบบไม่บันทึกลูกค้าใหม่และแสดงข้อความ "เลขประจำตัวผู้เสียภาษีต้องเป็นตัวเลข 13 หลักเท่านั้น" ใต้ช่องกรอก
- AC7 (Edge) **[เพิ่มรอบ Gate 2 — ⚠ ข้อสมมติ: tax_id ไม่บังคับตอนสร้าง]**: Given ฉันสร้างลูกค้าใหม่โดยเว้นว่างช่องเลขประจำตัวผู้เสียภาษีไว้ก่อน (ยังไม่ทราบข้อมูลในตอนนั้น), When ฉันกดบันทึก, Then ระบบบันทึกลูกค้าสำเร็จได้ตามปกติ (tax_id ไม่ใช่ช่องบังคับตอนสร้าง) แต่ระบบต้องแสดงป้ายเตือน "ลูกค้ารายนี้ยังไม่มีเลขประจำตัวผู้เสียภาษี จะไม่สามารถออกใบกำกับภาษีได้จนกว่าจะเพิ่มข้อมูลนี้" (สอดคล้อง ECP-042 AC4)

### ECP-002: แก้ไขข้อมูลลูกค้า
**Priority:** Should | Depends on: ECP-001

ในฐานะ Sales/CS ฉันต้องการแก้ไขข้อมูลลูกค้าที่มีอยู่แล้ว เพื่อให้ข้อมูลติดต่อ/ที่อยู่/ข้อมูลภาษีเป็นปัจจุบันเสมอ

- AC1 (Happy): Given ลูกค้ารายหนึ่งมีเบอร์โทรเดิมในระบบ, When ฉันแก้ไขเบอร์โทรเป็นเบอร์ใหม่แล้วกดบันทึก, Then ระบบอัปเดตเบอร์โทรทันทีและแสดงเบอร์ใหม่ในหน้ารายละเอียดลูกค้าโดยไม่ต้อง refresh
- AC2 (Edge): Given ลูกค้ารายนั้นมี PO ที่ยังไม่ปิด (status ≠ Closed/Cancelled) ผูกอยู่, When ฉันพยายามเปลี่ยนสถานะลูกค้าเป็น "Inactive", Then ระบบอนุญาตให้เปลี่ยนสถานะได้แต่แสดงข้อความเตือน "ลูกค้ารายนี้มี PO ที่ยังไม่ปิด N รายการ"
- AC3 (Error): Given ฉันลบข้อมูลอีเมลออกจากช่องอีเมลที่เคยกรอกไว้ (ปล่อยว่าง) โดยที่อีเมลเป็นช่องบังคับ, When ฉันกดบันทึก, Then ระบบไม่บันทึกการแก้ไขและแสดงข้อความ "กรุณากรอกอีเมล" พร้อมข้อมูลเดิมยังคงอยู่ไม่ถูกเขียนทับ
- AC4 (Edge) **[เพิ่มรอบ Gate 2 — tax_id snapshot]**: Given ลูกค้ารายหนึ่งมี tax_id เดิมในระบบและเคยถูกใช้ออกใบกำกับภาษีไปแล้วอย่างน้อย 1 ใบ, When ฉันแก้ไข tax_id เป็นเลขใหม่ (13 หลักถูกต้อง) แล้วกดบันทึก, Then ระบบอัปเดต tax_id ของลูกค้าทันทีสำหรับใบกำกับภาษีที่จะออกใหม่หลังจากนี้ แต่ใบกำกับภาษีที่เคยพิมพ์ไปแล้วก่อนหน้ายังคงแสดงเลขเดิม ณ ตอนที่ออกเอกสาร (snapshot ไม่แก้ไขย้อนหลัง เช่นเดียวกับหลักการ VATConfig snapshot ของ ECP-038 AC2)

### ECP-003: ค้นหา/ดูรายละเอียดลูกค้าและประวัติ PO
**Priority:** Should | Depends on: ECP-001, ECP-004

ในฐานะ Sales/CS ฉันต้องการค้นหาลูกค้าและดูประวัติ PO ทั้งหมดของลูกค้ารายนั้น เพื่อติดตามความสัมพันธ์และสถานะคำสั่งซื้อได้ในที่เดียว

- AC1 (Happy): Given มีลูกค้าชื่อ "บริษัท ABC" อยู่ในระบบพร้อม PO 3 รายการ, When ฉันพิมพ์ "ABC" ในช่องค้นหาแล้วกด Enter, Then ระบบแสดงผลลัพธ์ลูกค้า "บริษัท ABC" ภายใน 3 วินาที พร้อมรายการ PO ทั้ง 3 รายการเรียงจากล่าสุดไปเก่าสุด
- AC2 (Edge): Given ไม่มีลูกค้าที่ชื่อตรงกับคำค้นหาในระบบ, When ฉันค้นหาคำว่า "XYZ999", Then ระบบแสดงข้อความ "ไม่พบลูกค้าที่ตรงกับคำค้นหา" แทนหน้าว่างเปล่าที่ไม่มีคำอธิบาย
- AC3 (Error): Given ระบบฐานข้อมูลตอบสนองล่าช้าผิดปกติ (timeout), When ฉันค้นหาลูกค้า, Then ระบบแสดงข้อความ "ไม่สามารถค้นหาข้อมูลได้ในขณะนี้ กรุณาลองใหม่อีกครั้ง" พร้อมปุ่ม "ลองใหม่" แทนหน้าขาว/ค้าง

## Epic 2: Purchasing Order (Sales/CS)

### ECP-004: เปิด PO ใหม่จากลูกค้าพร้อมตรวจสอบ stock ทันที (บรรทัดสินค้าอ่านง่าย + ลบได้)
**Priority:** Must (KPI #1, #3, #4) | Depends on: ECP-001, ECP-009

ในฐานะ Sales/CS ฉันต้องการเปิด PO ให้ลูกค้าโดยระบุสินค้า/จำนวนที่ต้องการ เห็นรายการที่เพิ่มแล้วเป็น
ข้อความที่อ่านเข้าใจง่าย ลบรายการที่เพิ่มผิดได้ก่อนยืนยัน และเห็นผลตรวจสอบ stock วัตถุดิบตามสูตร
(BOM) ทันที เพื่อไม่ต้องเปิด PO ไปแล้วพบภายหลังว่าวัตถุดิบไม่พอ

- AC1 (Happy) **[แก้ไขรอบ Gate 2 — defect: บรรทัดสินค้าอ่านไม่ออก, feedback ข้อ 1]**: Given ลูกค้า "บริษัท ABC" มีอยู่ในระบบ, When ฉันเพิ่มสินค้า "ครีมกันแดด SPF50" จำนวน 100 หน่วย ราคาต่อหน่วย 150.00 บาท ลงในบรรทัด PO, Then ระบบแสดงบรรทัดสินค้าในตารางด้วยข้อมูลที่อ่านเข้าใจง่าย: ชื่อสินค้าจริง "ครีมกันแดด SPF50", จำนวน "100 หน่วย", ราคา "150.00 บาท/หน่วย", ยอดรวมบรรทัด "15,000.00 บาท" — **ต้องไม่พบรูปแบบข้อความ `"Product #<id> x <qty> @ <price>"` หรือรหัสภายในดิบอื่นใดปรากฏบนหน้าจอนี้เด็ดขาด** (regression guard ตรงตามฟีดแบ็คปอนด์) และเมื่อ stock วัตถุดิบตามสูตรพอสำหรับทั้ง PO แล้วกดยืนยัน ระบบสร้าง PO สถานะ "Confirmed" สำเร็จ พร้อมจองสต็อกวัตถุดิบที่เกี่ยวข้องทันที
- AC2 (Happy) **[เพิ่มรอบ Gate 2 — ลบบรรทัดก่อนยืนยัน, feedback ข้อ 1]**: Given ฉันเพิ่มบรรทัดสินค้าไว้ 2 บรรทัดในฟอร์ม PO ที่ยังไม่ได้กดยืนยัน (สถานะ Draft), When ฉันกดปุ่ม "ลบ" ที่บรรทัดหนึ่งก่อนกดยืนยัน PO, Then ระบบลบบรรทัดนั้นออกจากตารางทันที เหลือเพียง 1 บรรทัด และเมื่อกดยืนยัน PO ระบบตรวจสอบ/สร้าง PO ตามบรรทัดที่เหลืออยู่จริงเท่านั้น (ไม่มีร่องรอยของบรรทัดที่ลบไปแล้ว)
- AC3 (Edge): Given วัตถุดิบชนิดหนึ่งในสูตรของสินค้าที่สั่งมีคงเหลือพอสำหรับผลิตได้เพียงบางส่วน (ไม่พอเต็มจำนวนที่สั่ง 100 หน่วย), When ฉันกดยืนยัน PO, Then ระบบไม่อนุญาตให้เปลี่ยนสถานะเป็น "Confirmed" ยังคงสถานะ "Draft" และแสดงข้อความแจ้งเตือน "วัตถุดิบ [ชื่อวัตถุดิบ] ไม่เพียงพอ ขาดอยู่ [จำนวนที่ขาด] [หน่วย]"
- AC4 (Error): Given ฉันเลือกลูกค้าแล้วแต่ยังไม่ได้เพิ่มรายการสินค้าใดเลยในบรรทัด PO, When ฉันกดยืนยัน PO, Then ระบบไม่สร้าง PO และแสดงข้อความ "กรุณาเพิ่มรายการสินค้าอย่างน้อย 1 รายการก่อนยืนยัน"
- AC5 (Error) **[เพิ่มรอบ Gate 2]**: Given PO ถูกกดยืนยันไปแล้ว (สถานะ "Confirmed" ขึ้นไป), When ฉันพยายามลบบรรทัดสินค้าของ PO นี้ผ่านหน้าจอเดิม (เช่น cache เก่า/เปิดหลายแท็บ), Then ระบบปฏิเสธการลบและแสดงข้อความ "PO นี้ถูกยืนยันแล้ว ไม่สามารถลบบรรทัดสินค้าได้อีก กรุณาใช้ฟังก์ชันแก้ไข/ยกเลิก PO แทน" (ป้องกันข้อมูลไม่ตรงกับสต็อกที่จองไว้แล้ว)

### ECP-005: แก้ไข/ยกเลิก PO ก่อนเริ่มผลิต
**Priority:** Should | Depends on: ECP-004, ECP-010

ในฐานะ Sales/CS ฉันต้องการแก้ไขหรือยกเลิก PO ที่ยังไม่เข้าสู่กระบวนการผลิต เพื่อรองรับการเปลี่ยนแปลงคำสั่งซื้อของลูกค้า

- AC1 (Happy): Given PO สถานะ "Confirmed" ที่ยังไม่ถูกส่งเข้าคิวผลิต, When ฉันกดยกเลิก PO, Then ระบบเปลี่ยนสถานะ PO เป็น "Cancelled" และคืนสต็อกวัตถุดิบที่เคยจองไว้กลับเข้าสต็อกส่วนกลางทันที
- AC2 (Edge): Given PO สถานะ "InProduction" ผูกอยู่, When ฉันพยายามกดยกเลิก PO, Then ระบบไม่อนุญาตให้ยกเลิกและแสดงข้อความ "ไม่สามารถยกเลิก PO นี้ได้ เนื่องจากเริ่มกระบวนการผลิตแล้ว"
- AC3 (Error): Given PO ถูกยกเลิกไปแล้ว (สถานะ "Cancelled"), When ฉันพยายามกดยกเลิกซ้ำอีกครั้ง, Then ระบบแสดงข้อความ "PO นี้ถูกยกเลิกไปแล้วเมื่อ [วันที่-เวลา]" และไม่ทำรายการซ้ำ

### ECP-006: ติดตามสถานะ PO แบบ end-to-end
**Priority:** Must (KPI #3) | Depends on: ECP-004, ECP-013, ECP-015, ECP-018, ECP-020

ในฐานะ Sales/CS ฉันต้องการดูสถานะ PO ทุกขั้นตอนตั้งแต่เปิดจนถึง invoice ในหน้าเดียว เพื่อตอบคำถามลูกค้าได้ทันทีโดยไม่ต้องถามแผนกอื่น

- AC1 (Happy): Given PO หนึ่งรายการผ่านขั้นตอน Confirmed → InProduction → QC Approved → Shipped → Invoiced ครบแล้ว, When ฉันเปิดหน้ารายละเอียด PO นั้น, Then ระบบแสดง timeline สถานะทั้ง 5 ขั้นตอนพร้อมวันที่-เวลาของแต่ละสถานะ เรียงตามลำดับเวลาจริง
- AC2 (Edge): Given PO ถูกปฏิเสธจาก QA/QC (Batch สถานะ "QC Rejected") ทำให้ต้องผลิตใหม่, When ฉันเปิดหน้าสถานะ PO, Then ระบบแสดงสถานะ "รอผลิตใหม่ (QC ไม่ผ่าน)" พร้อมเหตุผลที่ QA บันทึกไว้
- AC3 (Error): Given PO ID ที่ระบุใน URL ไม่มีอยู่จริงในระบบ, When ฉันเปิดหน้ารายละเอียด PO ด้วย ID นั้น, Then ระบบแสดงข้อความ "ไม่พบคำสั่งซื้อนี้ในระบบ" แทนหน้าเปล่า/error 500

## Epic 3: Stock & Real-time Inventory (Warehouse) — Priority สูงสุดของ scope

### ECP-007: ดูยอดคงเหลือ stock วัตถุดิบแบบ real-time พร้อมค้นหา
**Priority:** Must (KPI #1)

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการดูยอดคงเหลือของวัตถุดิบทุกชนิดในหน้าเดียวแบบ real-time และ
ค้นหาวัตถุดิบที่ต้องการได้ เพื่อรู้ทันทีว่าวัตถุดิบชนิดใดเหลือเท่าไหร่โดยไม่ต้องเดินคลัง/โทรถาม

- AC1 (Happy): Given วัตถุดิบ "น้ำมันมะพร้าว" มียอดคงเหลือ 500 กก., When มีการบันทึกรับเข้า (Goods Receipt) เพิ่ม 100 กก., Then หน้าจอ stock ของฉัน (ที่เปิดค้างอยู่) แสดงยอดใหม่เป็น 600 กก. ภายใน ≤1 นาทีโดยไม่ต้องกด refresh หน้าเอง
- AC2 (Edge): Given วัตถุดิบชนิดหนึ่งมียอดคงเหลือเท่ากับ 0 พอดี, When ฉันเปิดหน้า stock, Then ระบบแสดงแถวของวัตถุดิบนั้นด้วยยอด "0" และป้ายสถานะ "หมดสต็อก" ไม่ใช่การซ่อนแถวนั้นออกจากรายการ
- AC3 (Error): Given ระบบเชื่อมต่อฐานข้อมูล stock ขัดข้องชั่วคราว, When ฉันเปิดหน้า stock, Then ระบบแสดงข้อความ "ไม่สามารถโหลดข้อมูล stock ได้ในขณะนี้ กรุณาลองใหม่" พร้อมเวลาล่าสุดที่ข้อมูลเคยอัปเดตสำเร็จ
- AC4 (Happy) **[เพิ่มรอบ Gate 2 — defect: search ไม่มี, feedback ข้อ 2]**: Given มีวัตถุดิบ 50 ชนิดในหน้า stock รวมถึง "น้ำมันมะพร้าว", When ฉันพิมพ์ "มะพร้าว" ในช่องค้นหาแล้วระบบ filter, Then ระบบแสดงเฉพาะแถววัตถุดิบที่ชื่อหรือรหัสวัตถุดิบตรงกับคำค้นหา (case-insensitive) ภายใน ≤2 วินาที ซ่อนแถวอื่นทั้งหมดที่ไม่ตรงคำค้นหา
- AC5 (Edge) **[เพิ่มรอบ Gate 2]**: Given ฉันค้นหาด้วยคำที่ไม่ตรงกับชื่อ/รหัสวัตถุดิบใดเลยในระบบ เช่น "XYZ999", When ฉันกดค้นหา, Then ระบบแสดงข้อความ "ไม่พบวัตถุดิบที่ตรงกับคำค้นหา" แทนตารางว่างเปล่า พร้อมปุ่ม "ล้างการค้นหา" ให้กลับไปดูรายการทั้งหมด

### ECP-008: บันทึกรับวัตถุดิบเข้าคลัง (Goods Receipt) พร้อม Lot number
**Priority:** Must

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการบันทึกการรับวัตถุดิบเข้าคลังพร้อมระบุ Lot number และผู้จำหน่าย
เพื่อให้ตรวจสอบย้อนกลับ (traceability) ได้ตั้งแต่ต้นทางและให้ QA ใช้ตรวจสอบขาเข้าต่อได้

- AC1 (Happy): Given วัตถุดิบ "แอลกอฮอล์" ถูกส่งมอบเข้าคลัง 200 ลิตร พร้อมเลข Lot ที่ผู้จำหน่ายระบุ, When ฉันบันทึกรับเข้าโดยกรอกชนิดวัตถุดิบ, จำนวน 200, หน่วย ลิตร, และเลข Lot, Then ระบบสร้างระเบียน Lot ใหม่ผูกกับวัตถุดิบนั้น เพิ่มยอด stock รวม +200 ลิตรทันที และ Lot นั้นพร้อมให้เลือกใช้ในการผลิต (หลังผ่าน QC ขาเข้าตาม ECP-017)
- AC2 (Edge): Given ฉันกรอกเลข Lot ที่ตรงกับ Lot number ที่มีอยู่แล้วของวัตถุดิบชนิดเดียวกัน, When ฉันบันทึกรับเข้า, Then ระบบแจ้งเตือน "พบ Lot number นี้อยู่แล้ว ต้องการรวมยอดเข้ากับ Lot เดิมหรือสร้าง Lot ใหม่?" ให้ผู้ใช้เลือกยืนยัน
- AC3 (Error): Given ฉันกรอกจำนวนรับเข้าเป็นค่าติดลบหรือ 0, When ฉันกดบันทึก, Then ระบบไม่บันทึกรายการและแสดงข้อความ "จำนวนรับเข้าต้องมากกว่า 0"
- AC4 (Edge) **[เพิ่มรอบ Gate 2 — เก็บชื่อผู้จำหน่ายสำหรับ QC ขาเข้า, ผูกกับ feedback ข้อ 5]**: Given ฉันบันทึกรับวัตถุดิบเข้าคลังตาม AC1, When ฉันกรอกชื่อผู้จำหน่าย (supplier_name) ควบคู่กับข้อมูล Lot, Then ระบบบันทึก supplier_name ผูกกับ Lot นั้นไว้ด้วย เพื่อให้ QA/QC ใช้อ้างอิงตอนตรวจสอบขาเข้า (ECP-017) โดยไม่ต้องกรอกซ้ำ

### ECP-009: ระบบตรวจสอบ stock วัตถุดิบตามสูตร (BOM) และแจ้งเตือน insufficient stock
**Priority:** Must (KPI #4 — 100% ของ test case ใน UAT)

ในฐานะเจ้าหน้าที่คลังสินค้า/Sales ฉันต้องการให้ระบบตรวจสอบสต็อกวัตถุดิบเทียบกับสูตร (BOM) ของสินค้าที่สั่งโดยอัตโนมัติทุกครั้งที่เปิด/ยืนยัน PO เพื่อไม่ให้เกิด surprise ว่าวัตถุดิบไม่พอหลังเปิด PO ไปแล้ว

- AC1 (Happy): Given สูตรของสินค้า A ต้องใช้วัตถุดิบ X 2 กก./หน่วย และสต็อก X คงเหลือ 1,000 กก., When มีการเปิด PO สั่งสินค้า A จำนวน 400 หน่วย (ต้องใช้ X รวม 800 กก.), Then ระบบคำนวณแล้วแสดงผล "stock เพียงพอ (เหลือ 200 กก. หลังหักจอง)" ก่อนให้ผู้ใช้ยืนยัน PO
- AC2 (Edge): Given สินค้า A ใช้วัตถุดิบ 3 ชนิดตามสูตร และมีเพียง 1 ใน 3 ชนิดที่ไม่พอ, When ระบบตรวจสอบ ณ จุดเปิด/ยืนยัน PO, Then ระบบแจ้งเตือนเจาะจงเฉพาะวัตถุดิบที่ไม่พอ (ระบุชื่อ+จำนวนที่ขาด) โดยไม่แจ้งเตือนวัตถุดิบอีก 2 ชนิดที่เพียงพอ
- AC3 (Error): Given สินค้าที่เลือกในบรรทัด PO ยังไม่มีสูตร (BOM) ผูกไว้ในระบบ, When ผู้ใช้พยายามยืนยัน PO ที่มีสินค้านั้น, Then ระบบไม่อนุญาตให้ยืนยันและแสดงข้อความ "สินค้านี้ยังไม่มีสูตรการผลิต (BOM) ในระบบ กรุณาติดต่อผู้ดูแลระบบ" (หรือใช้ ECP-039 เพื่อสร้าง BOM ให้สินค้านี้)

### ECP-010: ระบบจอง/คืนสต็อกอัตโนมัติตามธุรกรรม
**Priority:** Must (KPI #1)

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการให้ระบบจองสต็อกวัตถุดิบอัตโนมัติเมื่อ PO ถูกยืนยัน และคืนสต็อกอัตโนมัติเมื่อ PO ถูกยกเลิกหรือเบิกใช้จริงในการผลิต เพื่อให้ยอด stock ที่แสดงตรงกับความเป็นจริงเสมอแบบ real-time

- AC1 (Happy): Given วัตถุดิบ X มียอดคงเหลือ (available) 1,000 กก. ไม่มีการจอง, When PO ที่ต้องใช้ X จำนวน 300 กก. ถูกยืนยัน (Confirmed), Then ระบบลดยอด "available to promise" ของ X เหลือ 700 กก. ทันที ในขณะที่ยอด "physical stock" ยังคง 1,000 กก.
- AC2 (Edge): Given มีการจอง X ไว้ 300 กก. จาก PO หนึ่งใบ, When PO นั้นถูกยกเลิกก่อนเข้าสู่ขั้นตอนผลิต, Then ระบบคืนยอดจอง 300 กก. กลับเข้า "available to promise" ทันทีในธุรกรรมเดียว
- AC3 (Error): Given ฝ่ายผลิตพยายามเบิกใช้วัตถุดิบ X จำนวน 300 กก. ในการผลิตจริง แต่ยอด physical stock ของ X ขณะนั้นเหลือเพียง 250 กก., When ระบบบันทึกการเบิกผลิต, Then ระบบไม่อนุญาตให้เบิกเกินยอดจริงที่มีอยู่ และแสดงข้อความ "วัตถุดิบ X คงเหลือจริงไม่พอ (มี 250 กก. ต้องการ 300 กก.)"
- AC4 (Edge): Given วัตถุดิบ Y มีธุรกรรม stock สะสม 1,000 รายการในช่วงเวลาหนึ่งจากผู้ใช้หลายคนพร้อมกัน, When ระบบคำนวณผลรวมยอดคงเหลือจาก stock ledger เทียบกับยอด "physical stock" ปัจจุบันของวัตถุดิบ Y, Then ผลรวมทั้งสองต้องตรงกัน 100% ไม่มีส่วนต่างแม้แต่หน่วยเดียว

## Epic 4: Production & Traceability (Production)

### ECP-011: ดูคิวงานผลิตที่รอดำเนินการ
**Priority:** Must | Depends on: ECP-004, ECP-009

ในฐานะฝ่ายผลิต ฉันต้องการดูคิว PO/รายการที่ผ่านการตรวจสอบ stock แล้วและรอเข้าสู่กระบวนการผลิต เพื่อวางแผนงานผลิตได้

- AC1 (Happy): Given มี PO 3 รายการที่สถานะ "Confirmed" และผ่านการตรวจ stock เพียงพอแล้ว, When ฉันเปิดหน้าคิวงานผลิต, Then ระบบแสดงรายการทั้ง 3 PO เรียงตามวันที่ต้องการส่งมอบจากใกล้ไปไกล
- AC2 (Edge): Given ไม่มี PO ใดอยู่ในสถานะพร้อมผลิตในขณะนั้น, When ฉันเปิดหน้าคิวงานผลิต, Then ระบบแสดงข้อความ "ไม่มีงานผลิตที่รอดำเนินการในขณะนี้" แทนหน้าว่างเปล่า
- AC3 (Error): Given PO รายการหนึ่งถูกยกเลิกโดย Sales/CS ในระหว่างที่ฝ่ายผลิตกำลังดูรายการอยู่, When ฉัน refresh หน้าคิวงานผลิต, Then PO ที่ถูกยกเลิกหายไปจากคิวทันทีพร้อมไม่สามารถกด assign งานให้ PO นั้นได้อีก

### ECP-012: มอบหมาย (assign) งานผลิตให้ผู้ปฏิบัติงาน
**Priority:** Must | Depends on: ECP-011

ในฐานะฝ่ายผลิต (หัวหน้างาน) ฉันต้องการมอบหมายงานผลิตจาก PO ที่พร้อมให้ผู้ปฏิบัติงานคนใดคนหนึ่ง เพื่อระบุความรับผิดชอบชัดเจนตาม GMP

- AC1 (Happy): Given PO หนึ่งรายการอยู่ในคิวงานผลิตสถานะ "รอ assign", When ฉันเลือกผู้ปฏิบัติงาน "สมชาย" แล้วกดมอบหมาย, Then ระบบสร้าง Production Order ผูกกับ PO นั้น สถานะเปลี่ยนเป็น "Assigned" และแสดงชื่อ "สมชาย" เป็นผู้รับผิดชอบทันที
- AC2 (Edge): Given Production Order หนึ่งรายการถูก assign ให้ "สมชาย" ไปแล้ว, When หัวหน้างานอีกคนพยายาม assign Production Order เดียวกันซ้ำให้ "สมหญิง", Then ระบบแจ้งเตือน "งานนี้ถูกมอบหมายให้สมชายแล้วเมื่อ [เวลา]" และให้เลือกยืนยันเปลี่ยนผู้รับผิดชอบหรือยกเลิก
- AC3 (Error): Given ไม่มีผู้ใช้งาน role "ฝ่ายผลิต" ที่มีสถานะ Active อยู่ในระบบเลย, When ฉันพยายามเปิด dropdown เลือกผู้ปฏิบัติงาน, Then ระบบแสดงข้อความ "ไม่พบผู้ปฏิบัติงานที่พร้อมรับงานในขณะนี้ กรุณาติดต่อ Admin"

### ECP-013: บันทึกการดำเนินการผลิต — ระบบคำนวณวัตถุดิบจาก BOM ให้ review/ยืนยัน พร้อมเชื่อม Lot สร้าง Batch
**Priority:** Must (KPI #2) | Depends on: ECP-012, ECP-008, ECP-010, ECP-039

**[แก้ไขทั้ง story รอบ Gate 2 — defect: lot search error (feedback ข้อ 4a) + scope ใหม่: auto-calc จาก BOM (feedback ข้อ 4b)]**

ในฐานะฝ่ายผลิต ฉันต้องการให้ระบบคำนวณปริมาณวัตถุดิบที่ต้องเบิกจาก BOM ของสินค้าที่ผลิตให้อัตโนมัติ
พร้อมเสนอ Lot ที่แนะนำให้เลือก (review/ปรับได้) แทนการค้นหา/กรอก lot ID เอง เพื่อบันทึกผลิตได้เร็ว
ถูกต้อง และให้ Batch ทุกใบตรวจสอบย้อนกลับไปถึง Lot วัตถุดิบที่ใช้ได้ตาม GMP

- AC1 (Happy) — auto-calc: Given Production Order สถานะ "Assigned" สำหรับสินค้า "ครีมกันแดด SPF50" จำนวนที่ต้องผลิต 500 หน่วย และ BOM ของสินค้านี้ระบุใช้วัตถุดิบ "น้ำมันมะพร้าว" 2 กก./หน่วย, When ฉันเปิดหน้าบันทึกผลิตสำหรับ Production Order นี้, Then ระบบคำนวณและแสดงปริมาณที่ต้องใช้ล่วงหน้า = 2 × 500 = 1,000 กก. ของ "น้ำมันมะพร้าว" (และวัตถุดิบอื่นตาม BOM ทุกชนิด) โดยไม่ต้องให้ฝ่ายผลิตพิมพ์ตัวเลขเอง พร้อมเสนอ Lot ที่แนะนำอัตโนมัติ (เรียงจาก Lot ที่รับเข้าเก่าสุดก่อน — FIFO default) ให้ครบตามจำนวนที่ต้องใช้
- AC2 (Happy) — lot selection ใช้งานได้จริง (แก้ defect): Given ระบบเสนอ Lot "L-SEED-1" (เดิมเคยพิมพ์ค้นหาแล้ว error ตามฟีดแบ็คปอนด์) เป็นค่าที่แนะนำสำหรับวัตถุดิบชนิดหนึ่ง, When ฉันตรวจสอบ (review) แล้วกด "ยืนยันใช้ Lot ที่เสนอ" โดยไม่แก้ไข แล้วกรอกจำนวนสินค้าสำเร็จรูปที่ได้ 500 หน่วยและกด "บันทึกผลิตเสร็จสิ้น", Then ระบบสร้าง Batch ใหม่สำเร็จ ผูก Lot ที่เลือกทั้งหมดเข้ากับ Batch และหักยอด stock วัตถุดิบจริงตามจำนวนที่ใช้ทันที — ต้องไม่พบข้อความ error เกี่ยวกับการค้นหา Lot ไม่พบอีกต่อไป
- AC3 (Edge) — ปรับ Lot ที่เสนอ (multi-lot): Given ระบบเสนอ Lot เดียวสำหรับวัตถุดิบชนิดหนึ่งแต่ Lot นั้นเหลือไม่พอ (remaining < required), When ฉันแก้ไขโดยเพิ่ม Lot ที่สองมาเสริมให้ครบจำนวนที่ต้อง, Then ระบบยอมรับการปรับ บันทึกทั้ง 2 Lot เป็นแหล่งที่มาของ Batch และรวมยอดที่เบิกจริงต้องเท่ากับจำนวนที่ BOM กำหนดพอดี (ไม่ขาด/ไม่เกิน)
- AC4 (Error) — ไม่มี BOM: Given สินค้าที่ผลิตยังไม่มี BOM ผูกในระบบ, When ฉันเปิดหน้าบันทึกผลิตสำหรับ Production Order ของสินค้านั้น, Then ระบบไม่แสดงตัวเลขคำนวณและไม่อนุญาตให้บันทึกผลิตเสร็จสิ้น พร้อมข้อความ "สินค้านี้ยังไม่มีสูตรการผลิต (BOM) ในระบบ กรุณาติดต่อผู้ดูแลระบบ" (สอดคล้อง ECP-009 AC3, แก้ผ่าน ECP-039)
- AC5 (Error) — ปรับจนไม่พอ: Given ฝ่ายผลิตปรับลด/แก้ไขจำนวน lot ที่เลือกจนรวมแล้วน้อยกว่าจำนวนที่ BOM กำหนดต้องใช้จริง, When ฉันกด "บันทึกผลิตเสร็จสิ้น", Then ระบบไม่สร้าง Batch และแสดงข้อความ "ปริมาณวัตถุดิบ [ชื่อ] ที่เลือกไม่ครบตามสูตร (ต้องการ [X] มีเลือกไว้ [Y])"

### ECP-014: ค้นหา traceability เต็มสาย ผ่านช่องค้นหาเดียว (auto-detect Lot/Batch/PO/Invoice) พร้อมคำอธิบาย Lot vs Batch
**Priority:** Must (KPI #2, DoD) | Depends on: ECP-013

**[แก้ไขทั้ง story รอบ Gate 2 — defect: ค้นหา Lot ไม่เจอ (feedback ข้อ 3a) + scope ใหม่: multi-type search + legend (feedback ข้อ 3b/3c)]**

ในฐานะผู้ใช้ที่เกี่ยวข้อง (คลัง/ฝ่ายผลิต/QA/QC/Admin) ฉันต้องการค้นหาความเชื่อมโยงแบบเต็มสายผ่านช่อง
ค้นหาเดียวไม่ว่าจะถือเลข Lot, Batch, PO หรือ Invoice อยู่ในมือ พร้อมเข้าใจความแตกต่างของ Lot กับ
Batch จากหน้าจอเอง เพื่อสอบสวนย้อนกลับได้รวดเร็วเมื่อจำเป็น

- AC1 (Happy) — แก้ defect + auto-detect: Given Batch #B2026-010 ถูกผลิตจาก Lot #L-SEED-1 และ #L2026-002 ผูกกับ PO #PO-2026-005 และมี Invoice #INV-0001 ออกจาก PO นี้แล้ว, When ฉันพิมพ์ "L-SEED-1" ในช่องค้นหาเดียวของหน้า trace แล้วกดค้นหา, Then ระบบ auto-detect ว่าเป็นเลข Lot (ตามรูปแบบ ADR-006) และแสดงผลลัพธ์ห่วงโซ่เต็ม (Lot → Batch #B2026-010 → PO #PO-2026-005 → Invoice #INV-0001) ภายใน ≤5 นาที — Lot #L-SEED-1 ต้องถูกพบสำเร็จ ไม่ error ค้นหาไม่เจอเหมือนที่ปอนด์รายงาน
- AC2 (Happy) — ค้นด้วยเลขประเภทอื่นได้ผลลัพธ์เดียวกัน: Given ห่วงโซ่เดียวกับ AC1, When ฉันค้นหาด้วยเลข Batch "B2026-010" หรือเลข PO "PO-2026-005" หรือเลข Invoice "INV-0001" (คนละครั้ง คนละ field), Then ระบบ auto-detect ประเภทของแต่ละคำค้นหาได้ถูกต้องและแสดงผลลัพธ์ห่วงโซ่เต็มชุดเดียวกันทุกครั้ง ไม่ว่าจะเริ่มค้นจากจุดใดในห่วงโซ่
- AC3 (Edge) — legend Lot vs Batch: Given ฉันเปิดหน้าผลลัพธ์ trace ของห่วงโซ่ใดก็ตาม, When ผลลัพธ์แสดงขึ้น, Then หน้าจอต้องมี legend/คำอธิบายที่มองเห็นได้เสมอ อธิบายว่า "Lot = ล็อตวัตถุดิบที่รับเข้าจาก supplier (ต้นทาง)" และ "Batch = รอบการผลิตที่นำ Lot มาแปลงเป็นสินค้าสำเร็จรูป (ปลายทาง)" คู่กับ timeline/ผังทิศทาง Lot→Batch→สินค้าสำเร็จรูปในหน้าเดียวกัน (ตอบคำถามปอนด์ "Batch เอาไว้ทำอะไร")
- AC4 (Edge) — Lot ใช้ในหลาย Batch: Given Lot วัตถุดิบหนึ่งถูกใช้ในหลาย Batch ที่ต่างกัน, When ฉันค้นหาด้วยเลข Lot นั้น, Then ระบบแสดงผลลัพธ์ครบทุก Batch ที่เคยใช้ Lot นี้ ไม่ใช่แค่ Batch ล่าสุด
- AC5 (Error): Given ฉันพิมพ์คำค้นหาที่ไม่ตรงกับรูปแบบ Lot/Batch/PO/Invoice number ใดๆ เลย หรือค้นด้วยเลขที่ auto-detect ประเภทได้แต่ไม่พบจริงในระบบ, When ฉันกดค้นหา, Then ระบบแสดงข้อความ "ไม่พบข้อมูลที่ตรงกับคำค้นหา กรุณาตรวจสอบเลข Lot/Batch/PO/Invoice อีกครั้ง" แทนหน้าว่างเปล่า/error ทางเทคนิค

## Epic 5: QA/QC

> ⚠ **ข้อสมมติที่ต้องยืนยันจากปอนด์:** stories ด้านล่างยึดหลักว่า Batch ต้องมีสถานะ QA Approved
> ก่อนจึงจะสร้าง Shipment ได้เสมอ และเกณฑ์ผ่าน/ไม่ผ่านเป็นดุลยพินิจของเจ้าหน้าที่ QA (manual
> judgment) ตาม GMP domain default ไม่ block งาน

### ECP-015: บันทึกผลตรวจสอบคุณภาพและอนุมัติ/ไม่อนุมัติ Batch
**Priority:** Must (GMP requirement แฝง) | Depends on: ECP-013

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการบันทึกผลตรวจสอบคุณภาพของ Batch และอนุมัติหรือไม่อนุมัติ เพื่อควบคุมว่ามีเฉพาะ Batch ที่ผ่านมาตรฐานเท่านั้นที่จัดส่งได้

- AC1 (Happy): Given Batch #B2026-010 สถานะ "QC Pending", When ฉันบันทึกผลตรวจเป็น "Approved" พร้อมหมายเหตุ, Then สถานะ Batch เปลี่ยนเป็น "QC Approved" ทันที และ Batch นี้ปรากฏในรายการที่ฝ่ายจัดส่งสามารถเลือกสร้าง Shipment ได้
- AC2 (Edge): Given Batch #B2026-011 สถานะ "QC Pending", When ฉันบันทึกผลตรวจเป็น "Rejected" พร้อมเหตุผล "พบสิ่งปนเปื้อน", Then สถานะ Batch เปลี่ยนเป็น "QC Rejected" ทันที, Batch นี้ไม่ปรากฏในรายการที่ฝ่ายจัดส่งเลือกได้ และสถานะ PO ที่เกี่ยวข้องแสดงเป็น "รอผลิตใหม่ (QC ไม่ผ่าน)"
- AC3 (Error): Given Batch #B2026-012 มีสถานะ "QC Approved" ไปแล้ว, When เจ้าหน้าที่ QA พยายามบันทึกผลตรวจซ้ำสำหรับ Batch เดียวกัน, Then ระบบแสดงข้อความ "Batch นี้ถูกอนุมัติไปแล้วเมื่อ [วันที่-เวลา] โดย [ชื่อผู้ตรวจ]" และต้องยืนยันอีกครั้งหากต้องการบันทึกผลใหม่ทับ

### ECP-016: ดูประวัติ/สถานะ QC ของ Batch ทั้งหมด
**Priority:** Should | Depends on: ECP-015

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการดูรายการ Batch ทั้งหมดพร้อมสถานะ QC เพื่อติดตามงานที่ค้างตรวจและตรวจสอบย้อนหลังได้

- AC1 (Happy): Given มี Batch 10 รายการ อยู่ในสถานะ QC Pending 4 รายการ, Approved 5 รายการ, Rejected 1 รายการ, When ฉันเปิดหน้ารายการ Batch และกรองด้วยสถานะ "QC Pending", Then ระบบแสดงเฉพาะ 4 รายการที่รอตรวจ เรียงจากวันที่ผลิตเก่าสุดไปใหม่สุด
- AC2 (Edge): Given ไม่มี Batch ใดอยู่ในสถานะ "QC Pending" ในขณะนั้น, When ฉันกรองด้วยสถานะ "QC Pending", Then ระบบแสดงข้อความ "ไม่มีงานตรวจสอบที่ค้างอยู่ในขณะนี้"
- AC3 (Error): Given ฉันพยายามเข้าถึงหน้ารายการ Batch ด้วยบัญชีผู้ใช้ที่ไม่มีสิทธิ์ role QA/QC, When ฉันเปิด URL หน้านี้ตรงๆ, Then ระบบปฏิเสธการเข้าถึงและแสดงข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-017: บันทึกผลตรวจสอบวัตถุดิบขาเข้าก่อนใช้ผลิต (Incoming QC)
**Priority:** Must **[ยกระดับรอบ Gate 2 — เดิม Could, feedback ข้อ 5]** | Depends on: ECP-008

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการเปิดฟอร์มตรวจสอบวัตถุดิบที่รับเข้าคลัง (แสดงจำนวน/Lot ที่บันทึก
ไว้แล้วจาก ECP-008 พร้อมชื่อผู้จำหน่าย) และบันทึกผลตรวจก่อนอนุญาตให้นำไปใช้ผลิต เพื่อป้องกันการใช้
วัตถุดิบที่มีปัญหาตั้งแต่ต้นทาง

- AC1 (Happy): Given Lot วัตถุดิบใหม่ถูกบันทึกรับเข้า (ECP-008) พร้อมจำนวน/supplier_name สถานะเริ่มต้นเป็น "รอตรวจสอบ", When ฉันเปิดฟอร์มตรวจสอบขาเข้าของ Lot นี้ (เห็นจำนวน/เลข Lot/ผู้จำหน่ายที่ระบบดึงมาให้อัตโนมัติ) แล้วบันทึกผลตรวจเป็น "ผ่าน", Then Lot นั้นเปลี่ยนสถานะเป็น "พร้อมใช้ผลิต" และปรากฏให้เลือกใน ECP-013
- AC2 (Edge): Given ฉันบันทึกผลตรวจ Lot หนึ่งเป็น "ไม่ผ่าน", When ฝ่ายผลิตพยายามเลือก Lot นั้นในการบันทึกผลิต (ECP-013), Then ระบบไม่แสดง Lot นั้นเป็นตัวเลือกที่ระบบเสนอ และหากพยายามเลือกผ่าน API/URL ตรง ระบบปฏิเสธพร้อมข้อความ "Lot นี้ไม่ผ่านการตรวจสอบคุณภาพ ไม่สามารถนำไปใช้ผลิตได้"
- AC3 (Error): Given Lot วัตถุดิบยังไม่มีการบันทึกผลตรวจสอบใดๆ (สถานะ "รอตรวจสอบ"), When ฝ่ายผลิตพยายามเลือก Lot นั้นในการบันทึกผลิต, Then ระบบไม่อนุญาตและแสดงข้อความ "Lot นี้ยังไม่ผ่านการตรวจสอบคุณภาพขาเข้า"
- AC4 (Error) **[เพิ่มรอบ Gate 2]**: Given Lot วัตถุดิบเก่าที่ถูกบันทึกรับเข้าไว้ก่อนที่ระบบจะมีฟิลด์ supplier_name (ข้อมูลเก่าไม่ครบ), When เจ้าหน้าที่ QA เปิดฟอร์มตรวจสอบขาเข้าของ Lot นี้, Then ระบบยังคงแสดงฟอร์มตรวจสอบได้ตามปกติ (ไม่ block การตรวจ) โดยแสดงช่องผู้จำหน่ายเป็น "ไม่ระบุ" แทนการทำให้ทั้งฟอร์ม error

## Epic 6: Shipping (Logistics)

### ECP-018: สร้างบันทึกการจัดส่งจาก Batch ที่ผ่านการอนุมัติ QA แล้ว
**Priority:** Must | Depends on: ECP-015

ในฐานะฝ่ายจัดส่ง ฉันต้องการสร้างบันทึกการจัดส่งจาก Batch ที่ผ่าน QA แล้วเท่านั้น เพื่อรับประกันว่าไม่มีสินค้าที่ไม่ผ่านคุณภาพหลุดออกไปถึงลูกค้า

- AC1 (Happy): Given Batch #B2026-010 สถานะ "QC Approved" ผูกกับ PO ของลูกค้า ABC, When ฉันเลือก Batch นั้นและกด "สร้างบันทึกจัดส่ง" ระบุวันที่จัดส่งและจำนวน, Then ระบบสร้าง Shipment สำเร็จ สถานะ "Shipped" และอัปเดตสถานะ PO เป็น "Shipped"
- AC2 (Edge): Given Batch #B2026-011 สถานะ "QC Rejected", When ฉันพยายามค้นหา Batch นี้ในหน้าสร้างบันทึกจัดส่ง, Then ระบบไม่แสดง Batch นี้เป็นตัวเลือกให้จัดส่งเลย
- AC3 (Error): Given Batch #B2026-012 สถานะ "QC Pending", When ฉันพยายามเรียก action สร้างบันทึกจัดส่งสำหรับ Batch นี้โดยตรง, Then ระบบปฏิเสธและแสดงข้อความ "Batch นี้ยังไม่ผ่านการอนุมัติจาก QA/QC ไม่สามารถจัดส่งได้"

### ECP-019: ติดตาม/อัปเดตสถานะการจัดส่ง
**Priority:** Should | Depends on: ECP-018

ในฐานะฝ่ายจัดส่ง ฉันต้องการอัปเดตสถานะการจัดส่งเมื่อสินค้าถึงมือลูกค้า เพื่อให้ทุกแผนกเห็นสถานะล่าสุดตรงกัน

- AC1 (Happy): Given Shipment สถานะ "Shipped" ถูกส่งออกไปแล้ว, When ฉันอัปเดตสถานะเป็น "Delivered" พร้อมวันที่ส่งถึง, Then ระบบบันทึกสถานะใหม่ทันทีและ PO ที่เกี่ยวข้องแสดงสถานะ "Delivered" ในหน้าติดตามของ Sales/CS
- AC2 (Edge): Given Shipment หนึ่งรายการยังไม่ถึงสถานะ "Shipped" (ยังเป็น Draft), When ฉันพยายามอัปเดตสถานะเป็น "Delivered" ข้ามขั้นตอน, Then ระบบไม่อนุญาตและแสดงข้อความ "ไม่สามารถเปลี่ยนเป็น Delivered ได้ ต้องมีสถานะ Shipped ก่อน"
- AC3 (Error): Given ฉันกรอกวันที่ส่งถึงลูกค้าเป็นวันที่ในอนาคต, When ฉันกดบันทึก, Then ระบบแสดงข้อความ "วันที่ส่งถึงต้องไม่เกินวันที่ปัจจุบัน" และไม่บันทึกค่านั้น

## Epic 7: Invoice & Billing (Finance)

### ECP-020: ออก invoice จาก PO ที่จัดส่งแล้ว (รวมส่วนลด)
**Priority:** Must | Depends on: ECP-018

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการออก invoice จาก PO ที่จัดส่งสำเร็จแล้ว พร้อมระบุส่วนลด (ถ้ามี)
เพื่อเรียกเก็บเงินจากลูกค้าด้วยยอดที่ถูกต้อง

- AC1 (Happy): Given PO สถานะ "Shipped" มียอดรวมก่อนภาษี (subtotal) 50,000 บาท ยังไม่เคยออก invoice ในสายของ PO นี้มาก่อน และ ณ ขณะนั้นค่า VAT rate ที่ Admin ตั้งค่าไว้คือ 7%, When ฉันกด "ออก invoice" สำหรับ PO นั้นโดยไม่ระบุส่วนลด, Then ระบบสร้าง invoice version 1 (parent_invoice_id = null) พร้อมเลขที่ invoice ไม่ซ้ำกัน, subtotal 50,000.00 บาท, discount_amount 0.00 บาท, VAT 7% = 3,500.00 บาท, total 53,500.00 บาท, สถานะ "Issued" และสถานะ PO เปลี่ยนเป็น "Invoiced"
- AC2 (Edge): Given PO นั้นมี invoice ที่ออกไปแล้วก่อนหน้า (สถานะ PO = "Invoiced"), When ฉันพยายามกด "ออก invoice" ซ้ำสำหรับ PO เดียวกัน, Then ระบบไม่สร้าง invoice สายใหม่และแสดงข้อความ "PO นี้มี invoice เลขที่ [เลขที่] (version [N]) ออกไปแล้วเมื่อ [วันที่] หากต้องการแก้ไขกรุณาใช้ฟังก์ชันแก้ไข invoice แทน"
- AC3 (Error): Given PO ยังมีสถานะ "InProduction" (ยังไม่ได้จัดส่ง), When ฉันพยายามออก invoice สำหรับ PO นั้น, Then ระบบไม่อนุญาตและแสดงข้อความ "ไม่สามารถออก invoice ได้ PO นี้ยังไม่ถูกจัดส่ง"
- AC4 (Edge) **[เพิ่มรอบ Gate 2 — discount_amount, feedback ข้อ 10b]**: Given PO เดียวกับ AC1 subtotal 50,000 บาท, When ฝ่ายบัญชีกรอกส่วนลดคงที่ (discount_amount) 2,000 บาทตอนออก invoice แล้วกด "ออก invoice", Then ระบบคำนวณยอดหลังหักส่วนลด = 50,000 − 2,000 = 48,000.00 บาท, VAT 7% ของยอดหลังหักส่วนลด = 3,360.00 บาท, total = 51,360.00 บาท และแสดง subtotal/discount/VAT/total แยกกันชัดเจนในหน้ารายละเอียด invoice (ECP-040)
- AC5 (Error) **[เพิ่มรอบ Gate 2]**: Given subtotal ของ PO คือ 10,000 บาท, When ฝ่ายบัญชีกรอก discount_amount = 12,000 บาท (มากกว่า subtotal), Then ระบบไม่อนุญาตให้ออก invoice และแสดงข้อความ "ส่วนลดต้องไม่เกินยอดรวมก่อนหักส่วนลด (subtotal 10,000.00 บาท)"

### ECP-021: บันทึกการรับชำระเงิน (manual reconcile)
**Priority:** Must | Depends on: ECP-020

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการบันทึกการรับชำระเงินของ invoice ด้วยมือ เพื่ออัปเดตสถานะการเรียกเก็บเงินโดยไม่ต้องเชื่อม payment gateway จริง

- AC1 (Happy): Given invoice เลขที่ INV-0001 ยอด 50,000 บาท สถานะ "Issued", When ฉันบันทึกรับชำระเต็มจำนวน 50,000 บาท พร้อมวันที่รับเงิน, Then สถานะ invoice เปลี่ยนเป็น "Paid" ทันทีและแสดงยอดคงค้างเป็น 0.00 บาท
- AC2 (Edge): Given invoice เลขที่ INV-0002 ยอด 30,000 บาท สถานะ "Issued", When ฉันบันทึกรับชำระบางส่วน 20,000 บาท, Then สถานะ invoice เปลี่ยนเป็น "Partially Paid" และแสดงยอดคงค้าง 10,000.00 บาทอย่างถูกต้อง
- AC3 (Error): Given invoice เลขที่ INV-0003 ยอด 10,000 บาท, When ฉันพยายามบันทึกรับชำระเป็นจำนวน 15,000 บาท, Then ระบบไม่บันทึกรายการและแสดงข้อความ "จำนวนเงินที่รับชำระเกินยอดคงค้างของ invoice นี้ (คงค้าง 10,000.00 บาท)"

### ECP-022: ดูสถานะ invoice/การเรียกเก็บเงินทั้งหมด
**Priority:** Should | Depends on: ECP-020, ECP-021

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการดูรายการ invoice ทั้งหมดพร้อมสถานะการชำระเงิน เพื่อติดตามลูกหนี้ที่ค้างชำระ

- AC1 (Happy): Given มี invoice 5 ใบ สถานะ Paid 3 ใบ, Partially Paid 1 ใบ, Issued 1 ใบ, When ฉันกรองด้วยสถานะ "ยังไม่ชำระครบ" (Issued + Partially Paid), Then ระบบแสดงเฉพาะ 2 ใบที่ตรงเงื่อนไข พร้อมยอดคงค้างของแต่ละใบ
- AC2 (Edge): Given ไม่มี invoice ใดค้างชำระเลยในขณะนั้น, When ฉันกรองด้วยสถานะ "ยังไม่ชำระครบ", Then ระบบแสดงข้อความ "ไม่มี invoice ที่ค้างชำระในขณะนี้"
- AC3 (Error): Given ฉันพยายามเข้าถึงหน้ารายการ invoice ด้วยบัญชีที่ไม่มีสิทธิ์ role บัญชี/การเงินหรือ Admin, When ฉันเปิด URL หน้านี้ตรงๆ, Then ระบบปฏิเสธการเข้าถึงและแสดงข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-040: ดูรายละเอียด invoice ใบเดียว **(ใหม่รอบ Gate 2 — แก้ defect "เปิดดูไม่ได้", feedback ข้อ 10a)**
**Priority:** Must | Depends on: ECP-020

ในฐานะฝ่ายบัญชี/การเงิน (และ role อื่นที่มีสิทธิ์เกี่ยวข้อง) ฉันต้องการเปิดดูรายละเอียด invoice ใบใด
ใบหนึ่งแบบเจาะจง เพื่อตรวจสอบรายการ/ยอดเงิน/สถานะการชำระก่อนติดตามหรือพิมพ์เอกสาร

- AC1 (Happy): Given invoice เลขที่ INV-0001 version ล่าสุด สถานะ "Issued" subtotal 50,000.00, discount 2,000.00, VAT 3,360.00, total 51,360.00, When ฉันคลิกที่ invoice นี้จากหน้ารายการ (ECP-022) หรือเปิด URL รายละเอียดโดยตรง, Then ระบบแสดงหน้ารายละเอียดครบ: ข้อมูลลูกค้า, รายการสินค้าทุกบรรทัด (ชื่อ/จำนวน/ราคา/ยอดรวมต่อบรรทัด), subtotal, discount, VAT, total, สถานะ, ประวัติการชำระเงิน (ถ้ามี) ภายในหน้าเดียว
- AC2 (Edge): Given invoice นี้มีมากกว่า 1 version (version ก่อนหน้าเป็น Superseded), When ฉันเปิดหน้ารายละเอียด invoice ของ version ล่าสุด, Then ระบบแสดง version ล่าสุดเป็นค่าเริ่มต้นพร้อมลิงก์ "ดูประวัติ version" ให้สลับไปดู version เก่าที่ Superseded ได้ โดยไม่ปะปนตัวเลขของสอง version เข้าด้วยกัน
- AC3 (Error): Given invoice ID ที่ระบุใน URL ไม่มีอยู่จริงในระบบ, When ฉันเปิดหน้ารายละเอียดด้วย ID นั้น, Then ระบบแสดงข้อความ "ไม่พบ invoice นี้ในระบบ" แทนหน้าเปล่า/error 500 (แก้ defect เดิม)
- AC4 (Error): Given ผู้ใช้ role ที่ไม่มีสิทธิ์ดู invoice (เช่น ฝ่ายจัดส่ง) พยายามเปิด URL รายละเอียด invoice ตรง, When เปิดหน้านี้, Then ระบบปฏิเสธการเข้าถึงพร้อมข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

## Epic 8: User Management & Audit Log (Admin)

### ECP-023: สร้าง/แก้ไขผู้ใช้งานและกำหนด role
**Priority:** Must

ในฐานะ Admin ฉันต้องการสร้างและแก้ไขบัญชีผู้ใช้งานพร้อมกำหนด role ให้ตรงกับหน้าที่ เพื่อควบคุมการเข้าถึงระบบตาม 7 roles ที่กำหนด

- AC1 (Happy): Given ยังไม่มีบัญชีผู้ใช้ชื่อ "somchai", When ฉันสร้างผู้ใช้ใหม่ username "somchai" กำหนด role "ฝ่ายผลิต" และสถานะ "Active", Then ระบบสร้างบัญชีสำเร็จ พร้อมกำหนดรหัสผู้ใช้ (User ID) ให้อัตโนมัติแบบไม่ซ้ำกัน และผู้ใช้ "somchai" สามารถ login เข้าใช้งานเมนู/สิทธิ์ตาม role "ฝ่ายผลิต" ได้ทันที
- AC2 (Edge): Given ผู้ใช้ "somchai" มี role "ฝ่ายผลิต" อยู่แล้วและกำลัง login ใช้งานอยู่ในขณะนั้น, When Admin เปลี่ยน role ของ "somchai" เป็น "QA/QC", Then เมนู/สิทธิ์การเข้าถึงของ "somchai" เปลี่ยนตาม role ใหม่ภายในการเข้าใช้งานครั้งถัดไป
- AC3 (Error): Given ฉันพยายามสร้างผู้ใช้ใหม่ด้วย username ที่ซ้ำกับผู้ใช้ที่มีอยู่แล้ว, When ฉันกดบันทึก, Then ระบบไม่สร้างบัญชีใหม่และแสดงข้อความ "username นี้มีผู้ใช้งานอยู่แล้วในระบบ"
- AC4 (Edge): Given มีการส่งคำขอสร้างผู้ใช้ที่พยายามระบุค่า user_id มาเองจากฝั่ง client, When ระบบรับคำขอสร้างผู้ใช้นั้น, Then ระบบเพิกเฉยค่า user_id ที่ส่งมาทั้งหมดและ generate รหัสใหม่ให้เองเสมอ

### ECP-024: กำหนด/ปรับสิทธิ์ (permission) ต่อ role ผ่านหน้าจอ config
**Priority:** Must

ในฐานะ Admin ฉันต้องการกำหนด/ปรับสิทธิ์การเข้าถึงเมนูและการทำรายการของแต่ละ role ผ่านหน้าจอ config โดยไม่ต้องแก้โค้ดหรือ deploy ใหม่ เพื่อให้ปรับ role ได้เองระหว่างช่วงสำรวจ user journey

- AC1 (Happy): Given role "ฝ่ายจัดส่ง" ปัจจุบันไม่มีสิทธิ์เข้าเมนู "รายงานการเงิน", When ฉันเปิดหน้า config สิทธิ์ของ role "ฝ่ายจัดส่ง" และเปิดสิทธิ์ "ดูรายงานการเงิน" แล้วกดบันทึก, Then ผู้ใช้ที่มี role "ฝ่ายจัดส่ง" มองเห็นเมนู "รายงานการเงิน" ได้ทันทีในการเข้าใช้งานครั้งถัดไป
- AC2 (Edge): Given Admin พยายามถอดสิทธิ์การเข้าถึงหน้า "จัดการผู้ใช้งาน" ออกจาก role "Admin" ทั้งหมด, When ฉันกดบันทึกการเปลี่ยนแปลงนี้, Then ระบบไม่อนุญาตให้บันทึกและแสดงข้อความ "ไม่สามารถถอดสิทธิ์นี้ออกจาก role Admin ได้ เนื่องจากจะทำให้ไม่มีผู้ใช้ที่เข้าถึงหน้าจัดการสิทธิ์ได้อีก"
- AC3 (Error): Given ฉันกำลังแก้ไขสิทธิ์ของ role หนึ่งอยู่ แต่การเชื่อมต่อขาดหายระหว่างกดบันทึก, When ระบบไม่สามารถบันทึกการเปลี่ยนแปลงได้สำเร็จ, Then ระบบแสดงข้อความ "บันทึกการเปลี่ยนแปลงไม่สำเร็จ กรุณาลองใหม่" และสิทธิ์เดิมก่อนแก้ไขยังคงมีผลอยู่

### ECP-025: บันทึก audit log ทุกครั้งที่ login
**Priority:** Must

ในฐานะ Admin ฉันต้องการให้ระบบบันทึก audit log ทุกครั้งที่มีผู้ใช้ login (สำเร็จหรือไม่สำเร็จ) เพื่อตรวจสอบย้อนหลังด้านความปลอดภัยได้

- AC1 (Happy): Given ผู้ใช้ "somchai" กรอก username/password ถูกต้อง, When "somchai" กด login สำเร็จ, Then ระบบบันทึก audit log รายการใหม่ทันทีที่มี user_id, เวลา login, ผลลัพธ์ "Success"
- AC2 (Edge): Given ผู้ใช้ "somchai" กรอก password ผิด 1 ครั้ง, When ระบบปฏิเสธการ login, Then ระบบยังคงบันทึก audit log รายการนั้นด้วยผลลัพธ์ "Failed"
- AC3 (Error): Given ระบบบันทึก audit log ขัดข้องชั่วคราวขณะมีการ login, When เหตุการณ์ login เกิดขึ้นแต่การเขียน log ล้มเหลว, Then ระบบต้องไม่ปล่อยให้ผู้ใช้ login สำเร็จโดยไม่มีการพยายามบันทึก log เลย

### ECP-026: บันทึก audit log ทุกครั้งที่ทำรายการสำคัญและดูย้อนหลังได้
**Priority:** Must | Depends on: ECP-025

ในฐานะ Admin ฉันต้องการให้ระบบบันทึก audit log ทุกครั้งที่มีการสร้าง/แก้ไข/อนุมัติ/ยกเลิกรายการสำคัญ และสามารถค้นดูย้อนหลังได้ เพื่อตรวจสอบความรับผิดชอบตาม GMP/governance

- AC1 (Happy): Given เจ้าหน้าที่ QA อนุมัติ Batch #B2026-010, When การอนุมัตินั้นเกิดขึ้นสำเร็จ, Then ระบบบันทึก audit log ที่มี user_id ของผู้อนุมัติ, action = "Approve Batch", entity, เวลาที่ทำรายการ
- AC2 (Edge): Given มี audit log มากกว่า 1,000 รายการสะสมอยู่ในระบบ, When Admin ค้นหา log ด้วยช่วงวันที่และประเภทรายการเฉพาะ, Then ระบบแสดงเฉพาะรายการที่ตรงเงื่อนไขทั้งหมด
- AC3 (Error): Given Admin พยายามลบหรือแก้ไขรายการ audit log ที่มีอยู่แล้ว, When ระบบได้รับคำสั่งลบ/แก้ไข audit log, Then ระบบปฏิเสธการลบ/แก้ไขเสมอและแสดงข้อความ "ไม่สามารถแก้ไขหรือลบ audit log ได้"

- AC2 (Edge): Given ผู้ใช้ทดสอบคนหนึ่งทำ task ไม่จบภายในเวลาที่กำหนดของ scenario, When QA บันทึกผลการทดสอบ, Then ต้องบันทึกจุดที่ผู้ใช้ติดขัดเป็นข้อมูล user journey ที่แท้จริงของ role นั้น
- AC3 (Error): Given จำนวนผู้ใช้ทดสอบจริงต่อ role สำหรับรอบ UAT ได้รับคำตอบจากปอนด์แล้ว (2-3 คน/role), When ทีม QA วางแผน UAT, Then QA ใช้ตัวเลขนี้ในการนัดหมายผู้ทดสอบโดยไม่ต้องถามซ้ำ

## Epic 11: Gate 1 Amendment — Invoice Versioning & VAT Config

### ECP-037: แก้ไข invoice ที่ออกแล้ว → สร้าง version ใหม่พร้อม reference กลับไปยัง parent invoice
**Priority:** Must | Depends on: ECP-020

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการแก้ไขรายละเอียด invoice ที่ออกไปแล้วโดยระบบสร้างเป็น version ใหม่เสมอแทนการเขียนทับของเดิม เพื่อให้ตรวจสอบย้อนกลับได้ว่าแก้ไขจาก version ใดเมื่อไหร่และใคร

- AC1 (Happy): Given invoice เลขที่ INV-0001 version 1 สถานะ "Issued" (ยังไม่มีการชำระเงินใดๆ), When ฝ่ายบัญชีแก้ไขจำนวนสินค้าในบรรทัด invoice แล้วกด "บันทึกการแก้ไข", Then ระบบสร้าง invoice version 2 ใหม่พร้อม parent_invoice_id ชี้กลับไปยัง version 1, เปลี่ยนสถานะของ version 1 เป็น "Superseded" (read-only) และ version 2 กลายเป็น invoice ที่ใช้เรียกเก็บเงินจริงต่อไป
- AC2 (Edge): Given invoice ของ PO หนึ่งมี version ล่าสุดคือ version 2, When ผู้ใช้เปิดดูประวัติ invoice ของ PO นั้น, Then ระบบแสดงทั้ง version 1 และ version 2 เรียงตามลำดับเวลาในไทม์ไลน์เดียวกัน โดย version 1 แสดงป้าย "ถูกแทนที่โดย version 2 เมื่อ [วันที่-เวลา]"
- AC3 (Error): Given invoice ของ PO หนึ่งมี version ล่าสุดคือ version 2, When ผู้ใช้พยายามแก้ไข version 1 โดยตรง, Then ระบบไม่อนุญาตให้แก้ไขและแสดงข้อความ "invoice นี้ถูกแทนที่ด้วย version ใหม่กว่าแล้ว (version 2) กรุณาแก้ไขที่ version ล่าสุดแทน"
- AC4 (Error): Given ฉันกำลังแก้ไข invoice version ล่าสุดและลบรายการออกจนไม่เหลือรายการสินค้าใดเลย, When ฉันกดบันทึกการแก้ไข, Then ระบบไม่สร้าง version ใหม่และแสดงข้อความ "invoice ต้องมีอย่างน้อย 1 รายการสินค้า"

> ⚠ **ข้อสมมติที่ต้องยืนยันจากปอนด์ (ไม่ block):** ห้ามแก้ไข invoice ที่มีการบันทึกรับชำระแล้วหรือไม่ — default อนุญาตแก้ไขได้เสมอตราบเท่าที่เป็น version ล่าสุด

### ECP-038: Admin ตั้งค่า VAT rate (default 7%) ใน Admin Portal
**Priority:** Must | Depends on: ECP-023

ในฐานะ Admin ฉันต้องการตั้งค่าอัตรา VAT (default 7%) ในหน้า Admin Portal เดียวกับหน้าจัดการผู้ใช้งาน เพื่อให้ invoice ที่ออกใหม่คำนวณ VAT ตามค่าที่ตั้งไว้ ณ เวลาที่ออกเอกสารเสมอ

- AC1 (Happy): Given ค่า VAT rate ปัจจุบันในระบบคือ 7%, When Admin เปิดหน้า Admin Portal ส่วน "ตั้งค่า VAT" เปลี่ยนค่าเป็น 10% แล้วกดบันทึก, Then ระบบบันทึกค่า VAT rate ใหม่ = 10% ทันที และ invoice ทุกใบที่ออกใหม่หลังจากนี้คำนวณ VAT ที่อัตรา 10%
- AC2 (Edge): Given มี invoice INV-0001 ที่ออกไปแล้วก่อนหน้าด้วย VAT rate เดิม 7%, When Admin เปลี่ยนค่า VAT rate เป็น 10% ภายหลัง, Then invoice INV-0001 ยังคงแสดงยอด VAT/total เดิมที่คำนวณด้วย 7% เหมือนเดิมทุกประการ (snapshot)
- AC3 (Error): Given Admin กำลังแก้ไขค่า VAT rate, When Admin กรอกค่าเป็นค่าติดลบหรือค่าที่มากกว่า 100 แล้วกดบันทึก, Then ระบบไม่บันทึกค่าที่กรอกและแสดงข้อความ "อัตรา VAT ต้องอยู่ระหว่าง 0% ถึง 100%"

โดยตรง (ไม่ต้องพึ่ง developer/seed data) เพื่อให้ระบบตรวจสอบ stock และคำนวณวัตถุดิบที่ต้องใช้ในการ

- AC2 (Happy) — แก้ไข in-place (ไม่มี version history ตามที่ยืนยันเป็น out-of-scope): Given สินค้าหนึ่งมี BOM อยู่แล้ว 2 บรรทัด, When ฉันแก้ไขปริมาณต่อหน่วยของบรรทัดหนึ่งจาก 2 กก. เป็น 2.5 กก. แล้วกดบันทึก, Then ระบบอัปเดตค่าปริมาณของบรรทัดนั้นทันทีในที่เดิม (ไม่สร้าง version ใหม่/ไม่เก็บประวัติเดิม) และการคำนวณ stock check/production auto-calc ครั้งถัดไปใช้ค่าใหม่ทันที
- AC3 (Edge) — ลบบรรทัดสูตร: Given BOM ของสินค้าหนึ่งมี 3 บรรทัด, When ฉันลบบรรทัดวัตถุดิบหนึ่งออกแล้วกดบันทึก, Then ระบบอัปเดต BOM ให้เหลือ 2 บรรทัด และแจ้งเตือนยืนยันก่อนลบ ("ยืนยันการลบวัตถุดิบนี้ออกจากสูตร?")
- AC4 (Error) — ลบจนไม่เหลือบรรทัดใดเลย: Given BOM ของสินค้าหนึ่งมี 1 บรรทัดสุดท้าย, When ฉันพยายามลบบรรทัดสุดท้ายนี้จนไม่เหลือวัตถุดิบใดในสูตรแล้วกดบันทึก, Then ระบบไม่บันทึกการเปลี่ยนแปลงนี้และแสดงข้อความ "BOM ต้องมีวัตถุดิบอย่างน้อย 1 รายการ"
- AC5 (Error) — เพิ่มวัตถุดิบซ้ำในสูตรเดียวกัน: Given ฉันกำลังเพิ่มบรรทัดสูตรใหม่ให้กับ BOM ที่มีวัตถุดิบ "น้ำมันมะพร้าว" อยู่แล้ว, When ฉันพยายามเพิ่ม "น้ำมันมะพร้าว" ซ้ำเป็นอีกบรรทัดหนึ่ง, Then ระบบไม่อนุญาตและแสดงข้อความ "วัตถุดิบนี้มีอยู่ในสูตรแล้ว กรุณาแก้ไขปริมาณของบรรทัดเดิมแทนการเพิ่มซ้ำ"

โลโก้) ในหน้า Admin Portal เพื่อใช้เป็นข้อมูล "ผู้ออก" บนใบกำกับภาษี/ใบแจ้งหนี้ทุกใบโดยไม่ต้องกรอก
ซ้ำทุกครั้ง

- AC1 (Happy): Given ยังไม่เคยตั้งค่า CompanyProfile มาก่อน, When Admin กรอกชื่อบริษัท, ที่อยู่เต็ม, เลขประจำตัวผู้เสียภาษี 13 หลัก, เบอร์โทร และอัปโหลดโลโก้ แล้วกดบันทึก, Then ระบบบันทึกค่า CompanyProfile สำเร็จ (ค่าเดียวระดับบริษัท) และเอกสารใบกำกับภาษีที่ออกใหม่ทุกใบใช้ข้อมูลนี้เป็นส่วน "ผู้ออก" ทันที
- AC2 (Edge): Given มี CompanyProfile ที่ตั้งค่าไว้แล้ว, When Admin แก้ไขที่อยู่บริษัทแล้วกดบันทึก, Then ระบบอัปเดตค่าทันที และใบกำกับภาษีที่ออกใหม่หลังจากนี้แสดงที่อยู่ใหม่ แต่ใบกำกับภาษีที่เคยพิมพ์/ออกไปก่อนหน้ายังคง snapshot ที่อยู่เดิม ณ ตอนออกเอกสาร (สอดคล้องหลักการเดียวกับ VATConfig snapshot)
- AC3 (Error): Given Admin กรอกเลขประจำตัวผู้เสียภาษีไม่ครบ 13 หลัก หรือมีตัวอักษรที่ไม่ใช่ตัวเลขปน, When ฉันกดบันทึก, Then ระบบไม่บันทึกค่าและแสดงข้อความ "เลขประจำตัวผู้เสียภาษีต้องเป็นตัวเลข 13 หลักเท่านั้น"
- AC4 (Error): Given ยังไม่มีการตั้งค่า CompanyProfile ใดๆ เลยในระบบ, When ฝ่ายบัญชีพยายามออก/พิมพ์เอกสารใบกำกับภาษีใบแรก, Then ระบบไม่อนุญาตให้พิมพ์เอกสารและแสดงข้อความ "กรุณาตั้งค่าข้อมูลบริษัทผู้ออกเอกสารก่อน (ติดต่อ Admin)"

### ECP-042: พิมพ์/ออกเอกสารใบแจ้งหนี้-ใบกำกับภาษี (Invoice/Tax Invoice) ตามรูปแบบมาตรฐานไทย
**Priority:** Must | Depends on: ECP-020, ECP-040, ECP-041, ECP-001

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการพิมพ์/ดาวน์โหลดเอกสารใบแจ้งหนี้/ใบกำกับภาษีของ invoice ใบใดใบ
หนึ่งในรูปแบบที่ตรงตามมาตรฐานเอกสารไทย เพื่อส่งมอบให้ลูกค้าเป็นหลักฐานทางบัญชี/ภาษีที่ถูกต้อง

- AC1 (Happy): Given invoice เลขที่ INV-0001 version ล่าสุด สถานะ "Issued" subtotal 50,000.00 บาท discount_amount 2,000.00 บาท VAT 7% = 3,360.00 บาท total = 51,360.00 บาท ลูกค้ามี tax_id/registered_address ครบ และมี CompanyProfile ตั้งค่าไว้แล้ว, When ฉันกด "พิมพ์/ดาวน์โหลด" เอกสารของ invoice นี้, Then ระบบสร้างเอกสาร (PDF/หน้าพิมพ์) ที่มีครบทุกส่วนตามตัวอย่างที่ปอนด์แนบ: หัวเอกสาร "ใบแจ้งหนี้ / ใบกำกับภาษี | Invoice / Tax Invoice" พร้อมโลโก้, ข้อมูลผู้ออก (จาก CompanyProfile), ข้อมูลลูกค้า (ชื่อ/ที่อยู่/เลขผู้เสียภาษี/โทรศัพท์), เลขที่เอกสาร, วันที่ (dd/mm/yy), เงื่อนไขชำระ/เครดิต, ตารางรายการ (ลำดับ/รายการ/จำนวน/ราคา/ราคารวม) ครบทุกบรรทัด, subtotal 50,000.00, หักส่วนลด 2,000.00, ยอดหลังหักส่วนลด 48,000.00, VAT 7% = 3,360.00, total = 51,360.00, ตัวหนังสือไทยของยอด = "ห้าหมื่นหนึ่งพันสามร้อยหกสิบบาทถ้วน", หมายเหตุเงื่อนไขชำระ, ช่องลายเซ็น 2 ช่อง ("ผู้รับใบแจ้งหนี้"/"ผู้ออกใบแจ้งหนี้" พร้อมวงเล็บชื่อ) และพื้นที่ตรายาง — ปอนด์เทียบกับตัวอย่างจริงแล้วยืนยันว่าตรงกัน
- AC2 (Edge) — ไม่มีส่วนลด: Given invoice ใบหนึ่งไม่มีการกรอก discount_amount (ค่าเริ่มต้น 0.00), When ฉันพิมพ์เอกสารของ invoice นี้, Then เอกสารยังคงแสดงบรรทัด "หักส่วนลด: 0.00 บาท" (ไม่ซ่อนบรรทัดนี้) และ "จำนวนเงินหลังหักส่วนลด" เท่ากับ subtotal เดิมพอดี
- AC3 (Edge) — invoice ที่ถูกแทนที่ (Superseded): Given invoice version 1 ถูกแทนที่ด้วย version 2 แล้ว, When ฉันพยายามพิมพ์เอกสารของ version 1 (Superseded) โดยตรง, Then ระบบยังคงอนุญาตให้พิมพ์ได้ (เพื่อตรวจสอบย้อนหลัง) แต่ต้องมีป้าย/ลายน้ำ "ยกเลิกแล้ว - ใช้ version ล่าสุดแทน (เลขที่ [เลขที่ version ล่าสุด])" แสดงชัดเจน
- AC4 (Error) — ข้อมูลลูกค้าไม่ครบ: Given invoice ของลูกค้าที่ยังไม่มี tax_id กรอกไว้ในระบบ, When ฉันพยายามพิมพ์เอกสารใบกำกับภาษีของ invoice นี้, Then ระบบไม่สร้างเอกสารและแสดงข้อความ "ลูกค้ารายนี้ยังไม่มีเลขประจำตัวผู้เสียภาษีในระบบ กรุณาแก้ไขข้อมูลลูกค้าก่อนออกใบกำกับภาษี"
- AC5 (Error) — ยอดเป็น 0 (edge เชิงคำนวณ): Given discount_amount ที่กรอกไว้เท่ากับ subtotal พอดี (ยอดหลังหักส่วนลด = 0.00 บาท), When ฉันพิมพ์เอกสาร, Then ระบบคำนวณ VAT = 0.00 บาท total = 0.00 บาท และตัวหนังสือไทยแสดง "ศูนย์บาทถ้วน" อย่างถูกต้อง ไม่เกิด error

## Epic 14: UI Consistency & Responsive (Gate 2 scope-down) **(ใหม่รอบ Gate 2 — feedback ข้อ 7/8)**

### ECP-043: ความสม่ำเสมอของ UI ในหน้าจอที่ถูกแก้ไขรอบนี้
**Priority:** Should | Depends on: ECP-004, ECP-007, ECP-013, ECP-014, ECP-017, ECP-039, ECP-040, ECP-042, ECP-001/002

ในฐานะผู้ใช้งานทุก role ที่ใช้งานหน้าจอที่ถูกแก้ไขในรอบนี้ ฉันต้องการให้หน้าจอเหล่านั้นมีรูปแบบ
spacing/typography/สถานะ empty-error ที่สม่ำเสมอกัน เพื่อให้ดูเป็นมืออาชีพขึ้นอย่างเห็นได้ชัดเมื่อ
เทียบกับรอบก่อน โดยไม่ต้องรอ redesign เต็มรูปทั้งระบบ (scope-down ตามที่ PO เสนอ)

- AC1 (Happy): Given หน้าจอทั้ง 8 หน้าที่ถูกแก้ไข/สร้างใหม่ในรอบนี้ (PO create/detail, stock, trace, production, QC incoming, BOM management, customer form, invoice detail+print) ถูกเปิดเทียบกัน, When ตรวจสอบ spacing, ขนาด/น้ำหนักตัวอักษร, และสีของปุ่ม primary/secondary, Then ทั้ง 8 หน้าใช้ชุดค่า design token เดียวกันจาก ui/ wrapper layer ไม่มีหน้าใดใช้ inline style เฉพาะตัวที่ขัดกับหน้าอื่น
- AC2 (Edge) — empty state สม่ำเสมอ: Given หน้าจอใดก็ตามใน 8 หน้าที่ถูกแก้ไขไม่มีข้อมูลให้แสดง, When หน้านั้นแสดงสถานะว่าง, Then ทุกหน้าใช้รูปแบบ empty-state เดียวกัน (ไอคอน+ข้อความอธิบาย+ปุ่ม action ถ้ามี)
- AC3 (Error) — error state สม่ำเสมอ: Given หน้าจอใดก็ตามใน 8 หน้าเกิด error จากระบบ, When error แสดงผล, Then ข้อความ error ใช้ตำแหน่ง/สี/ไอคอนเดียวกันทุกหน้า (ตาม ECP-036)
- AC4 (Error) — หน้านอกขอบเขตไม่ถูกแตะ: Given หน้าจอ dashboard หรือหน้า admin อื่นที่ไม่ได้อยู่ใน 8 หน้าที่แก้ไขรอบนี้, When เปรียบเทียบกับรอบก่อน, Then หน้าเหล่านั้นไม่จำเป็นต้องเปลี่ยนแปลง (full redesign อยู่ backlog)

### ECP-044: การใช้งานได้ดีบน Desktop และ Tablet (portrait/landscape) สำหรับหน้าจอที่ถูกแก้ไขรอบนี้
**Priority:** Should | Depends on: เหมือน ECP-043

ในฐานะผู้ใช้งาน (โดยเฉพาะฝ่ายผลิต/QC ที่อาจใช้แท็บเล็ตหน้างานจริง) ฉันต้องการให้หน้าจอที่ถูกแก้ไขใน
รอบนี้ใช้งานได้ดีทั้งบน desktop และ tablet (portrait/landscape) เพื่อบันทึกข้อมูลหน้างานจริงในโรงงาน
ได้สะดวก โดยยังไม่ต้องรองรับหน้าจอมือถือเต็มรูปแบบในรอบนี้ (scope-down)

- AC1 (Happy) — desktop: Given หน้าจอ 8 หน้าที่ถูกแก้ไขรอบนี้เปิดที่ความกว้างจอ 1366px ขึ้นไป, When ใช้งาน form/ตาราง/ปุ่ม action ทั้งหมด, Then ทุก element แสดงผลครบ ไม่มีการซ้อนทับ/ล้นขอบจอ และทำ task หลักของหน้านั้นจบได้ครบ
- AC2 (Happy) — tablet landscape/portrait: Given หน้าจอ QC incoming inspection และหน้า production เปิดบนแท็บเล็ตขนาดจอ 768px–1024px ทั้ง portrait และ landscape, When ผู้ใช้กรอกฟอร์ม/เลือก lot/กดยืนยัน, Then ปุ่มและช่องกรอกมีขนาดกดได้สะดวกด้วยนิ้ว, เนื้อหาจัดเรียงใหม่ให้พอดีความกว้างจอโดยไม่ต้อง scroll แนวนอน และทำ task จบได้ครบเหมือนบน desktop
- AC3 (Edge) — breakpoint เปลี่ยนระหว่างใช้งาน: Given ผู้ใช้กำลังกรอกฟอร์มบันทึกผลิตค้างอยู่บนแท็บเล็ต, When ผู้ใช้หมุนหน้าจอจาก portrait เป็น landscape กลางคัน, Then ข้อมูลที่กรอกไว้ในฟอร์มไม่หายไป และ layout ปรับตัวตามความกว้างจอใหม่ทันที
- AC4 (Error) — ขนาดจอเล็กกว่าที่รองรับ: Given ผู้ใช้เปิดหน้าจอใดก็ตามใน 8 หน้าด้วยความกว้างจอน้อยกว่า 768px, When หน้าจอ render, Then ระบบไม่จำเป็นต้อง layout สวยงามเต็มรูปแบบ (full phone responsive อยู่ backlog) แต่ต้องไม่ทำให้ปุ่ม action สำคัญหายไปจนใช้งานไม่ได้เลย (minimum safety)

### Currency & Company Scope
- Single company, สกุลเงินเดียว THB ทุกฟิลด์จำนวนเงินในระบบ
- ตัวเลขจำนวนเงินเก็บเป็นทศนิยม 2 ตำแหน่งเสมอ
- VATConfig และ CompanyProfile เป็นค่าตั้งค่าเดียวระดับบริษัท (single company) สอดคล้องกัน

### หมายเหตุเปิด (ไม่ block แต่ Tech-Lead/QA ควรทราบ)
1. รูปแบบ (format) ของ Lot number และ Batch number ยังไม่ถูกกำหนดจากปอนด์ — Tech-Lead ออกแบบใน ADR-006
2. เกณฑ์ QA/QC ผ่าน/ไม่ผ่านเชิงรายละเอียด เป็น ⚠ assumption — ควรยืนยันกับปอนด์ระหว่าง UAT
3. ตัวเลข Business Key Value ทั้ง 5 ข้อที่ใช้ใน AC เป็นข้อเสนอของ PO ที่ยังรอปอนด์ยืนยัน
4. จำนวนผู้ใช้ทดสอบจริงต่อ role สำหรับ UAT ได้รับคำตอบแล้ว (2-3 คน/role)
5. ยังไม่ระบุจากปอนด์ว่าห้ามแก้ไข invoice ที่ถูกชำระเงินแล้วหรือไม่ — default ไม่ block ในรอบนี้
6. รูปแบบจริงของ Customer ID / User ID ยังไม่กำหนดจากปอนด์ — Tech-Lead กำหนดใน ADR-006
7. **(เพิ่มรอบ Gate 2)** สิทธิ์แก้ไข BOM (ECP-039) BA สมมติเป็นของ ฝ่ายผลิต + Admin — ยังไม่มีคำยืนยันเจาะจงจากปอนด์ ไม่ block เพราะไม่กระทบข้อมูลการเงิน/กฎหมาย
8. **(เพิ่มรอบ Gate 2)** tax_id ของ Customer ไม่บังคับกรอกตอนสร้าง (ECP-001 AC7) แต่บังคับก่อนพิมพ์ใบกำกับภาษี (ECP-042 AC4) — เป็น default ของ BA ที่ไม่ปิดกั้น sales flow เดิม ควรยืนยันกับปอนด์หากต้องการบังคับตั้งแต่ตอนสร้างลูกค้า
9. **(เพิ่มรอบ Gate 2)** registered_address ของ Customer ให้ fallback ไปที่ address เดิมถ้าไม่กรอกแยก — ทางเลือกที่ PO ระบุให้ BA ตัดสินใจเอง ไม่ใช่ business rule ที่ต้องยืนยันจากปอนด์
