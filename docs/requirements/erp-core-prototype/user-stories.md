## Epic 1: Customer Management (Sales/CS)

### ECP-001: สร้างข้อมูลลูกค้าใหม่
**Priority:** Must
ในฐานะ Sales/CS ฉันต้องการสร้างข้อมูลลูกค้าใหม่ในระบบ เพื่อให้เปิด PO ให้ลูกค้ารายนั้นได้

- AC1 (Happy): Given ฉันอยู่หน้า "สร้างลูกค้าใหม่" และกรอกชื่อลูกค้า, ที่อยู่, เบอร์โทร, อีเมล ครบทุกช่องที่บังคับ, When ฉันกดบันทึก, Then ระบบสร้างลูกค้าใหม่สำเร็จ แสดงสถานะ "Active" และแสดงในหน้ารายการลูกค้าทันทีโดยไม่ต้อง refresh หน้า
- AC2 (Edge): Given ฉันกรอกชื่อลูกค้าที่ซ้ำกับชื่อลูกค้าที่มีอยู่แล้วในระบบ (ชื่อตรงกันทุกตัวอักษร), When ฉันกดบันทึก, Then ระบบยังคงบันทึกลูกค้าใหม่ให้สำเร็จแต่แสดงข้อความเตือน "พบชื่อลูกค้าที่คล้ายกันในระบบ กรุณาตรวจสอบก่อนดำเนินการต่อ" (ไม่ block การบันทึก เพราะชื่อซ้ำเป็นไปได้ในธุรกิจจริง)
- AC3 (Error): Given ฉันไม่กรอกชื่อลูกค้า (ปล่อยว่าง) แล้วกดบันทึก, When ระบบตรวจสอบความถูกต้องของฟอร์ม, Then ระบบไม่บันทึกข้อมูลและแสดงข้อความ "กรุณากรอกชื่อลูกค้า" ใต้ช่องชื่อลูกค้าทันทีโดยไม่ต้องโหลดหน้าใหม่

### ECP-002: แก้ไขข้อมูลลูกค้า
**Priority:** Should | Depends on: ECP-001

ในฐานะ Sales/CS ฉันต้องการแก้ไขข้อมูลลูกค้าที่มีอยู่แล้ว เพื่อให้ข้อมูลติดต่อ/ที่อยู่เป็นปัจจุบันเสมอ

- AC1 (Happy): Given ลูกค้ารายหนึ่งมีเบอร์โทรเดิมในระบบ, When ฉันแก้ไขเบอร์โทรเป็นเบอร์ใหม่แล้วกดบันทึก, Then ระบบอัปเดตเบอร์โทรทันทีและแสดงเบอร์ใหม่ในหน้ารายละเอียดลูกค้าโดยไม่ต้อง refresh
- AC2 (Edge): Given ลูกค้ารายนั้นมี PO ที่ยังไม่ปิด (status ≠ Closed/Cancelled) ผูกอยู่, When ฉันพยายามเปลี่ยนสถานะลูกค้าเป็น "Inactive", Then ระบบอนุญาตให้เปลี่ยนสถานะได้แต่แสดงข้อความเตือน "ลูกค้ารายนี้มี PO ที่ยังไม่ปิด N รายการ" (N = จำนวนจริง) เพื่อให้ Sales/CS ตัดสินใจต่อ
- AC3 (Error): Given ฉันลบข้อมูลอีเมลออกจากช่องอีเมลที่เคยกรอกไว้ (ปล่อยว่าง) โดยที่อีเมลเป็นช่องบังคับ, When ฉันกดบันทึก, Then ระบบไม่บันทึกการแก้ไขและแสดงข้อความ "กรุณากรอกอีเมล" พร้อมข้อมูลเดิมยังคงอยู่ไม่ถูกเขียนทับ

### ECP-003: ค้นหา/ดูรายละเอียดลูกค้าและประวัติ PO
**Priority:** Should | Depends on: ECP-001, ECP-004

ในฐานะ Sales/CS ฉันต้องการค้นหาลูกค้าและดูประวัติ PO ทั้งหมดของลูกค้ารายนั้น เพื่อติดตามความสัมพันธ์และสถานะคำสั่งซื้อได้ในที่เดียว

- AC1 (Happy): Given มีลูกค้าชื่อ "บริษัท ABC" อยู่ในระบบพร้อม PO 3 รายการ, When ฉันพิมพ์ "ABC" ในช่องค้นหาแล้วกด Enter, Then ระบบแสดงผลลัพธ์ลูกค้า "บริษัท ABC" ภายใน 3 วินาที พร้อมรายการ PO ทั้ง 3 รายการเรียงจากล่าสุดไปเก่าสุด
- AC2 (Edge): Given ไม่มีลูกค้าที่ชื่อตรงกับคำค้นหาในระบบ, When ฉันค้นหาคำว่า "XYZ999", Then ระบบแสดงข้อความ "ไม่พบลูกค้าที่ตรงกับคำค้นหา" แทนหน้าว่างเปล่าที่ไม่มีคำอธิบาย
- AC3 (Error): Given ระบบฐานข้อมูลตอบสนองล่าช้าผิดปกติ (timeout), When ฉันค้นหาลูกค้า, Then ระบบแสดงข้อความ "ไม่สามารถค้นหาข้อมูลได้ในขณะนี้ กรุณาลองใหม่อีกครั้ง" พร้อมปุ่ม "ลองใหม่" แทนหน้าขาว/ค้าง

## Epic 2: Purchasing Order (Sales/CS)

### ECP-004: เปิด PO ใหม่จากลูกค้าพร้อมตรวจสอบ stock ทันที
**Priority:** Must (KPI #1, #3, #4) | Depends on: ECP-001, ECP-009

ในฐานะ Sales/CS ฉันต้องการเปิด PO ให้ลูกค้าโดยระบุสินค้า/จำนวนที่ต้องการ และเห็นผลตรวจสอบ stock วัตถุดิบตามสูตรทันที เพื่อไม่ต้องเปิด PO ไปแล้วพบภายหลังว่าวัตถุดิบไม่พอ

- AC1 (Happy): Given ลูกค้า "บริษัท ABC" มีอยู่ในระบบ และสินค้าที่เลือก 1 รายการมีวัตถุดิบตามสูตร (BOM) เพียงพอในสต็อกครบทุกชนิด, When ฉันกรอกสินค้า จำนวน 100 หน่วย และกดยืนยัน PO, Then ระบบสร้าง PO สถานะ "Confirmed" สำเร็จ, จองสต็อกวัตถุดิบที่เกี่ยวข้องตามสูตรทันที และแสดงข้อความ "stock เพียงพอ" ก่อนยืนยัน
- AC2 (Edge): Given วัตถุดิบชนิดหนึ่งในสูตรของสินค้าที่สั่งมีคงเหลือพอสำหรับผลิตได้เพียงบางส่วน (ไม่พอเต็มจำนวนที่สั่ง 100 หน่วย), When ฉันกดยืนยัน PO, Then ระบบไม่อนุญาตให้เปลี่ยนสถานะเป็น "Confirmed" ยังคงสถานะ "Draft" และแสดงข้อความแจ้งเตือน "วัตถุดิบ [ชื่อวัตถุดิบ] ไม่เพียงพอ ขาดอยู่ [จำนวนที่ขาด] [หน่วย]" ระบุชื่อวัตถุดิบและจำนวนที่ขาดชัดเจน — ครอบคลุม 100% ของ test case ที่ QA ทดสอบ
- AC3 (Error): Given ฉันเลือกลูกค้าแล้วแต่ยังไม่ได้เพิ่มรายการสินค้าใดเลยในบรรทัด PO, When ฉันกดยืนยัน PO, Then ระบบไม่สร้าง PO และแสดงข้อความ "กรุณาเพิ่มรายการสินค้าอย่างน้อย 1 รายการก่อนยืนยัน"

### ECP-005: แก้ไข/ยกเลิก PO ก่อนเริ่มผลิต
**Priority:** Should | Depends on: ECP-004, ECP-010

ในฐานะ Sales/CS ฉันต้องการแก้ไขหรือยกเลิก PO ที่ยังไม่เข้าสู่กระบวนการผลิต เพื่อรองรับการเปลี่ยนแปลงคำสั่งซื้อของลูกค้า

- AC1 (Happy): Given PO สถานะ "Confirmed" ที่ยังไม่ถูกส่งเข้าคิวผลิต (ไม่มี Production Order ผูกอยู่), When ฉันกดยกเลิก PO, Then ระบบเปลี่ยนสถานะ PO เป็น "Cancelled" และคืนสต็อกวัตถุดิบที่เคยจองไว้กลับเข้าสต็อกส่วนกลางทันที (ยอด stock เพิ่มขึ้นตามจำนวนที่คืน)
- AC2 (Edge): Given PO สถานะ "InProduction" (มี Production Order ที่เริ่มดำเนินการแล้ว) ผูกอยู่, When ฉันพยายามกดยกเลิก PO, Then ระบบไม่อนุญาตให้ยกเลิกและแสดงข้อความ "ไม่สามารถยกเลิก PO นี้ได้ เนื่องจากเริ่มกระบวนการผลิตแล้ว"
- AC3 (Error): Given PO ถูกยกเลิกไปแล้ว (สถานะ "Cancelled"), When ฉันพยายามกดยกเลิกซ้ำอีกครั้ง, Then ระบบแสดงข้อความ "PO นี้ถูกยกเลิกไปแล้วเมื่อ [วันที่-เวลา]" และไม่ทำรายการซ้ำ (ไม่คืนสต็อกซ้ำสอง)

### ECP-006: ติดตามสถานะ PO แบบ end-to-end
**Priority:** Must (KPI #3) | Depends on: ECP-004, ECP-013, ECP-015, ECP-018, ECP-020

ในฐานะ Sales/CS ฉันต้องการดูสถานะ PO ทุกขั้นตอนตั้งแต่เปิดจนถึง invoice ในหน้าเดียว เพื่อตอบคำถามลูกค้าได้ทันทีโดยไม่ต้องถามแผนกอื่น

- AC1 (Happy): Given PO หนึ่งรายการผ่านขั้นตอน Confirmed → InProduction → QC Approved → Shipped → Invoiced ครบแล้ว, When ฉันเปิดหน้ารายละเอียด PO นั้น, Then ระบบแสดง timeline สถานะทั้ง 5 ขั้นตอนพร้อมวันที่-เวลาของแต่ละสถานะ เรียงตามลำดับเวลาจริง ภายในหน้าเดียวไม่ต้องสลับหน้าไปแผนกอื่น
- AC2 (Edge): Given PO ถูกปฏิเสธจาก QA/QC (Batch สถานะ "QC Rejected") ทำให้ต้องผลิตใหม่, When ฉันเปิดหน้าสถานะ PO, Then ระบบแสดงสถานะ "รอผลิตใหม่ (QC ไม่ผ่าน)" พร้อมเหตุผลที่ QA บันทึกไว้ ไม่ใช่ค้างอยู่ที่สถานะเดิมโดยไม่มีคำอธิบาย
- AC3 (Error): Given PO ID ที่ระบุใน URL ไม่มีอยู่จริงในระบบ, When ฉันเปิดหน้ารายละเอียด PO ด้วย ID นั้น, Then ระบบแสดงข้อความ "ไม่พบคำสั่งซื้อนี้ในระบบ" แทนหน้าเปล่า/error 500 ที่ไม่มีคำอธิบาย

## Epic 3: Stock & Real-time Inventory (Warehouse) — Priority สูงสุดของ scope

### ECP-007: ดูยอดคงเหลือ stock วัตถุดิบแบบ real-time
**Priority:** Must (KPI #1)

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการดูยอดคงเหลือของวัตถุดิบทุกชนิดในหน้าเดียวแบบ real-time เพื่อรู้ทันทีว่าวัตถุดิบชนิดใดเหลือเท่าไหร่โดยไม่ต้องเดินคลัง/โทรถาม

- AC1 (Happy): Given วัตถุดิบ "น้ำมันมะพร้าว" มียอดคงเหลือ 500 กก., When มีการบันทึกรับเข้า (Goods Receipt) เพิ่ม 100 กก., Then หน้าจอ stock ของฉัน (ที่เปิดค้างอยู่) แสดงยอดใหม่เป็น 600 กก. ภายใน ≤1 นาทีโดยไม่ต้องกด refresh หน้าเอง
- AC2 (Edge): Given วัตถุดิบชนิดหนึ่งมียอดคงเหลือเท่ากับ 0 พอดี, When ฉันเปิดหน้า stock, Then ระบบแสดงแถวของวัตถุดิบนั้นด้วยยอด "0" และป้ายสถานะ "หมดสต็อก" ไม่ใช่การซ่อนแถวนั้นออกจากรายการ
- AC3 (Error): Given ระบบเชื่อมต่อฐานข้อมูล stock ขัดข้องชั่วคราว, When ฉันเปิดหน้า stock, Then ระบบแสดงข้อความ "ไม่สามารถโหลดข้อมูล stock ได้ในขณะนี้ กรุณาลองใหม่" พร้อมเวลาล่าสุดที่ข้อมูลเคยอัปเดตสำเร็จ แทนการแสดงยอด 0 ที่ทำให้เข้าใจผิดว่าของหมดจริง

### ECP-008: บันทึกรับวัตถุดิบเข้าคลัง (Goods Receipt) พร้อม Lot number
**Priority:** Must

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการบันทึกการรับวัตถุดิบเข้าคลังพร้อมระบุ Lot number เพื่อให้ตรวจสอบย้อนกลับ (traceability) ได้ตั้งแต่ต้นทาง

- AC1 (Happy): Given วัตถุดิบ "แอลกอฮอล์" ถูกส่งมอบเข้าคลัง 200 ลิตร พร้อมเลข Lot ที่ผู้จำหน่ายระบุ, When ฉันบันทึกรับเข้าโดยกรอกชนิดวัตถุดิบ, จำนวน 200, หน่วย ลิตร, และเลข Lot, Then ระบบสร้างระเบียน Lot ใหม่ผูกกับวัตถุดิบนั้น เพิ่มยอด stock รวม +200 ลิตรทันที และ Lot นั้นพร้อมให้เลือกใช้ในการผลิต
- AC2 (Edge): Given ฉันกรอกเลข Lot ที่ตรงกับ Lot number ที่มีอยู่แล้วของวัตถุดิบชนิดเดียวกัน (รับล็อตเดิมเพิ่มอีกรอบ), When ฉันบันทึกรับเข้า, Then ระบบแจ้งเตือน "พบ Lot number นี้อยู่แล้ว ต้องการรวมยอดเข้ากับ Lot เดิมหรือสร้าง Lot ใหม่?" ให้ผู้ใช้เลือกยืนยัน ไม่บันทึกซ้ำโดยอัตโนมัติ
- AC3 (Error): Given ฉันกรอกจำนวนรับเข้าเป็นค่าติดลบหรือ 0, When ฉันกดบันทึก, Then ระบบไม่บันทึกรายการและแสดงข้อความ "จำนวนรับเข้าต้องมากกว่า 0"

### ECP-009: ระบบตรวจสอบ stock วัตถุดิบตามสูตร (BOM) และแจ้งเตือน insufficient stock
**Priority:** Must (KPI #4 — 100% ของ test case ใน UAT)

ในฐานะเจ้าหน้าที่คลังสินค้า/Sales ฉันต้องการให้ระบบตรวจสอบสต็อกวัตถุดิบเทียบกับสูตร (BOM) ของสินค้าที่สั่งโดยอัตโนมัติทุกครั้งที่เปิด/ยืนยัน PO เพื่อไม่ให้เกิด surprise ว่าวัตถุดิบไม่พอหลังเปิด PO ไปแล้ว

- AC1 (Happy): Given สูตรของสินค้า A ต้องใช้วัตถุดิบ X 2 กก./หน่วย และสต็อก X คงเหลือ 1,000 กก., When มีการเปิด PO สั่งสินค้า A จำนวน 400 หน่วย (ต้องใช้ X รวม 800 กก.), Then ระบบคำนวณแล้วแสดงผล "stock เพียงพอ (เหลือ 200 กก. หลังหักจอง)" ก่อนให้ผู้ใช้ยืนยัน PO
- AC2 (Edge): Given สินค้า A ใช้วัตถุดิบ 3 ชนิดตามสูตร และมีเพียง 1 ใน 3 ชนิดที่ไม่พอ, When ระบบตรวจสอบ ณ จุดเปิด/ยืนยัน PO, Then ระบบแจ้งเตือนเจาะจงเฉพาะวัตถุดิบที่ไม่พอ (ระบุชื่อ+จำนวนที่ขาด) โดยไม่แจ้งเตือนวัตถุดิบอีก 2 ชนิดที่เพียงพอว่าเป็นปัญหา
- AC3 (Error): Given สินค้าที่เลือกในบรรทัด PO ยังไม่มีสูตร (BOM) ผูกไว้ในระบบ (ข้อมูลจำลองไม่ครบ), When ผู้ใช้พยายามยืนยัน PO ที่มีสินค้านั้น, Then ระบบไม่อนุญาตให้ยืนยันและแสดงข้อความ "สินค้านี้ยังไม่มีสูตรการผลิต (BOM) ในระบบ กรุณาติดต่อผู้ดูแลระบบ" แทนการปล่อยผ่านโดยไม่ตรวจสอบ

### ECP-010: ระบบจอง/คืนสต็อกอัตโนมัติตามธุรกรรม
**Priority:** Must (KPI #1)

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการให้ระบบจองสต็อกวัตถุดิบอัตโนมัติเมื่อ PO ถูกยืนยัน และคืนสต็อกอัตโนมัติเมื่อ PO ถูกยกเลิกหรือเบิกใช้จริงในการผลิต เพื่อให้ยอด stock ที่แสดงตรงกับความเป็นจริงเสมอแบบ real-time

- AC1 (Happy): Given วัตถุดิบ X มียอดคงเหลือ (available) 1,000 กก. ไม่มีการจอง, When PO ที่ต้องใช้ X จำนวน 300 กก. ถูกยืนยัน (Confirmed), Then ระบบลดยอด "available to promise" ของ X เหลือ 700 กก. ทันที ในขณะที่ยอด "physical stock" ยังคง 1,000 กก. (แยกยอดจองกับยอดจริงชัดเจน)
- AC2 (Edge): Given มีการจอง X ไว้ 300 กก. จาก PO หนึ่งใบ, When PO นั้นถูกยกเลิกก่อนเข้าสู่ขั้นตอนผลิต, Then ระบบคืนยอดจอง 300 กก. กลับเข้า "available to promise" ทันทีในธุรกรรมเดียว (ไม่มีการคืนซ้ำสองหรือคืนไม่ครบ)
- AC3 (Error): Given ฝ่ายผลิตพยายามเบิกใช้วัตถุดิบ X จำนวน 300 กก. ในการผลิตจริง แต่ยอด physical stock ของ X ขณะนั้นเหลือเพียง 250 กก. (เช่น มีการปรับปรุงยอด/สูญเสียระหว่างทาง), When ระบบบันทึกการเบิกผลิต, Then ระบบไม่อนุญาตให้เบิกเกินยอดจริงที่มีอยู่ และแสดงข้อความ "วัตถุดิบ X คงเหลือจริงไม่พอ (มี 250 กก. ต้องการ 300 กก.)"

## Epic 4: Production & Traceability (Production)

### ECP-011: ดูคิวงานผลิตที่รอดำเนินการ
**Priority:** Must | Depends on: ECP-004, ECP-009

ในฐานะฝ่ายผลิต ฉันต้องการดูคิว PO/รายการที่ผ่านการตรวจสอบ stock แล้วและรอเข้าสู่กระบวนการผลิต เพื่อวางแผนงานผลิตได้

- AC1 (Happy): Given มี PO 3 รายการที่สถานะ "Confirmed" และผ่านการตรวจ stock เพียงพอแล้ว, When ฉันเปิดหน้าคิวงานผลิต, Then ระบบแสดงรายการทั้ง 3 PO เรียงตามวันที่ต้องการส่งมอบ (requested delivery date) จากใกล้ไปไกล
- AC2 (Edge): Given ไม่มี PO ใดอยู่ในสถานะพร้อมผลิตในขณะนั้น, When ฉันเปิดหน้าคิวงานผลิต, Then ระบบแสดงข้อความ "ไม่มีงานผลิตที่รอดำเนินการในขณะนี้" แทนหน้าว่างเปล่า
- AC3 (Error): Given PO รายการหนึ่งถูกยกเลิกโดย Sales/CS ในระหว่างที่ฝ่ายผลิตกำลังดูรายการอยู่, When ฉัน refresh หน้าคิวงานผลิต, Then PO ที่ถูกยกเลิกหายไปจากคิวทันทีพร้อมไม่สามารถกด assign งานให้ PO นั้นได้อีก

### ECP-012: มอบหมาย (assign) งานผลิตให้ผู้ปฏิบัติงาน
**Priority:** Must | Depends on: ECP-011

ในฐานะฝ่ายผลิต (หัวหน้างาน) ฉันต้องการมอบหมายงานผลิตจาก PO ที่พร้อมให้ผู้ปฏิบัติงานคนใดคนหนึ่ง เพื่อระบุความรับผิดชอบชัดเจนตาม GMP

- AC1 (Happy): Given PO หนึ่งรายการอยู่ในคิวงานผลิตสถานะ "รอ assign", When ฉันเลือกผู้ปฏิบัติงาน "สมชาย" แล้วกดมอบหมาย, Then ระบบสร้าง Production Order ผูกกับ PO นั้น สถานะเปลี่ยนเป็น "Assigned" และแสดงชื่อ "สมชาย" เป็นผู้รับผิดชอบทันที
- AC2 (Edge): Given Production Order หนึ่งรายการถูก assign ให้ "สมชาย" ไปแล้ว, When หัวหน้างานอีกคนพยายาม assign Production Order เดียวกันซ้ำให้ "สมหญิง", Then ระบบแจ้งเตือน "งานนี้ถูกมอบหมายให้สมชายแล้วเมื่อ [เวลา]" และให้เลือกยืนยันเปลี่ยนผู้รับผิดชอบหรือยกเลิก แทนการมอบหมายซ้ำเงียบๆ
- AC3 (Error): Given ไม่มีผู้ใช้งาน role "ฝ่ายผลิต" ที่มีสถานะ Active อยู่ในระบบเลย, When ฉันพยายามเปิด dropdown เลือกผู้ปฏิบัติงาน, Then ระบบแสดงข้อความ "ไม่พบผู้ปฏิบัติงานที่พร้อมรับงานในขณะนี้ กรุณาติดต่อ Admin" แทน dropdown ว่างเปล่าไม่มีคำอธิบาย

### ECP-013: บันทึกการดำเนินการผลิต สร้าง Batch พร้อมเชื่อม Lot
**Priority:** Must (KPI #2 — ข้อมูลตั้งต้นของ traceability) | Depends on: ECP-012, ECP-008, ECP-010

ในฐานะฝ่ายผลิต ฉันต้องการบันทึกการเบิกวัตถุดิบตาม Lot และผลการผลิตเป็น Batch เพื่อให้ทุก Batch ตรวจสอบย้อนกลับไปถึง Lot วัตถุดิบที่ใช้ได้ตาม GMP

- AC1 (Happy): Given Production Order สถานะ "Assigned" พร้อมดำเนินการ, When ฉันเลือก Lot วัตถุดิบที่จะใช้ (เช่น Lot น้ำมันมะพร้าว #L2026-001 จำนวน 50 กก.) และกด "บันทึกผลิตเสร็จสิ้น" พร้อมจำนวนสินค้าสำเร็จรูปที่ได้ 500 หน่วย, Then ระบบสร้าง Batch ใหม่พร้อม batch number ที่ไม่ซ้ำกัน ผูก Lot ที่ใช้ทั้งหมดเข้ากับ Batch นั้น และหักยอด stock วัตถุดิบจริง (physical stock) ลงตามจำนวนที่เบิกทันที
- AC2 (Edge): Given การผลิตจริงต้องใช้ Lot วัตถุดิบชนิดเดียวกันมากกว่า 1 Lot ผสมกัน (Lot เดิมเหลือไม่พอ ต้องดึง Lot ใหม่มาเสริม), When ฉันเลือก 2 Lot สำหรับวัตถุดิบชนิดเดียวกันในการบันทึกผลิตครั้งเดียว, Then ระบบบันทึกทั้ง 2 Lot เป็นแหล่งที่มาของ Batch นั้น (traceability เก็บได้ทั้งสอง Lot ไม่ใช่แค่ Lot เดียว)
- AC3 (Error): Given ฉันพยายามบันทึกผลิตเสร็จสิ้นโดยไม่ได้เลือก Lot วัตถุดิบใดเลย, When ฉันกดบันทึก, Then ระบบไม่สร้าง Batch และแสดงข้อความ "กรุณาระบุ Lot วัตถุดิบที่ใช้ในการผลิตอย่างน้อย 1 Lot ต่อวัตถุดิบแต่ละชนิดตามสูตร"

### ECP-014: ค้นหา traceability เต็มสาย Lot → Batch → สินค้าสำเร็จรูป
**Priority:** Must (KPI #2, DoD) | Depends on: ECP-013

ในฐานะผู้ใช้ที่เกี่ยวข้อง (คลัง/ฝ่ายผลิต/QA/QC/Admin) ฉันต้องการค้นหาความเชื่อมโยงแบบเต็มสายระหว่าง Lot วัตถุดิบ, Batch การผลิต และสินค้าสำเร็จรูป เพื่อสอบสวนย้อนกลับได้รวดเร็วเมื่อจำเป็น (เช่น พบปัญหาคุณภาพวัตถุดิบ)

- AC1 (Happy): Given Batch #B2026-010 ถูกผลิตจาก Lot #L2026-001 และ #L2026-002, When ฉันค้นหาด้วยเลข Lot #L2026-001, Then ระบบแสดงผลลัพธ์ครบทั้งสาย (Lot #L2026-001 → Batch #B2026-010 → รายการสินค้าสำเร็จรูปที่ผลิตได้ และ PO ที่เกี่ยวข้อง) ภายใน ≤5 นาทีนับจากเริ่มค้นหาจนเห็นผลลัพธ์ครบ
- AC2 (Edge): Given Lot วัตถุดิบหนึ่งถูกใช้ในหลาย Batch ที่ต่างกัน (ใช้ซ้ำในหลายรอบผลิต), When ฉันค้นหาด้วยเลข Lot นั้น, Then ระบบแสดงผลลัพธ์ครบทุก Batch ที่เคยใช้ Lot นี้ ไม่ใช่แค่ Batch ล่าสุดรายการเดียว
- AC3 (Error): Given ฉันค้นหาด้วยเลข Lot ที่ไม่มีอยู่จริงในระบบ, When ฉันกดค้นหา, Then ระบบแสดงข้อความ "ไม่พบ Lot นี้ในระบบ กรุณาตรวจสอบเลข Lot อีกครั้ง" แทนหน้าว่างเปล่า

## Epic 5: QA/QC

> ⚠ **ข้อสมมติที่ต้องยืนยันจากปอนด์:** stories ด้านล่างยึดหลักว่า **Batch ต้องมีสถานะ QA
> Approved ก่อนจึงจะสร้าง Shipment ได้เสมอ** (ไม่มีทางลัดข้ามการอนุมัติ) และ **เกณฑ์ผ่าน/ไม่ผ่าน
> เป็นดุลยพินิจของเจ้าหน้าที่ QA (manual judgment) ไม่มีเกณฑ์ตัวเลขตายตัวใน prototype** เพราะ
> brief ยังไม่ระบุรายละเอียดเชิงวิทยาศาสตร์ (pH/ความหนืด ฯลฯ) — เป็น domain default ตาม GMP
> ไม่ใช่การ block งาน ตามที่ระบุไว้ใน brief §7 ข้อ 2

### ECP-015: บันทึกผลตรวจสอบคุณภาพและอนุมัติ/ไม่อนุมัติ Batch
**Priority:** Must (GMP requirement แฝง) | Depends on: ECP-013

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการบันทึกผลตรวจสอบคุณภาพของ Batch และอนุมัติหรือไม่อนุมัติ เพื่อควบคุมว่ามีเฉพาะ Batch ที่ผ่านมาตรฐานเท่านั้นที่จัดส่งได้

- AC1 (Happy): Given Batch #B2026-010 สถานะ "QC Pending" (ผลิตเสร็จรอตรวจ), When ฉันบันทึกผลตรวจเป็น "Approved" พร้อมหมายเหตุ, Then สถานะ Batch เปลี่ยนเป็น "QC Approved" ทันที และ Batch นี้ปรากฏในรายการที่ฝ่ายจัดส่งสามารถเลือกสร้าง Shipment ได้
- AC2 (Edge): Given Batch #B2026-011 สถานะ "QC Pending", When ฉันบันทึกผลตรวจเป็น "Rejected" พร้อมเหตุผล "พบสิ่งปนเปื้อน", Then สถานะ Batch เปลี่ยนเป็น "QC Rejected" ทันที, Batch นี้ไม่ปรากฏในรายการที่ฝ่ายจัดส่งเลือกได้ และสถานะ PO ที่เกี่ยวข้องแสดงเป็น "รอผลิตใหม่ (QC ไม่ผ่าน)" ตาม ECP-006
- AC3 (Error): Given Batch #B2026-012 มีสถานะ "QC Approved" ไปแล้ว, When เจ้าหน้าที่ QA พยายามบันทึกผลตรวจซ้ำสำหรับ Batch เดียวกัน, Then ระบบแสดงข้อความ "Batch นี้ถูกอนุมัติไปแล้วเมื่อ [วันที่-เวลา] โดย [ชื่อผู้ตรวจ]" และต้องยืนยันอีกครั้งหากต้องการบันทึกผลใหม่ทับ (ป้องกันการอนุมัติซ้ำโดยไม่ตั้งใจ)

### ECP-016: ดูประวัติ/สถานะ QC ของ Batch ทั้งหมด
**Priority:** Should | Depends on: ECP-015

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการดูรายการ Batch ทั้งหมดพร้อมสถานะ QC เพื่อติดตามงานที่ค้างตรวจและตรวจสอบย้อนหลังได้

- AC1 (Happy): Given มี Batch 10 รายการ อยู่ในสถานะ QC Pending 4 รายการ, Approved 5 รายการ, Rejected 1 รายการ, When ฉันเปิดหน้ารายการ Batch และกรองด้วยสถานะ "QC Pending", Then ระบบแสดงเฉพาะ 4 รายการที่รอตรวจ เรียงจากวันที่ผลิตเก่าสุดไปใหม่สุด
- AC2 (Edge): Given ไม่มี Batch ใดอยู่ในสถานะ "QC Pending" ในขณะนั้น (ตรวจครบหมดแล้ว), When ฉันกรองด้วยสถานะ "QC Pending", Then ระบบแสดงข้อความ "ไม่มีงานตรวจสอบที่ค้างอยู่ในขณะนี้"
- AC3 (Error): Given ฉันพยายามเข้าถึงหน้ารายการ Batch ด้วยบัญชีผู้ใช้ที่ไม่มีสิทธิ์ role QA/QC, When ฉันเปิด URL หน้านี้ตรงๆ, Then ระบบปฏิเสธการเข้าถึงและแสดงข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-017: บันทึกผลตรวจสอบวัตถุดิบขาเข้าก่อนใช้ผลิต (Incoming QC)
**Priority:** Could | Depends on: ECP-008

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการบันทึกผลตรวจสอบคุณภาพวัตถุดิบที่รับเข้าคลังก่อนอนุญาตให้นำไปใช้ผลิต เพื่อป้องกันการใช้วัตถุดิบที่มีปัญหาตั้งแต่ต้นทาง

- AC1 (Happy): Given Lot วัตถุดิบใหม่ถูกบันทึกรับเข้า (ECP-008) สถานะเริ่มต้นเป็น "รอตรวจสอบ", When ฉันบันทึกผลตรวจเป็น "ผ่าน", Then Lot นั้นเปลี่ยนสถานะเป็น "พร้อมใช้ผลิต" และปรากฏให้เลือกใน ECP-013
- AC2 (Edge): Given ฉันบันทึกผลตรวจ Lot หนึ่งเป็น "ไม่ผ่าน", When ฝ่ายผลิตพยายามเลือก Lot นั้นในการบันทึกผลิต (ECP-013), Then ระบบไม่แสดง Lot นั้นเป็นตัวเลือก และหากพยายามเลือกผ่าน API/URL ตรง ระบบปฏิเสธพร้อมข้อความ "Lot นี้ไม่ผ่านการตรวจสอบคุณภาพ ไม่สามารถนำไปใช้ผลิตได้"
- AC3 (Error): Given Lot วัตถุดิบยังไม่มีการบันทึกผลตรวจสอบใดๆ (สถานะ "รอตรวจสอบ"), When ฝ่ายผลิตพยายามเลือก Lot นั้นในการบันทึกผลิต, Then ระบบไม่อนุญาตและแสดงข้อความ "Lot นี้ยังไม่ผ่านการตรวจสอบคุณภาพขาเข้า"

## Epic 6: Shipping (Logistics)

### ECP-018: สร้างบันทึกการจัดส่งจาก Batch ที่ผ่านการอนุมัติ QA แล้ว
**Priority:** Must | Depends on: ECP-015

ในฐานะฝ่ายจัดส่ง ฉันต้องการสร้างบันทึกการจัดส่งจาก Batch ที่ผ่าน QA แล้วเท่านั้น เพื่อรับประกันว่าไม่มีสินค้าที่ไม่ผ่านคุณภาพหลุดออกไปถึงลูกค้า

- AC1 (Happy): Given Batch #B2026-010 สถานะ "QC Approved" ผูกกับ PO ของลูกค้า ABC, When ฉันเลือก Batch นั้นและกด "สร้างบันทึกจัดส่ง" ระบุวันที่จัดส่งและจำนวน, Then ระบบสร้าง Shipment สำเร็จ สถานะ "Shipped" และอัปเดตสถานะ PO เป็น "Shipped" ตาม ECP-006
- AC2 (Edge): Given Batch #B2026-011 สถานะ "QC Rejected", When ฉันพยายามค้นหา Batch นี้ในหน้าสร้างบันทึกจัดส่ง, Then ระบบไม่แสดง Batch นี้เป็นตัวเลือกให้จัดส่งเลย (กรองออกตั้งแต่ต้น ไม่ใช่แสดงแล้วค่อย block ตอนกดยืนยัน)
- AC3 (Error): Given Batch #B2026-012 สถานะ "QC Pending" (ยังไม่มีผลตรวจ), When ฉันพยายามเรียก action สร้างบันทึกจัดส่งสำหรับ Batch นี้โดยตรง (เช่นผ่าน URL/รายการเก่าที่ cache ไว้), Then ระบบปฏิเสธและแสดงข้อความ "Batch นี้ยังไม่ผ่านการอนุมัติจาก QA/QC ไม่สามารถจัดส่งได้"

### ECP-019: ติดตาม/อัปเดตสถานะการจัดส่ง
**Priority:** Should | Depends on: ECP-018

ในฐานะฝ่ายจัดส่ง ฉันต้องการอัปเดตสถานะการจัดส่งเมื่อสินค้าถึงมือลูกค้า เพื่อให้ทุกแผนกเห็นสถานะล่าสุดตรงกัน

- AC1 (Happy): Given Shipment สถานะ "Shipped" ถูกส่งออกไปแล้ว, When ฉันอัปเดตสถานะเป็น "Delivered" พร้อมวันที่ส่งถึง, Then ระบบบันทึกสถานะใหม่ทันทีและ PO ที่เกี่ยวข้องแสดงสถานะ "Delivered" ในหน้าติดตามของ Sales/CS (ECP-006)
- AC2 (Edge): Given Shipment หนึ่งรายการยังไม่ถึงสถานะ "Shipped" (ยังเป็น Draft), When ฉันพยายามอัปเดตสถานะเป็น "Delivered" ข้ามขั้นตอน, Then ระบบไม่อนุญาตและแสดงข้อความ "ไม่สามารถเปลี่ยนเป็น Delivered ได้ ต้องมีสถานะ Shipped ก่อน"
- AC3 (Error): Given ฉันกรอกวันที่ส่งถึงลูกค้าเป็นวันที่ในอนาคต (หลังวันที่ปัจจุบัน), When ฉันกดบันทึก, Then ระบบแสดงข้อความ "วันที่ส่งถึงต้องไม่เกินวันที่ปัจจุบัน" และไม่บันทึกค่านั้น

## Epic 7: Invoice & Billing (Finance)

### ECP-020: ออก invoice จาก PO ที่จัดส่งแล้ว
**Priority:** Must | Depends on: ECP-018

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการออก invoice จาก PO ที่จัดส่งสำเร็จแล้ว เพื่อเรียกเก็บเงินจากลูกค้า

- AC1 (Happy): Given PO สถานะ "Shipped" มียอดรวม 50,000 บาท (คำนวณจาก quantity x unit price ของทุกบรรทัด) ยังไม่เคยออก invoice, When ฉันกด "ออก invoice" สำหรับ PO นั้น, Then ระบบสร้าง invoice ใหม่พร้อมเลขที่ invoice ไม่ซ้ำกัน ยอด 50,000.00 บาท สถานะ "Issued" และสถานะ PO เปลี่ยนเป็น "Invoiced"
- AC2 (Edge): Given PO นั้นมี invoice ที่ออกไปแล้วก่อนหน้า (สถานะ PO = "Invoiced"), When ฉันพยายามกด "ออก invoice" ซ้ำสำหรับ PO เดียวกัน, Then ระบบไม่สร้าง invoice ใบใหม่และแสดงข้อความ "PO นี้มี invoice เลขที่ [เลขที่] ออกไปแล้วเมื่อ [วันที่]"
- AC3 (Error): Given PO ยังมีสถานะ "InProduction" (ยังไม่ได้จัดส่ง), When ฉันพยายามออก invoice สำหรับ PO นั้น, Then ระบบไม่อนุญาตและแสดงข้อความ "ไม่สามารถออก invoice ได้ PO นี้ยังไม่ถูกจัดส่ง"

### ECP-021: บันทึกการรับชำระเงิน (manual reconcile)
**Priority:** Must | Depends on: ECP-020

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการบันทึกการรับชำระเงินของ invoice ด้วยมือ (เทียบกับหลักฐานโอนเงินภายนอก) เพื่ออัปเดตสถานะการเรียกเก็บเงินโดยไม่ต้องเชื่อม payment gateway จริง

- AC1 (Happy): Given invoice เลขที่ INV-0001 ยอด 50,000 บาท สถานะ "Issued", When ฉันบันทึกรับชำระเต็มจำนวน 50,000 บาท พร้อมวันที่รับเงิน, Then สถานะ invoice เปลี่ยนเป็น "Paid" ทันทีและแสดงยอดคงค้างเป็น 0.00 บาท
- AC2 (Edge): Given invoice เลขที่ INV-0002 ยอด 30,000 บาท สถานะ "Issued", When ฉันบันทึกรับชำระบางส่วน 20,000 บาท, Then สถานะ invoice เปลี่ยนเป็น "Partially Paid" และแสดงยอดคงค้าง 10,000.00 บาท อย่างถูกต้อง
- AC3 (Error): Given invoice เลขที่ INV-0003 ยอด 10,000 บาท, When ฉันพยายามบันทึกรับชำระเป็นจำนวน 15,000 บาท (เกินยอด invoice), Then ระบบไม่บันทึกรายการและแสดงข้อความ "จำนวนเงินที่รับชำระเกินยอดคงค้างของ invoice นี้ (คงค้าง 10,000.00 บาท)"

### ECP-022: ดูสถานะ invoice/การเรียกเก็บเงินทั้งหมด
**Priority:** Should | Depends on: ECP-020, ECP-021

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการดูรายการ invoice ทั้งหมดพร้อมสถานะการชำระเงิน เพื่อติดตามลูกหนี้ที่ค้างชำระ

- AC1 (Happy): Given มี invoice 5 ใบ สถานะ Paid 3 ใบ, Partially Paid 1 ใบ, Issued (ยังไม่ชำระ) 1 ใบ, When ฉันกรองด้วยสถานะ "ยังไม่ชำระครบ" (Issued + Partially Paid), Then ระบบแสดงเฉพาะ 2 ใบที่ตรงเงื่อนไข พร้อมยอดคงค้างของแต่ละใบ
- AC2 (Edge): Given ไม่มี invoice ใดค้างชำระเลยในขณะนั้น, When ฉันกรองด้วยสถานะ "ยังไม่ชำระครบ", Then ระบบแสดงข้อความ "ไม่มี invoice ที่ค้างชำระในขณะนี้"
- AC3 (Error): Given ฉันพยายามเข้าถึงหน้ารายการ invoice ด้วยบัญชีที่ไม่มีสิทธิ์ role บัญชี/การเงินหรือ Admin, When ฉันเปิด URL หน้านี้ตรงๆ, Then ระบบปฏิเสธการเข้าถึงและแสดงข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

## Epic 8: User Management & Audit Log (Admin)

### ECP-023: สร้าง/แก้ไขผู้ใช้งานและกำหนด role
**Priority:** Must

ในฐานะ Admin ฉันต้องการสร้างและแก้ไขบัญชีผู้ใช้งานพร้อมกำหนด role ให้ตรงกับหน้าที่ เพื่อควบคุมการเข้าถึงระบบตาม 7 roles ที่กำหนด

- AC1 (Happy): Given ยังไม่มีบัญชีผู้ใช้ชื่อ "somchai", When ฉันสร้างผู้ใช้ใหม่ username "somchai" กำหนด role "ฝ่ายผลิต" และสถานะ "Active", Then ระบบสร้างบัญชีสำเร็จ และผู้ใช้ "somchai" สามารถ login เข้าใช้งานเมนู/สิทธิ์ตาม role "ฝ่ายผลิต" ได้ทันที
- AC2 (Edge): Given ผู้ใช้ "somchai" มี role "ฝ่ายผลิต" อยู่แล้วและกำลัง login ใช้งานอยู่ในขณะนั้น, When Admin เปลี่ยน role ของ "somchai" เป็น "QA/QC", Then เมนู/สิทธิ์การเข้าถึงของ "somchai" เปลี่ยนตาม role ใหม่ภายในการเข้าใช้งานครั้งถัดไป (login ใหม่หรือ session refresh) โดยไม่ต้องแก้ไขที่ตัวโค้ด
- AC3 (Error): Given ฉันพยายามสร้างผู้ใช้ใหม่ด้วย username ที่ซ้ำกับผู้ใช้ที่มีอยู่แล้ว, When ฉันกดบันทึก, Then ระบบไม่สร้างบัญชีใหม่และแสดงข้อความ "username นี้มีผู้ใช้งานอยู่แล้วในระบบ"

### ECP-024: กำหนด/ปรับสิทธิ์ (permission) ต่อ role ผ่านหน้าจอ config
**Priority:** Must (ข้อกำหนดเน้นย้ำจากปอนด์ — role ปรับบ่อยระหว่างสำรวจ user journey)

ในฐานะ Admin ฉันต้องการกำหนด/ปรับสิทธิ์การเข้าถึงเมนูและการทำรายการของแต่ละ role ผ่านหน้าจอ config โดยไม่ต้องแก้โค้ดหรือ deploy ใหม่ เพื่อให้ปรับ role ได้เองระหว่างช่วงสำรวจ user journey

- AC1 (Happy): Given role "ฝ่ายจัดส่ง" ปัจจุบันไม่มีสิทธิ์เข้าเมนู "รายงานการเงิน", When ฉันเปิดหน้า config สิทธิ์ของ role "ฝ่ายจัดส่ง" และเปิดสิทธิ์ "ดูรายงานการเงิน" แล้วกดบันทึก, Then ผู้ใช้ที่มี role "ฝ่ายจัดส่ง" มองเห็นเมนู "รายงานการเงิน" ได้ทันทีในการเข้าใช้งานครั้งถัดไป โดยไม่ต้องมีการแก้ไขซอร์สโค้ดหรือ deploy ระบบใหม่
- AC2 (Edge): Given Admin พยายามถอดสิทธิ์การเข้าถึงหน้า "จัดการผู้ใช้งาน" ออกจาก role "Admin" ทั้งหมด (จะทำให้ไม่มีใครเข้าหน้า config สิทธิ์ได้อีก), When ฉันกดบันทึกการเปลี่ยนแปลงนี้, Then ระบบไม่อนุญาตให้บันทึกและแสดงข้อความ "ไม่สามารถถอดสิทธิ์นี้ออกจาก role Admin ได้ เนื่องจากจะทำให้ไม่มีผู้ใช้ที่เข้าถึงหน้าจัดการสิทธิ์ได้อีก"
- AC3 (Error): Given ฉันกำลังแก้ไขสิทธิ์ของ role หนึ่งอยู่ แต่การเชื่อมต่อขาดหายระหว่างกดบันทึก, When ระบบไม่สามารถบันทึกการเปลี่ยนแปลงได้สำเร็จ, Then ระบบแสดงข้อความ "บันทึกการเปลี่ยนแปลงไม่สำเร็จ กรุณาลองใหม่" และสิทธิ์เดิมก่อนแก้ไขยังคงมีผลอยู่ (ไม่บันทึกค่าที่แก้ไขค้างคาแบบครึ่งๆ กลางๆ)

### ECP-025: บันทึก audit log ทุกครั้งที่ login
**Priority:** Must (DoD ระบุชัดเจน)

ในฐานะ Admin ฉันต้องการให้ระบบบันทึก audit log ทุกครั้งที่มีผู้ใช้ login (สำเร็จหรือไม่สำเร็จ) เพื่อตรวจสอบย้อนหลังด้านความปลอดภัยได้

- AC1 (Happy): Given ผู้ใช้ "somchai" กรอก username/password ถูกต้อง, When "somchai" กด login สำเร็จ, Then ระบบบันทึก audit log รายการใหม่ทันทีที่มี user_id, เวลา login, ผลลัพธ์ "Success" และสามารถค้นดูได้จากหน้า audit log ของ Admin
- AC2 (Edge): Given ผู้ใช้ "somchai" กรอก password ผิด 1 ครั้ง, When ระบบปฏิเสธการ login, Then ระบบยังคงบันทึก audit log รายการนั้นด้วยผลลัพธ์ "Failed" พร้อมเวลาที่พยายาม login (ไม่ใช่บันทึกเฉพาะ login ที่สำเร็จเท่านั้น)
- AC3 (Error): Given ระบบบันทึก audit log ขัดข้องชั่วคราวขณะมีการ login, When เหตุการณ์ login เกิดขึ้นแต่การเขียน log ล้มเหลว, Then ระบบต้องไม่ปล่อยให้ผู้ใช้ login สำเร็จโดยไม่มีการพยายามบันทึก log เลย (อย่างน้อยต้องมีการ retry หรือ error ปรากฏใน system log ฝั่ง Admin ให้ตรวจพบภายหลังได้ — ไม่ใช่ความเงียบแบบไม่มีร่องรอย)

### ECP-026: บันทึก audit log ทุกครั้งที่ทำรายการสำคัญและดูย้อนหลังได้
**Priority:** Must (DoD ระบุชัดเจน) | Depends on: ECP-025

ในฐานะ Admin ฉันต้องการให้ระบบบันทึก audit log ทุกครั้งที่มีการสร้าง/แก้ไข/อนุมัติ/ยกเลิกรายการสำคัญ (เช่น สร้าง PO, อนุมัติ Batch, ออก invoice, แก้ไขสิทธิ์ role) และสามารถค้นดูย้อนหลังได้ เพื่อตรวจสอบความรับผิดชอบตาม GMP/governance

- AC1 (Happy): Given เจ้าหน้าที่ QA อนุมัติ Batch #B2026-010 (ECP-015), When การอนุมัตินั้นเกิดขึ้นสำเร็จ, Then ระบบบันทึก audit log ที่มี user_id ของผู้อนุมัติ, action = "Approve Batch", entity = "Batch #B2026-010", เวลาที่ทำรายการ และ Admin สามารถค้นหา log นี้ได้จากหน้า audit log โดยกรองด้วยชื่อผู้ใช้หรือประเภทรายการ
- AC2 (Edge): Given มี audit log มากกว่า 1,000 รายการสะสมอยู่ในระบบ, When Admin ค้นหา log ด้วยช่วงวันที่และประเภทรายการเฉพาะ (เช่น "Approve Batch" ระหว่างวันที่ 1-30 มิ.ย.), Then ระบบแสดงเฉพาะรายการที่ตรงเงื่อนไขทั้งหมดภายในหน้าเดียว/แบ่งหน้าที่ค้นหาต่อได้ ไม่ปะปนกับรายการประเภทอื่น
- AC3 (Error): Given Admin พยายามลบหรือแก้ไขรายการ audit log ที่มีอยู่แล้ว (ผ่านหน้าจอหรือเรียก action ตรง), When ระบบได้รับคำสั่งลบ/แก้ไข audit log, Then ระบบปฏิเสธการลบ/แก้ไขเสมอและแสดงข้อความ "ไม่สามารถแก้ไขหรือลบ audit log ได้" (audit log ต้องเป็น append-only ไม่มีใครแก้ย้อนหลังได้แม้แต่ Admin)

## Epic 9: Dashboard (แยกตาม 7 ส่วนงาน)

> ทุก story ในกลุ่มนี้ Priority = Must (DoD ระบุชัดเจนว่าต้องมี Dashboard ครบทั้ง 7 ส่วนงาน)

### ECP-027: Dashboard ของ Sales/CS
Depends on: ECP-004, ECP-006

ในฐานะ Sales/CS ฉันต้องการ dashboard สรุปจำนวน PO ตามสถานะและลูกค้าที่มี PO ค้างอยู่ เพื่อติดตามงานขายภาพรวมได้ในหน้าเดียว

- AC1 (Happy): Given มี PO 20 รายการกระจายในสถานะต่างๆ, When ฉันเปิด dashboard, Then ระบบแสดงจำนวน PO แยกตามสถานะ (Draft/Confirmed/InProduction/Shipped/Invoiced/Cancelled) เป็นตัวเลขที่ตรงกับข้อมูลจริงในระบบ ณ ขณะนั้น
- AC2 (Edge): Given ยังไม่มี PO ใดในระบบเลย (ข้อมูลเริ่มต้น), When ฉันเปิด dashboard, Then ระบบแสดงตัวเลขทุกสถานะเป็น 0 พร้อมข้อความแนะนำ "เริ่มต้นสร้าง PO แรกของคุณ" แทนหน้าว่างเปล่าไม่มีคำอธิบาย
- AC3 (Error): Given ผู้ใช้ role อื่น (เช่น ฝ่ายผลิต) พยายามเข้าถึง dashboard ของ Sales/CS โดยตรง, When เปิด URL ของ dashboard นี้, Then ระบบปฏิเสธการเข้าถึงตามสิทธิ์ role (อ้างอิง ECP-024) และแสดงข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-028: Dashboard ของคลังสินค้า (เน้น stock real-time)
Depends on: ECP-007, ECP-010

ในฐานะเจ้าหน้าที่คลังสินค้า ฉันต้องการ dashboard แสดงวัตถุดิบที่ใกล้หมด/หมดสต็อกแบบ real-time เพื่อวางแผนการสั่งซื้อล่วงหน้าได้

- AC1 (Happy): Given มีวัตถุดิบ 3 ชนิดที่ยอดคงเหลือต่ำกว่า 100 หน่วย, When ฉันเปิด dashboard คลัง, Then ระบบแสดงรายการวัตถุดิบทั้ง 3 ชนิดในส่วน "ใกล้หมด" พร้อมยอดคงเหลือปัจจุบันที่ตรงกับหน้า ECP-007 เป๊ะ (ไม่ใช่ตัวเลขคนละชุด)
- AC2 (Edge): Given มีการรับวัตถุดิบเข้าคลังเพิ่มเติมระหว่างที่ dashboard เปิดค้างอยู่ (ทำให้พ้นเกณฑ์ใกล้หมด), When ยอด stock เปลี่ยนแปลง, Then วัตถุดิบชนิดนั้นหายไปจากส่วน "ใกล้หมด" ภายใน ≤1 นาทีโดยไม่ต้อง refresh หน้า (สอดคล้องกับ ECP-007 AC1)
- AC3 (Error): Given ข้อมูลสูตร (BOM) ของสินค้าบางรายการหายไปจากระบบ (ข้อมูลจำลองไม่ครบ), When dashboard คำนวณยอดที่ถูกจองล่วงหน้าจากสูตร, Then ระบบยังคงแสดง dashboard ได้ปกติสำหรับวัตถุดิบที่มีข้อมูลครบ และแสดงข้อความเตือนแยกต่างหาก "มีสินค้า N รายการที่ยังไม่มีสูตรในระบบ" แทนการทำให้ทั้งหน้า dashboard ล่ม

### ECP-029: Dashboard ของฝ่ายผลิต
Depends on: ECP-011, ECP-013

ในฐานะฝ่ายผลิต ฉันต้องการ dashboard สรุปงานผลิตที่ต้องทำวันนี้และ Batch ที่เพิ่งผลิตเสร็จ เพื่อจัดลำดับความสำคัญของงาน

- AC1 (Happy): Given มี Production Order 5 รายการสถานะ "Assigned" ที่ยังไม่เริ่มดำเนินการ, When ฉันเปิด dashboard, Then ระบบแสดงจำนวนงานที่ต้องทำ = 5 พร้อมรายชื่อ PO/สินค้าที่เกี่ยวข้อง
- AC2 (Edge): Given งานผลิตทั้งหมดถูกดำเนินการจนเสร็จสิ้นครบแล้วในวันนั้น, When ฉันเปิด dashboard, Then ระบบแสดง "ไม่มีงานผลิตค้างในขณะนี้" แทนตัวเลข 0 เปล่าๆ ที่ไม่มีบริบท
- AC3 (Error): Given ผู้ใช้ role "ฝ่ายจัดส่ง" พยายามเข้าถึง dashboard ฝ่ายผลิตโดยตรงผ่าน URL, When เปิดหน้านี้, Then ระบบปฏิเสธการเข้าถึงและแสดงข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-030: Dashboard ของ QA/QC
Depends on: ECP-016

ในฐานะเจ้าหน้าที่ QA/QC ฉันต้องการ dashboard สรุปจำนวน Batch ที่รอตรวจ/ผ่าน/ไม่ผ่าน เพื่อจัดลำดับงานตรวจสอบคุณภาพ

- AC1 (Happy): Given มี Batch สถานะ QC Pending 6 รายการ, Approved 20 รายการ, Rejected 2 รายการ, When ฉันเปิด dashboard, Then ระบบแสดงตัวเลขทั้ง 3 กลุ่มตรงกับข้อมูลจริง และคลิกที่กลุ่ม "QC Pending" แล้วนำไปหน้ารายการที่กรองแล้วตาม ECP-016
- AC2 (Edge): Given ไม่มี Batch ใดถูก Rejected เลยตั้งแต่เริ่มใช้งานระบบ, When ฉันเปิด dashboard, Then ระบบแสดงตัวเลข "Rejected = 0" อย่างถูกต้อง ไม่ใช่ซ่อนแถบนั้นออกจาก dashboard
- AC3 (Error): Given ผู้ใช้ role อื่นที่ไม่ใช่ QA/QC หรือ Admin พยายามเข้าถึง dashboard นี้, When เปิด URL ตรง, Then ระบบปฏิเสธการเข้าถึงพร้อมข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-031: Dashboard ของฝ่ายจัดส่ง
Depends on: ECP-018, ECP-019

ในฐานะฝ่ายจัดส่ง ฉันต้องการ dashboard สรุปรายการที่รอจัดส่งและที่จัดส่งแล้ว เพื่อวางแผนเส้นทาง/ลำดับงานจัดส่งประจำวัน

- AC1 (Happy): Given มี Batch ที่ผ่าน QC Approved แล้ว 4 รายการที่ยังไม่ถูกสร้าง Shipment, When ฉันเปิด dashboard, Then ระบบแสดง "รอจัดส่ง = 4 รายการ" พร้อมลิงก์ไปหน้าสร้างบันทึกจัดส่ง (ECP-018)
- AC2 (Edge): Given ไม่มี Batch ใดพร้อมจัดส่งในขณะนั้น (ทุก Batch ยังไม่ผ่าน QC หรือจัดส่งไปหมดแล้ว), When ฉันเปิด dashboard, Then ระบบแสดง "ไม่มีรายการรอจัดส่งในขณะนี้"
- AC3 (Error): Given ผู้ใช้ role อื่นพยายามเข้าถึง dashboard ฝ่ายจัดส่งโดยตรง, When เปิด URL ตรง, Then ระบบปฏิเสธการเข้าถึงพร้อมข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-032: Dashboard ของฝ่ายบัญชี/การเงิน
Depends on: ECP-020, ECP-021, ECP-022

ในฐานะฝ่ายบัญชี/การเงิน ฉันต้องการ dashboard สรุปยอด invoice ค้างชำระและยอดที่รับชำระแล้ว เพื่อติดตามกระแสเงินสดของโรงงาน

- AC2 (Edge): Given ทุก invoice ในระบบมีสถานะ "Paid" ครบทั้งหมด (ไม่มีค้างชำระ), When ฉันเปิด dashboard, Then ระบบแสดง "ยอดค้างชำระรวม = 0.00 บาท" พร้อมข้อความ "ไม่มี invoice ค้างชำระในขณะนี้"
- AC3 (Error): Given ผู้ใช้ role อื่นที่ไม่ใช่บัญชี/การเงินหรือ Admin พยายามเข้าถึง dashboard นี้โดยตรง, When เปิด URL ตรง, Then ระบบปฏิเสธการเข้าถึงพร้อมข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

### ECP-033: Dashboard ของ Admin (ภาพรวม user/audit log)
Depends on: ECP-023, ECP-025, ECP-026

ในฐานะ Admin ฉันต้องการ dashboard สรุปจำนวนผู้ใช้งานตาม role และกิจกรรมล่าสุดใน audit log เพื่อเฝ้าระวังการใช้งานระบบโดยรวม

- AC2 (Edge): Given มี role หนึ่ง (เช่น ฝ่ายจัดส่ง) ที่ยังไม่มีผู้ใช้งานคนใดถูกกำหนดเลย, When ฉันเปิด dashboard, Then ระบบแสดง role นั้นด้วยจำนวน "0 คน" ไม่ใช่ซ่อน role นั้นออกจากรายการสรุป
- AC3 (Error): Given ผู้ใช้ role อื่นที่ไม่ใช่ Admin พยายามเข้าถึง dashboard นี้โดยตรง, When เปิด URL ตรง, Then ระบบปฏิเสธการเข้าถึงพร้อมข้อความ "คุณไม่มีสิทธิ์เข้าถึงหน้านี้"

## Epic 10: Usability & Onboarding (cross-cutting, ทุก role)

### ECP-034: เมนู/หน้า home ตาม role พร้อม onboarding เบื้องต้น
**Priority:** Must (KPI #5) | Depends on: ECP-023, ECP-024

ในฐานะผู้ใช้งานทุก role ฉันต้องการเห็นเมนูและหน้า home ที่แสดงเฉพาะฟังก์ชันที่เกี่ยวข้องกับ role ของฉันพร้อมคำแนะนำเบื้องต้น เพื่อเริ่มใช้งานได้เองโดยไม่ต้องมี training

- AC1 (Happy): Given ผู้ใช้ "somchai" มี role "ฝ่ายผลิต" login เข้าระบบเป็นครั้งแรก, When ระบบแสดงหน้า home, Then เมนูที่แสดงมีเฉพาะฟังก์ชันของ role ฝ่ายผลิต (คิวงานผลิต, บันทึกผลิต, traceability) และไม่แสดงเมนูของ role อื่น (เช่น ออก invoice, จัดการสิทธิ์) เลย
- AC2 (Edge): Given ผู้ใช้ใหม่ยังไม่เคย login เข้าระบบมาก่อนเลย (first login), When login สำเร็จครั้งแรก, Then ระบบแสดงข้อความ/tooltip แนะนำจุดเริ่มต้น (เช่น "เริ่มต้นที่เมนูนี้เพื่อดูงานที่ต้องทำวันนี้") อย่างน้อย 1 จุดบนหน้า home โดยไม่ต้องให้ผู้ใช้เปิดคู่มือแยก
- AC3 (Error): Given ผู้ใช้คนหนึ่งถูกกำหนด role ที่ยังไม่ได้ตั้งค่าสิทธิ์เมนูใดๆ ไว้เลย (role ว่างเปล่า/ตั้งค่าไม่สมบูรณ์), When ผู้ใช้ login เข้าระบบ, Then ระบบแสดงหน้า home พร้อมข้อความ "ยังไม่มีเมนูที่กำหนดให้บทบาทนี้ กรุณาติดต่อ Admin" แทนหน้าเปล่าที่ไม่มีคำอธิบายว่าเกิดอะไรขึ้น

### ECP-035: ผู้ใช้ใหม่ทำ task หลักของ role ตนเองจบได้โดยไม่ต้องมี training
**Priority:** Must (KPI #5 — เป้าหมาย 80% ของ scenario ใน UAT) | Depends on: ทุก epic ที่เกี่ยวข้องกับ task หลักของแต่ละ role

ในฐานะทีมพัฒนา/ปอนด์ ฉันต้องการวัดผลว่าผู้ใช้ทดสอบใหม่ (ไม่เคยใช้ระบบมาก่อน) สามารถทำ task หลักของ role ตนเองจนจบได้เองโดยไม่ต้องมี training/คู่มือ เพื่อยืนยันว่าระบบ self-navigate ได้จริงตามเป้าหมาย prototype

- AC1 (Happy): Given ผู้ใช้ทดสอบใหม่ 1 คนต่อ role (ไม่เคยเห็นระบบมาก่อน) ได้รับโจทย์ task หลักของ role ตนเอง (เช่น Sales: เปิด PO ให้ลูกค้า 1 ใบให้จบ), When ผู้ใช้ทดสอบพยายามทำ task โดยไม่ได้รับ training/คู่มือใดๆ ล่วงหน้า, Then อย่างน้อย 80% ของ scenario ที่ QA ออกแบบใน UAT ผู้ใช้ทดสอบทำ task จบสำเร็จได้เอง (วัดผลจริงใน UAT ตามที่ brief ระบุ)
- AC2 (Edge): Given ผู้ใช้ทดสอบคนหนึ่งทำ task ไม่จบภายในเวลาที่กำหนดของ scenario, When QA บันทึกผลการทดสอบ, Then ต้องบันทึกจุดที่ผู้ใช้ติดขัด (เช่น หาไม่เจอว่าปุ่มไหนต้องกด) เป็นข้อมูล user journey ที่แท้จริงของ role นั้น เพื่อป้อนกลับเข้าสู่การปรับปรุง UX ในรอบถัดไป (ตรงตามวัตถุประสงค์คู่ของ prototype ในการสำรวจ user journey)
- AC3 (Error): Given ยังไม่มีการกำหนดจำนวนผู้ใช้ทดสอบจริงต่อ role สำหรับรอบ UAT (เป็น open item ตาม brief §6), When ทีม QA วางแผน UAT, Then QA ต้องสอบถามปอนด์เรื่องจำนวนผู้ใช้ทดสอบต่อ role ก่อนเริ่ม UAT รอบนี้ (ส่งต่อเป็น dependency ของ QA ไม่ใช่ของ BA)

### ECP-036: ข้อความ error/แจ้งเตือนของระบบสื่อสารชัดเจน
**Priority:** Should | Depends on: ทุก story ที่มี error case (ECP-001 AC3, ECP-004 AC2, ECP-009 AC2 ฯลฯ)

ในฐานะผู้ใช้งานทุก role ฉันต้องการให้ข้อความ error/แจ้งเตือนของระบบอธิบายปัญหาและสิ่งที่ต้องทำต่อด้วยภาษาที่ไม่ใช่ศัพท์เทคนิค เพื่อแก้ปัญหาต่อได้เองโดยไม่ต้องโทรถามทีมพัฒนา

- AC1 (Happy): Given ผู้ใช้กรอกข้อมูลผิดพลาดในฟอร์มใดก็ตามในระบบ (เช่น ช่องบังคับว่าง), When ระบบแสดงข้อความ error, Then ข้อความต้องระบุ "ช่องไหนผิด" และ "ต้องแก้อย่างไร" เป็นภาษาไทยที่เข้าใจได้โดยไม่มีคำว่า null/undefined/error code/stack trace ปรากฏต่อผู้ใช้
- AC2 (Edge): Given เกิดข้อผิดพลาดฝั่งระบบที่ไม่คาดคิด (unexpected server error) ระหว่างผู้ใช้ทำรายการ, When ระบบตรวจพบข้อผิดพลาดนี้, Then ระบบแสดงข้อความกลาง "เกิดข้อผิดพลาดที่ไม่คาดคิด กรุณาลองใหม่อีกครั้ง หากยังพบปัญหากรุณาแจ้ง Admin" แทนการแสดงรายละเอียด technical error ให้ผู้ใช้ปลายทางเห็น
- AC3 (Error): Given ข้อความ error ที่เคยแสดงผลเป็นภาษาอังกฤษปนศัพท์เทคนิค (regression), When QA ตรวจสอบทุกหน้าจอที่มี error case ตามที่ระบุใน AC ของ story อื่นทั้งหมด, Then ต้องไม่พบข้อความ error ใดที่เป็นภาษาอังกฤษล้วนหรือมีศัพท์เทคนิคที่ผู้ใช้ทั่วไปไม่เข้าใจ (0 รายการที่หลุดผ่าน)

## Data Rules (สรุปจาก Brief)

### Entities หลักและฟิลด์สำคัญ

| Entity | ฟิลด์หลัก | Validation/หมายเหตุ |
|---|---|---|
| Customer | customer_id, name*, address, phone*, email*, contact_person, status (Active/Inactive) | name/phone/email เป็นช่องบังคับ (ECP-001 AC3); ชื่อซ้ำไม่ block แต่เตือน (ECP-001 AC2) |
| PurchaseOrder (PO) | po_id, po_number (unique), customer_id, order_date, requested_delivery_date, status, lines[] | status: `Draft → Confirmed → InProduction → Shipped → Invoiced → Closed` หรือ `Cancelled` ได้จาก Draft/Confirmed เท่านั้น (ไม่ cancel ได้เมื่อ InProduction ขึ้นไป — ECP-005 AC2) |
| POLine | line_id, po_id, product_id, quantity (>0), uom, unit_price (≥0) | ต้องมีอย่างน้อย 1 บรรทัดก่อนยืนยัน PO (ECP-004 AC3) |
| RawMaterial | material_id, name, uom, status (Active/Discontinued) | — |
| BOM/Formula | bom_id, finished_good_id, lines[material_id, qty_per_unit] | สินค้าที่ไม่มี BOM ต้อง block การยืนยัน PO (ECP-009 AC3) — รับสูตรสำเร็จรูป/mock เท่านั้น ไม่มี version control (out-of-scope) |
| Lot | lot_id, material_id, lot_number, received_qty, remaining_qty, received_date | lot_number: unique ต่อ material — **format ยังไม่กำหนดตายตัวจากปอนด์** ระบบต้องรองรับการกรอกแบบอิสระในรอบ prototype (ดูหมายเหตุด้านล่าง) |
| StockTransaction | txn_id, material_id, type (Receipt/Reservation/ReservationRelease/Issue), qty, lot_id, ref_doc, timestamp | ทุกธุรกรรมต้องอัปเดตยอดทันที (real-time, ไม่ batch job) ตาม ECP-007/010; แยกยอด "physical stock" กับ "available to promise" เสมอ |
| ProductionOrder | production_order_id, po_line_id, assigned_to, status, planned_qty | status: `Pending → Assigned → InProgress → Completed` (หรือ `Cancelled` ถ้า PO ต้นทางถูกยกเลิกก่อน Assigned) |
| Batch | batch_id, batch_number (unique), production_order_id, finished_good_id, produced_qty, lots_used[], status | status: `InProgress → Completed → QCPending → QCApproved/QCRejected → (ถ้า Approved) ReadyToShip → Shipped`; batch_number format ยังไม่กำหนดตายตัวจากปอนด์ เช่นเดียวกับ lot_number |
| QCInspection | qc_id, batch_id, inspector_id, result (Approved/Rejected), remarks, inspected_at | ผลตรวจเป็น manual judgment ของเจ้าหน้าที่ QA ไม่มีเกณฑ์ตัวเลขตายตัวใน prototype (⚠ assumption ตาม Epic 5) |
| Shipment | shipment_id, po_id, batch_id(s), shipped_date (≤ today), status (Draft/Shipped/Delivered) | สร้างได้เฉพาะจาก Batch สถานะ QCApproved เท่านั้น (ECP-018) |
| Invoice | invoice_id, invoice_number (unique), po_id, issue_date, amount (THB, ทศนิยม 2 ตำแหน่ง), status (Issued/PartiallyPaid/Paid) | amount = Σ(quantity × unit_price) ของ PO lines — **ไม่มีการคำนวณภาษี/VAT อัตโนมัติ** (out-of-scope, ไม่เชื่อมระบบบัญชี/ภาษีจริง); ออกได้ 1 ใบต่อ PO เท่านั้นในรอบ prototype (ECP-020 AC2) |
| Payment | payment_id, invoice_id, amount (>0, ≤ยอดคงค้าง), payment_date (≤ today), method (text, manual) | ห้ามรับชำระเกินยอดคงค้างของ invoice (ECP-021 AC3) |
| User | user_id, username (unique), full_name, role_id, status (Active/Inactive) | username ห้ามซ้ำ (ECP-023 AC3) |
| Role | role_id, role_name, permissions[] (screen/action → allow/deny) | ปรับผ่านหน้าจอ config ได้โดยไม่ต้องแก้โค้ด (ECP-024); ห้าม config จนไม่มี role ใดเข้าถึงหน้าจัดการสิทธิ์ได้เลย (ECP-024 AC2) |
| AuditLog | log_id, user_id, action_type (Login/LoginFailed/Create/Update/Approve/Cancel/...), entity_type, entity_id, timestamp, detail | **Append-only เสมอ — ห้ามแก้ไข/ลบแม้แต่โดย Admin** (ECP-026 AC3); ต้องบันทึกทั้ง login สำเร็จและไม่สำเร็จ (ECP-025) |

### Currency & Company Scope
- Single company, สกุลเงินเดียว THB ทุกฟิลด์จำนวนเงินในระบบ (invoice, payment, unit_price) — ไม่มีฟิลด์ currency/company ให้เลือกในรอบ prototype (ตาม brief §3 "Single company, สกุลเงินเดียว")
- ตัวเลขจำนวนเงินเก็บเป็นทศนิยม 2 ตำแหน่งเสมอ (บาท.สตางค์)

### หมายเหตุเปิด (ไม่ block แต่ Tech-Lead/QA ควรทราบ)
1. รูปแบบ (format) ของ Lot number และ Batch number ยังไม่ถูกกำหนดจากปอนด์ — เสนอให้ Tech-Lead ออกแบบ format กลาง (เช่น `L-YYYYMMDD-NNN`, `B-YYYYMMDD-NNN`) ในขั้นตอนถัดไป แล้วนำกลับมายืนยันกับปอนด์ผ่าน CLAUDE.md domain knowledge (ตามที่ CLAUDE.md เตรียมช่องไว้แล้ว: "เพิ่มกติกาที่ปอนด์ตอบซ้ำๆ ไว้ตรงนี้ เช่น format Lot number")
2. เกณฑ์ QA/QC ผ่าน/ไม่ผ่านเชิงรายละเอียด (Epic 5) เป็น ⚠ assumption — ควรยืนยันกับปอนด์ระหว่าง UAT
3. ตัวเลข Business Key Value ทั้ง 5 ข้อที่ใช้ใน AC (≤1 นาที, ≤5 นาที, 100%, 80%) เป็นข้อเสนอของ PO ที่ยังรอปอนด์ยืนยัน — QA ควรออกแบบ test case ให้ปรับตัวเลข target ได้ง่ายหากปอนด์เปลี่ยนภายหลัง
4. จำนวนผู้ใช้ทดสอบจริงต่อ role สำหรับ UAT (ECP-035 AC3) เป็นหน้าที่ QA สอบถามปอนด์ตอนวางแผน UAT ไม่ใช่ของ BA
