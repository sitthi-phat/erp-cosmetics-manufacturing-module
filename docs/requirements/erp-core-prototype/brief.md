# Product Brief — ERP Core Prototype (Order-to-Cash) สำหรับโรงงานผลิตเครื่องสำอาง

Status: **BLOCKED_NEED_INPUT** (รอคำตอบจากปอนด์ก่อนส่งต่อ BA)

## สรุปภาษาไทย
ปอนด์ต้องการ prototype ระบบ ERP ที่ครอบคลุม flow ตั้งแต่จัดการข้อมูลลูกค้า → เปิด Purchasing
Order → ตรวจ stock วัตถุดิบ/สูตร → ส่งงานฝ่ายผลิต → ติดตามสถานะ → จัดส่ง → ออก invoice/เก็บเงิน
พร้อม User Management, audit log และ dashboard แยกตามส่วนงาน ทีม PO เห็นว่า flow หลักที่ต้องการ
ชัดเจนดีในระดับภาพกว้าง แต่ **ยังขาดข้อมูล 2 เรื่องที่กระทบ Business Key Value และ Definition of
Done โดยตรง** คือ (1) ปัญหา/pain point ปัจจุบันที่ทำให้ต้องสร้างระบบนี้ตอนนี้ และ (2) ตัวชี้วัด
ความสำเร็จที่วัดผลได้จริงพร้อมตัวเลข baseline/target นอกจากนี้ยังมีรายละเอียดขอบเขต prototype
(บทบาทผู้ใช้, การเชื่อมระบบภายนอก, จำนวนบริษัท/สกุลเงิน) ที่ต้องยืนยันก่อน เพื่อไม่ให้ BA ต้องเดา
business logic เอง จึงขอส่งคำถามชุดเดียวกลับไปยังปอนด์ก่อนตั้งสถานะ READY_FOR_BA

---

## 1. ภาพรวม/ปัญหาทางธุรกิจ (Business Problem)

จากข้อความ raw requirement ปอนด์ระบุ "อยากได้ระบบที่ช่วยบริหารจัดการ" ครอบคลุม flow
Order-to-Cash เต็มรูปแบบของโรงงานผลิตเครื่องสำอาง (ลูกค้า → PO → stock/สูตร → ผลิต → จัดส่ง →
invoice/เก็บเงิน) — นี่คือ **สิ่งที่อยากได้ (the "what")** แต่ยังไม่มีการระบุ **ปัญหาที่แท้จริงที่ทำให้
ต้องสร้างระบบนี้ตอนนี้ (the "why")** เช่น ปัจจุบันทำงานด้วยวิธีใด (Excel/กระดาษ/ระบบแยกส่วน),
เกิด pain point อะไร (ข้อมูลไม่ตรงกันระหว่างแผนก, ตรวจสอบ traceability ช้า, PO ตกหล่น, invoice
ผิดพลาด ฯลฯ), และความรุนแรง/ความถี่ของปัญหานั้น

จากความรู้โดเมน (โรงงานผลิตเครื่องสำอางต้องมี GMP + traceability Lot วัตถุดิบ → Batch การผลิต →
สินค้าสำเร็จรูปเสมอ) PO **สันนิษฐาน** ว่าปัญหาที่เป็นไปได้คือกระบวนการปัจจุบันแยกส่วนกันตามแผนก
(sales, คลัง, ผลิต, บัญชี) ทำให้ตรวจสอบย้อนกลับและติดตามสถานะงานข้ามแผนกทำได้ช้า/ไม่ตรงกัน —
**แต่นี่เป็นเพียงสมมติฐาน ไม่ใช่ข้อเท็จจริงที่ยืนยันแล้ว** จึงต้องขอคำยืนยันจากปอนด์ (ดูคำถามข้อ 1)

## 2. Business Key Value (ตัวชี้วัดความสำเร็จ)

**PENDING — ยังไม่สามารถระบุได้อย่างมีความหมาย** เนื่องจากไม่มี baseline ปัจจุบันหรือ target
ที่ปอนด์ต้องการวัด PO จะไม่เดาตัวเลขหรือ metric แทนปอนด์ (เช่น "ลด lead time จาก X เป็น Y")
เพราะเป็นการเดา business intent ซึ่งขัดกับกติกาการทำงาน — ต้องรอคำตอบจากปอนด์ (ดูคำถามข้อ 2)
ก่อนจึงจะเติมหัวข้อนี้ให้สมบูรณ์และผ่าน gate ได้

## 3. ขอบเขต (Scope) — ระดับ Prototype

หมายเหตุ: นี่คือ **prototype** ไม่ใช่ production เต็มรูปแบบ ขอบเขตด้านล่างเป็นร่างจาก raw
requirement โดยตรง ส่วนที่มีเครื่องหมาย "⚠ รอยืนยัน" คือส่วนที่ยังไม่มีข้อมูลพอจะฟันธง

### In-scope (draft)
- Master data ลูกค้า (สร้าง/แก้ไข/ดูข้อมูลลูกค้า)
- เปิด Purchasing Order (PO) จากลูกค้า
- ตรวจสอบ stock วัตถุดิบ/ส่วนผสมตามสูตร (formula/BOM) ว่าพอสำหรับ PO หรือไม่
- ส่งงานให้ฝ่ายผลิต: ตรวจสอบคำสั่ง, assign งาน, บันทึกการดำเนินการผลิต (พร้อม traceability
  Lot วัตถุดิบ → Batch ผลิต ตาม GMP)
- ติดตามสถานะงานผลิต/คำสั่งซื้อแบบ end-to-end
- บันทึกการจัดส่ง
- ออก invoice และบันทึกการเรียกเก็บเงิน
- User Management: สร้าง/แก้ไขผู้ใช้, กำหนดสิทธิ์ตาม role, audit log ทุกครั้งที่ login/ทำรายการสำคัญ
- Dashboard แยกตามส่วนงาน (อย่างน้อย: sales/PO, คลัง/stock, ผลิต, จัดส่ง, บัญชี/การเงิน) ⚠ รอยืนยัน
  รายชื่อ dashboard ที่ต้องมี

### Out-of-scope (draft, สำหรับรอบ prototype นี้)
- การเชื่อมต่อระบบบัญชี/ภาษี หรือ payment gateway จริง ⚠ รอยืนยัน (ข้อ 3)
- การจัดการสูตร/BOM แบบเต็มรูป (version control ของสูตร, การคำนวณต้นทุนสูตรละเอียด) — สมมติว่า
  prototype รับสูตรสำเร็จรูป/ข้อมูลจำลองมาใช้ตรวจ stock เท่านั้น ⚠ รอยืนยัน
- Multi-company / multi-currency ⚠ รอยืนยัน (ข้อ 5)
- Mobile app (สมมติว่า prototype เป็น web application เท่านั้น) ⚠ รอยืนยันถ้าไม่ใช่
- การเชื่อมต่อ IoT/เครื่องจักรฝ่ายผลิตแบบ real-time (สมมติว่าฝ่ายผลิตบันทึกผลด้วยมือผ่านระบบ)
- Automated notification/alert (email/SMS) เว้นแต่ระบุเพิ่มเติม

## 4. Definition of Done (feature level)

**PENDING finalization** — DoD ต้องอิงกับขอบเขตที่ยืนยันแล้ว (คำถามข้อ 3, 4, 5) จึงยังระบุแบบ
verifiable ครบถ้วนไม่ได้ในตอนนี้ ร่างเบื้องต้นที่จะปรับหลังได้คำตอบ:
- [ ] สามารถ demo flow ทั้งเส้นทาง (ลูกค้า → PO → ตรวจ stock → ผลิต → จัดส่ง → invoice/เก็บเงิน)
      แบบ end-to-end ด้วยข้อมูลตัวอย่าง (seed/mock data) ได้ในรอบเดียวไม่สะดุด
- [ ] ทุก Lot วัตถุดิบที่ถูกใช้ในการผลิตสามารถตรวจสอบย้อนกลับไปถึง Batch และสินค้าสำเร็จรูปได้
      (traceability)
- [ ] มี User Management ที่แบ่ง role ได้ตามที่ตกลง และทุกการ login/ทำรายการสำคัญถูกบันทึกใน
      audit log ตรวจสอบย้อนหลังได้
- [ ] มี Dashboard แยกตามส่วนงานตามรายการที่ตกลง (ข้อ 3 ด้านบน)
- [ ] ปอนด์ทดสอบ/รีวิว prototype แล้วยืนยันว่าครอบคลุม flow ธุรกิจตามที่ต้องการ

## 5. ผู้ใช้งาน/บทบาทหลัก (Users/Roles) — ร่าง ⚠ รอยืนยัน

สมมติฐานเบื้องต้น (อิงจาก flow ที่ปอนด์อธิบาย):
- Sales/CS — จัดการลูกค้า, เปิด PO
- คลังสินค้า (Warehouse) — ตรวจสอบ stock วัตถุดิบ
- ฝ่ายผลิต (Production) — ตรวจสอบ/assign/ดำเนินการผลิต
- ฝ่ายจัดส่ง (Logistics) — บันทึกการจัดส่ง
- ฝ่ายบัญชี/การเงิน (Finance) — ออก invoice, เรียกเก็บเงิน
- ผู้ดูแลระบบ (Admin) — User Management, ดู audit log

ยังไม่ยืนยันว่าต้องมี role ฝ่ายควบคุมคุณภาพ (QA/QC) แยกต่างหากหรือไม่ (สำคัญต่อ GMP)

## 6. ข้อจำกัด/Constraint ที่ทราบแล้ว

- ต้องรองรับ GMP และ traceability: Lot วัตถุดิบ → Batch การผลิต → สินค้าสำเร็จรูป (เป็น requirement
  แฝงเสมอสำหรับโรงงานผลิตเครื่องสำอาง ตาม domain knowledge กลาง)
- ต้องมี audit log บันทึกการเข้าใช้งานระบบทุกครั้ง
- ยังไม่ทราบว่ามีระบบเดิม (ERP/accounting/CRM) ที่ต้องอยู่ร่วมกันหรือ migrate ข้อมูลหรือไม่
- ยังไม่ทราบ target ผู้ใช้งานจริงจำนวนกี่คน/กี่แผนก ใช้ทดสอบ prototype

## 7. ข้อสมมติ (Assumptions ที่ PO ตั้งเอง หากปอนด์ไม่ตอบ/ไม่แย้ง)

หากปอนด์ไม่ระบุเป็นอย่างอื่น PO จะใช้สมมติฐาน default ต่อไปนี้ในการเขียน brief ฉบับสมบูรณ์:
1. Prototype ใช้ข้อมูลจำลอง (seed/mock data) ไม่เชื่อมระบบบัญชี/payment gateway จริง
2. Single company, สกุลเงินเดียว (THB)
3. เป็น web application, ไม่รวม mobile app
4. Role ผู้ใช้งานตามร่างในข้อ 5 (ไม่มี QA/QC แยก role) เว้นแต่ปอนด์ระบุเพิ่ม
5. ฝ่ายผลิตบันทึกผลการผลิตด้วยมือผ่านระบบ ไม่เชื่อมต่อเครื่องจักร/IoT

ข้อสมมติเหล่านี้กระทบ **scope** เท่านั้น ไม่ใช้แทนการเดา **Business Key Value** (ข้อ 2) ซึ่งเป็น
business intent ที่ต้องให้ปอนด์ยืนยันเองเท่านั้น

## 8. คำถามถึงปอนด์ (Open Questions ที่ block การส่งต่อ BA)

ดูรายการเต็มใน `pipeline/status.json` → `history[].questions_for_pond` ของรอบนี้ สรุปหัวข้อ:
1. ปัญหา/pain point ปัจจุบันที่แท้จริงคืออะไร (baseline)
2. Business Key Value / ตัวชี้วัดความสำเร็จที่วัดผลได้ พร้อมตัวเลข target
3. ขอบเขตการเชื่อมระบบภายนอก (บัญชี/payment) ใน prototype นี้
4. รายชื่อ role ผู้ใช้งานที่ต้องมีครบถ้วน (รวม QA/QC หรือไม่)
5. จำนวนบริษัท/สาขา และสกุลเงินที่ต้องรองรับ
