# Product Brief — ERP Core Prototype (Order-to-Cash) สำหรับโรงงานผลิตเครื่องสำอาง

Status: **READY_FOR_BA** (Gate 2 Feedback Round 1 — re-triage complete, see new section at end of file)

## สรุปภาษาไทย

ปอนด์ต้องการ prototype ระบบ ERP ตัวแรกของโรงงาน (ปัจจุบันยังไม่มีระบบ ERP ใช้ Excel/กระดาษแยก
ตามแผนก) ครอบคลุม flow ลูกค้า → เปิด PO → ตรวจสอบ stock วัตถุดิบ/สูตร → ส่งงานฝ่ายผลิต →
ติดตามสถานะ → จัดส่ง → ออก invoice/เก็บเงิน พร้อม User Management (7 roles รวม QA/QC),
audit log และ dashboard แยกตามส่วนงาน **จุดเน้นหลักคือปัญหา stock วัตถุดิบที่มองไม่เห็นแบบ
real-time** และ **ความง่ายในการใช้งาน/นำทางผู้ใช้เอง** เพราะ prototype นี้จะถูกใช้เป็นเครื่องมือ
สำรวจ/นิยาม user journey ของแต่ละแผนกไปพร้อมกัน ปอนด์ตอบคำถามครบทั้ง 5 ข้อแล้ว brief ฉบับนี้
จึงปรับเป็นฉบับสมบูรณ์และผ่าน gate พร้อมส่งต่อ BA — มีเพียง**ตัวเลข target ของ Business Key
Value ที่ PO เสนอเป็นข้อสมมติเริ่มต้น** (ปอนด์ไม่ได้ให้ baseline ที่วัดจริง) ซึ่งทำเครื่องหมายไว้
ชัดเจนว่า "รอปอนด์ยืนยัน" — ไม่กระทบการส่งต่อ BA เพราะมีตัวเลขตั้งต้นที่วัดผลได้แล้ว

**อัปเดต 2026-07-07 (Gate 2 รอบ 1):** ปอนด์ทดสอบ prototype บน localhost แล้ว **ยังไม่ approve
Gate 2** ส่ง feedback 10 ข้อกลับมาพร้อมสั่งตรงให้ PO/BA ให้ความสำคัญเป็นพิเศษ ดูรายละเอียดการ
triage ทั้งหมดที่ท้ายเอกสารนี้ (หัวข้อ "Gate 2 Feedback Round 1 — Scope Update") สรุปสั้นๆ:
5 ข้อเป็น defect ของฟีเจอร์เดิมที่ต้องแก้ก่อน (ต้อง Must ทั้งหมด), 5 ข้อเป็น scope ใหม่ที่จำเป็น
ต่อการใช้งานจริง (BOM management, production auto-calc, QC ขาเข้า, ข้อมูลภาษีลูกค้า, เอกสาร
ใบกำกับภาษีไทย — ก็ Must ทั้งหมดเพราะปอนด์ระบุตรง), และ 2 ข้อ (UX/UI overhaul ทั้งระบบ,
responsive ทุกขนาดจอ) ใหญ่เกินสำหรับ prototype รอบเดียว จึงเสนอ scope ที่ทำได้จริงรอบนี้ +
ส่วนที่เหลือเข้า backlog อย่างชัดเจน ไม่มีคำถามที่ต้อง block กลับปอนด์ในรอบนี้

## 1. ภาพรวม/ปัญหาทางธุรกิจ (Business Problem)

**ยืนยันแล้วจากปอนด์:** ปัจจุบันโรงงานยังไม่เคยมีระบบ ERP มาก่อน (นี่คือระบบ ERP ตัวแรก) และ
แต่ละแผนก (Sales, คลัง, ฝ่ายผลิต, QA/QC, จัดส่ง, บัญชี) ทำงานแยกส่วนกันด้วย Excel/เอกสารกระดาษ
ผลที่ตามมา:
- ข้อมูลระหว่างแผนกไม่เชื่อมกัน ต้องเดินเอกสาร/โทร/ส่งไฟล์ Excel ข้ามแผนกเอง
- **ปัญหาหลักที่สุดคือ stock วัตถุดิบ**: ผู้ใช้งาน (โดยเฉพาะฝ่ายจัดซื้อ/ผลิต) ไม่รู้ว่าวัตถุดิบ
  ชนิดใดเหลือเท่าไหร่/ขาดหรือไม่ในเวลานั้นๆ หรือรู้แต่ตัวเลขไม่ update ตามเวลาจริง (นับสต็อก/
  reconcile เป็นรอบ ไม่ใช่ real-time) ทำให้เสี่ยงเปิด PO ไปแล้วพบภายหลังว่าวัตถุดิบไม่พอผลิต
- ตรวจสอบ traceability ของ Lot วัตถุดิบ → Batch การผลิต → สินค้าสำเร็จรูป (ต้องมีตาม GMP) ทำได้
  ช้าเพราะข้อมูลกระจัดกระจายอยู่คนละที่คนละแผนก

**ข้อสังเกตสำคัญเพิ่มเติมจากปอนด์:** ผู้ใช้งานจริง (staff แต่ละแผนก) ไม่มีความชำนาญเรื่อง
User Journey ของระบบ ERP มาก่อน ดังนั้น prototype นี้มี "หน้าที่คู่" คือ (1) แก้ปัญหาข้างต้น
และ (2) **เป็นเครื่องมือให้ปอนด์และทีมใช้สำรวจ/นิยาม user journey ที่แท้จริงของแต่ละแผนก** —
หมายความว่าระบบต้องใช้งานง่าย นำทางผู้ใช้ได้เองโดยไม่ต้องอบรมมาก (ดูหัวข้อ Usability ในข้อ 3
และ DoD ข้อ 4)

## 2. Business Key Value (ตัวชี้วัดความสำเร็จ)

ปอนด์ยืนยันให้เอาทุกมิติ (traceability, cycle time, stock) โดย**เน้นหนักที่ stock visibility
เป็น pain หลัก** เนื่องจากปอนด์ไม่มีตัวเลข baseline ที่วัดจริงในปัจจุบัน (ไม่เคยมีระบบวัด) PO จึง
เสนอ**ตัวเลข baseline โดยประมาณจากลักษณะการทำงานแบบ manual ปัจจุบัน + target เริ่มต้นที่
สมเหตุสมผลสำหรับ prototype** ทุกตัวเลขด้านล่างมีป้ายกำกับ **"⚠ ข้อสมมติ (รอปอนด์ยืนยัน)"**
ชัดเจน — ไม่ใช่ข้อเท็จจริงที่ยืนยันแล้ว

| # | ตัวชี้วัด | Baseline ปัจจุบัน (⚠ ข้อสมมติ) | Target ของ Prototype (⚠ ข้อสมมติ) |
|---|---|---|---|
| 1 | **Stock visibility** (เวลาที่ผู้ใช้ต้องใช้เพื่อรู้ว่าวัตถุดิบชนิดหนึ่งเหลือเท่าไหร่ ณ ปัจจุบัน) | ~15–30 นาที/ครั้ง (ต้องเดินคลัง/โทรถาม/เปิดหลายไฟล์ Excel) และตัวเลขอาจไม่ตรงกับของจริงถึง 1 วัน (update เป็นรอบ ไม่ real-time) | ดูยอดคงเหลือ real-time ในหน้าเดียว ภายใน ≤ 1 นาที และตัวเลขอัปเดตทันทีที่มีธุรกรรม (รับเข้า/เบิกผลิต/จอง PO) |
| 2 | **Traceability lookup time** (Lot วัตถุดิบ → Batch ผลิต → สินค้าสำเร็จรูป) | ~30–60 นาที/ครั้ง (ค้นเอกสารกระดาษ/ไฟล์แยกแผนก) | ค้นหาและแสดงผล traceability เต็มสาย ภายใน ≤ 5 นาที ผ่านระบบ |
| 3 | **PO-to-invoice cycle (เชิงกระบวนการ ไม่ใช่ lead time การผลิตจริง)** | ไม่มีข้อมูลวัดจริง (กระบวนการกระจายหลายแผนก ไม่เคยจับเวลา) | ผู้ใช้สามารถ demo/เดินกระบวนการ PO → ตรวจ stock → ผลิต → จัดส่ง → invoice ครบ end-to-end ในระบบเดียวโดยไม่ต้องออกไปใช้ Excel/เอกสารภายนอกระหว่างทาง (วัดผลจาก UAT ไม่ใช่ production data) |
| 4 | **Stock-out surprise** (เปิด PO ไปแล้วพบภายหลังว่าวัตถุดิบไม่พอ) | ไม่มีข้อมูล % ที่วัดจริง (ทราบว่าเกิดขึ้นเป็นประจำจากคำบอกเล่า) | ระบบต้องแจ้งเตือน insufficient stock ณ จุดเปิด PO/ก่อนส่งงานผลิต **100% ของ test case ที่ทดสอบใน UAT** (prototype-level, ไม่ใช่ production metric) |
| 5 | **Usability / self-navigation** (ผู้ใช้แต่ละ role ไม่ชำนาญ user journey มาก่อน) | ไม่เคยใช้ระบบ ERP มาก่อน (baseline = 0) | ผู้ใช้ทดสอบใหม่ (คนละคนต่อ role) ทำ task หลักของ role ตนเองให้จบได้โดยไม่ต้องมี training/คู่มือ อย่างน้อย 80% ของ scenario ที่ทดสอบใน UAT (ใช้เป็นข้อมูล generate/ยืนยัน user journey จริงของแต่ละแผนกไปพร้อมกัน) |

**หมายเหตุ:** ตัวเลข baseline/target ทั้งหมดข้างต้นเป็นข้อเสนอเริ่มต้นของ PO เพื่อให้ brief มี
Business Key Value ที่วัดผลได้พอส่งต่อ BA/QA เขียน acceptance criteria ได้ — **ปอนด์ควรยืนยันหรือ
แก้ไขตัวเลขเหล่านี้ก่อนหรือระหว่าง UAT** (ไม่ block การส่งต่อ BA แต่ QA ต้องออกแบบ test case ให้
วัดตัวเลขเหล่านี้ได้จริง)

## 3. ขอบเขต (Scope) — ระดับ Prototype

หมายเหตุ: นี่คือ **prototype** ไม่ใช่ production เต็มรูปแบบ ทุกข้อด้านล่างยืนยันจากคำตอบปอนด์แล้ว

### In-scope
- Master data ลูกค้า (สร้าง/แก้ไข/ดูข้อมูลลูกค้า)
- เปิด Purchasing Order (PO) จากลูกค้า
- ตรวจสอบ stock วัตถุดิบ/ส่วนผสมตามสูตร (formula/BOM) ว่าพอสำหรับ PO หรือไม่ — **real-time
  stock visibility คือ priority สูงสุดของ scope นี้** ยอดคงเหลือต้องอัปเดตทันทีเมื่อมีธุรกรรม
  (รับเข้า/เบิกผลิต/จอง PO) ไม่ใช่ batch update เป็นรอบ
- ส่งงานให้ฝ่ายผลิต: ตรวจสอบคำสั่ง, assign งาน, บันทึกการดำเนินการผลิต พร้อม traceability
  Lot วัตถุดิบ → Batch ผลิต ตาม GMP
- ขั้นตอนตรวจสอบคุณภาพ (QA/QC) ก่อน/หลังผลิต ตามบทบาท QA/QC ที่ระบุในข้อ 5 (อย่างน้อยระดับ
  บันทึกผลตรวจ/อนุมัติ-ไม่อนุมัติ Batch — รายละเอียด business logic ให้ BA ชี้แจงเพิ่มกับปอนด์
  ถ้าจำเป็น)
- ติดตามสถานะงานผลิต/คำสั่งซื้อแบบ end-to-end
- บันทึกการจัดส่ง
- ออก invoice และบันทึกการเรียกเก็บเงิน (mock/manual reconcile ไม่เชื่อม payment gateway จริง)
- User Management: สร้าง/แก้ไขผู้ใช้, **กำหนด/ปรับ role และสิทธิ์ได้ง่ายผ่านหน้าจอ config**
  (ไม่ต้องแก้โค้ด) เนื่องจาก role/permission จะถูกปรับบ่อยระหว่างช่วงสำรวจ user journey,
  audit log ทุกครั้งที่ login/ทำรายการสำคัญ
- Dashboard แยกตามส่วนงาน ครบ 7 roles: Sales/CS, คลังสินค้า (เน้น stock real-time), ฝ่ายผลิต,
  QA/QC, ฝ่ายจัดส่ง, บัญชี/การเงิน, Admin (ภาพรวม user/audit log)
- **Usability/Navigation**: ระบบต้องมี onboarding/นำทางในตัว (เช่น เมนูตาม role ชัดเจน,
  ทำ task สำคัญได้โดยไม่ต้องอ่านคู่มือ) เพราะผู้ใช้ไม่มีประสบการณ์ ERP มาก่อน และ prototype
  จะถูกใช้สำรวจ user journey จริงของแต่ละแผนกไปพร้อมกัน
- Single company, สกุลเงินเดียว (THB)
- **(เพิ่ม Gate 2 Round 1)** BOM Management UI, Production auto-calc จาก BOM, QC incoming
  inspection (ยกระดับ priority), Customer tax fields, เอกสารใบกำกับภาษีไทย — ดูรายละเอียดเต็ม
  ในหัวข้อ "Gate 2 Feedback Round 1 — Scope Update" ท้ายเอกสาร

### Out-of-scope (สำหรับรอบ prototype นี้ — ยืนยันแล้ว)
- การเชื่อมต่อระบบบัญชี/ภาษี หรือ payment gateway จริง — ใช้ mock/seed data แทนทั้งหมด
- Multi-company / multi-currency
- การจัดการสูตร/BOM แบบเต็มรูป (version control ของสูตร, cost roll-up ละเอียด) — รับสูตร
  สำเร็จรูป/ข้อมูลจำลองมาใช้ตรวจ stock เท่านั้น (BOM Management UI ที่เพิ่มรอบ Gate 2 ยังคง
  **ไม่มี version history** — แก้ไข BOM ปัจจุบันในที่เดิม (in-place) เท่านั้น ตามที่ยืนยันไว้เดิม)
- Mobile app (เป็น web application เท่านั้น)
- การเชื่อมต่อ IoT/เครื่องจักรฝ่ายผลิตแบบ real-time (ฝ่ายผลิตบันทึกผลด้วยมือผ่านระบบ)
- Automated notification/alert ภายนอกระบบ (email/SMS)
- Production-grade performance/security hardening, load testing, disaster recovery
  (เป็น prototype สำหรับ demo/สำรวจ user journey ไม่ใช่ระบบ production จริง)
- **(เพิ่ม Gate 2 Round 1)** UX/UI redesign แบบเต็มรูปทุกหน้าจอ, responsive แบบเต็มทุกขนาด
  หน้าจอ (รวมมือถือ) — ย้ายเข้า backlog ดูรายละเอียด scope-down ท้ายเอกสาร

## 4. Definition of Done (feature level — Gate 2 Round 1 เดิม)

- [ ] สามารถ demo flow ทั้งเส้นทาง (ลูกค้า → PO → ตรวจ stock → ผลิต (รวมขั้นตอน QA/QC) →
      จัดส่ง → invoice/เก็บเงิน) แบบ end-to-end ด้วยข้อมูลตัวอย่าง (seed/mock data) ได้ในรอบเดียว
      ไม่สะดุด
- [ ] ยอดคงเหลือ stock วัตถุดิบแสดงผลแบบ real-time ในหน้าคลัง/dashboard — ทดสอบแล้วว่าตัวเลข
      เปลี่ยนทันทีหลังทำธุรกรรม (รับเข้า/เบิกผลิต/จอง PO) โดยไม่ต้อง refresh/รอ batch job
- [ ] ทุก Lot วัตถุดิบที่ถูกใช้ในการผลิตสามารถตรวจสอบย้อนกลับไปถึง Batch และสินค้าสำเร็จรูปได้
      (traceability) ภายในเวลาที่กำหนดในข้อ 2
- [ ] มี User Management ครบ 7 roles (Sales/CS, คลังสินค้า, ฝ่ายผลิต, QA/QC, ฝ่ายจัดส่ง,
      บัญชี/การเงิน, Admin) กำหนด/ปรับสิทธิ์ role ได้ผ่านหน้าจอโดยไม่ต้องแก้โค้ด และทุกการ
      login/ทำรายการสำคัญถูกบันทึกใน audit log ตรวจสอบย้อนหลังได้
- [ ] มี Dashboard แยกตามทั้ง 7 ส่วนงานตามที่ระบุในข้อ 3
- [ ] ผ่าน UAT รอบ usability: ผู้ใช้ทดสอบใหม่แต่ละ role ทำ task หลักของตนได้โดยไม่ต้องมี
      training/คู่มือ ตาม target ในข้อ 2 (ข้อ 5 ของตาราง)
- [ ] ปอนด์ทดสอบ/รีวิว prototype แล้วยืนยันว่าครอบคลุม flow ธุรกิจตามที่ต้องการ และใช้งานได้จริง
      ในการสำรวจ user journey ของแต่ละแผนก

**หมายเหตุ:** DoD ชุดนี้คือของรอบ Gate 2 Round 1 (ที่ปอนด์เพิ่งทดสอบแล้วยังไม่ผ่าน) ดู DoD ฉบับ
ใหม่ของรอบถัดไป (Round 2) ในหัวข้อท้ายเอกสาร "Gate 2 Feedback Round 1 — Scope Update"

## 5. ผู้ใช้งาน/บทบาทหลัก (Users/Roles) — ยืนยันแล้ว (ครบ 7 roles)

1. **Sales/CS** — จัดการข้อมูลลูกค้า, เปิด PO
2. **คลังสินค้า (Warehouse)** — ตรวจสอบ/บริหาร stock วัตถุดิบ real-time
3. **ฝ่ายผลิต (Production)** — ตรวจสอบ/assign/ดำเนินการผลิต, บันทึก Lot/Batch
4. **QA/QC** — ตรวจสอบคุณภาพ/อนุมัติ Batch ตาม GMP (role ใหม่ที่เพิ่มตามคำยืนยันปอนด์)
5. **ฝ่ายจัดส่ง (Logistics)** — บันทึกการจัดส่ง
6. **ฝ่ายบัญชี/การเงิน (Finance)** — ออก invoice, บันทึกการเรียกเก็บเงิน
7. **ผู้ดูแลระบบ (Admin)** — User Management, กำหนด role/สิทธิ์, ดู audit log

**ข้อกำหนดเพิ่มเติม:** role/permission ต้อง config ได้ง่ายผ่านหน้าจอ (ไม่ต้องแก้โค้ด/deploy ใหม่)
เนื่องจาก prototype นี้จะถูกใช้สำรวจและปรับ user journey ของแต่ละแผนกอย่างต่อเนื่อง — รายละเอียด
สิทธิ์แบบละเอียด (permission matrix ต่อ role ต่อ action) ให้ BA ออกแบบร่วมกับ Tech Lead ใน
ขั้นตอนถัดไป โดยยึดหลัก least-privilege ตาม role ข้างต้น

## 6. ข้อจำกัด/Constraint ที่ทราบแล้ว

- ต้องรองรับ GMP และ traceability: Lot วัตถุดิบ → Batch การผลิต → สินค้าสำเร็จรูป (เป็น
  requirement แฝงเสมอสำหรับโรงงานผลิตเครื่องสำอาง ตาม domain knowledge กลาง) รวมขั้นตอน QA/QC
  อนุมัติ Batch
- ต้องมี audit log บันทึกการเข้าใช้งานระบบทุกครั้ง
- ไม่มีระบบเดิม (ERP/accounting/CRM) ที่ต้องอยู่ร่วมกันหรือ migrate ข้อมูล — เป็นระบบ ERP ตัวแรก
  ของโรงงาน ใช้ mock/seed data สำหรับ prototype
- Single company, สกุลเงินเดียว (THB)
- **Constraint ด้าน user**: ผู้ใช้งานจริงไม่มีประสบการณ์ ERP/ไม่ชำนาญ user journey มาก่อน —
  ทุกหน้าจอ/flow ต้องออกแบบให้ใช้งานได้เองโดยไม่ต้องมี training ยาว (ผลต่อ UX requirement ใน
  ขั้นตอน BA/design ถัดไป)
- ยังไม่ทราบ target จำนวนผู้ใช้งานจริง/กี่คนต่อแผนกที่จะเข้าทดสอบ prototype (ไม่ block BA
  แต่ QA ควรสอบถามเพิ่มตอนวางแผน UAT) — **อัปเดต:** ปอนด์ตอบแล้วระหว่างทาง QA ทำ test-plan =
  2–3 คน/role
- **(เพิ่ม Gate 2 Round 1)** ผู้ใช้งานจริงทดสอบบน localhost แล้วพบว่า prototype ยังมี defect/
  gap ในหลายหน้าจอหลัก และมีความต้องการ scope ใหม่ที่จำเป็นต่อการใช้งานจริง (ดูหัวข้อท้าย
  เอกสาร) — สิ่งนี้ยืนยันว่า prototype นี้กำลังทำหน้าที่ "เครื่องมือสำรวจ user journey" ตามที่
  ตั้งใจไว้ตั้งแต่ต้น (ข้อ 1) ได้จริง

## 7. ข้อสมมติที่ยังเปิดอยู่ (Assumptions ที่ต้องติดตาม)

ข้อสมมติต่อไปนี้ไม่ block การส่งต่อ BA แต่ควรติดตาม/ยืนยันระหว่างทาง:
1. ตัวเลข baseline/target ทั้งหมดในตาราง Business Key Value (ข้อ 2) เป็นข้อเสนอของ PO —
   ควรยืนยัน/ปรับกับปอนด์ก่อนหรือระหว่าง UAT
2. รายละเอียด business logic ของขั้นตอน QA/QC (เช่น ต้องอนุมัติ Batch ก่อนจัดส่งเสมอหรือไม่,
   เกณฑ์ผ่าน/ไม่ผ่าน) ยังไม่ระบุละเอียด — ให้ BA สอบถามปอนด์เพิ่มเมื่อเขียน user story ของ
   QA/QC หากจำเป็น
3. จำนวนผู้ใช้งานจริงที่จะเข้าทดสอบ prototype ต่อ role — ตอบแล้ว (2–3 คน/role)
4. **(เพิ่ม Gate 2 Round 1)** รูปแบบ discount บน invoice เป็นจำนวนเงินคงที่ต่อใบ (ไม่ใช่ % หรือ
   ต่อบรรทัด) — เป็น default ที่ PO เสนอจากตัวอย่างเอกสารที่ปอนด์แนบ ควรยืนยันกับปอนด์ระหว่าง UAT
5. **(เพิ่ม Gate 2 Round 1)** การรองรับหน้าจอมือถือ (phone) สำหรับ role ฝ่ายผลิต/QC ที่อาจบันทึก
   ข้อมูลหน้างานจริงในโรงงาน — ยังไม่ยืนยันจากปอนด์ว่าจำเป็นแค่ไหน PO เสนอ default = desktop/
   tablet ก่อนในรอบนี้ ดูรายละเอียดในหัวข้อท้ายเอกสาร

## 8. คำถามถึงปอนด์

ไม่มีคำถามที่ block การส่งต่อ BA ในรอบนี้ (ทั้งรอบแรกและ Gate 2 Round 1 นี้) — คำถามทั้ง 5 ข้อ
จากรอบแรกได้รับคำตอบครบแล้ว (ดู `pipeline/status.json` → `history[0].questions_for_pond`
สำหรับคำถาม-คำตอบฉบับเต็ม) ข้อสมมติที่ยังเปิดอยู่ในข้อ 7 ด้านบนมอบให้ BA/QA ติดตามต่อระหว่าง
เขียน/ทดสอบ user stories ไม่ใช่ประเด็นที่ต้อง block PO อีกต่อไป

## เอกสารใบกำกับภาษี — รายละเอียด scope ใหม่ (ข้อ 10b)

จากตัวอย่างที่ปอนด์แนบ (ดู `pond-gate2-feedback.md` §"รายละเอียดเอกสารตัวอย่าง") ต้องเพิ่ม/ปรับ
ดังนี้ (ให้ BA ออกแบบ story/AC โดยละเอียดต่อ ให้ Tech-Lead ปรับ ER diagram):

1. **CompanyProfile (entity ใหม่, admin-managed, ค่าเดียวระดับบริษัทเหมือน VATConfig)**:
   ชื่อบริษัท, ที่อยู่เต็ม, เลขประจำตัวผู้เสียภาษี 13 หลัก, โทรศัพท์, พื้นที่โลโก้ — ใช้เป็นข้อมูล
   "ผู้ออก" บนหัวเอกสาร
2. **Customer**: เพิ่ม `tax_id`, `registered_address` (ตามข้อ 9 ด้านบน)
3. **Invoice**: เพิ่มฟิลด์ `discount_amount` (จำนวนเงินคงที่ต่อใบ บาท ไม่ใช่ % — ⚠ assumption
   ของ PO จากตัวอย่างที่มีแค่ "หักส่วนลด" เป็นยอดเงินบรรทัดเดียว ยืนยันกับปอนด์ระหว่าง UAT ถ้า
   ต้องการ % หรือส่วนลดต่อบรรทัดในอนาคต), `payment_terms_days`/เงื่อนไขเครดิต (text) — ปรับสูตร
   คำนวณเป็น subtotal → หัก discount_amount → after-discount → VAT 7% (ใช้ VATConfig เดิมที่มี
   อยู่แล้ว ไม่ต้องสร้างใหม่) → grand total
4. **ฟังก์ชันแปลงจำนวนเงินเป็นตัวอักษรไทย** (Thai baht text) — algorithm ตายตัวตามหลักภาษาไทย
   ไม่มีความกำกวมทางธุรกิจ ไม่ต้องถามปอนด์
5. **Layout เอกสารพิมพ์/PDF**: หัวเอกสาร (ชื่อเอกสารสองภาษา+โลโก้+ผู้ออก+ลูกค้า+เลขที่+วันที่+
   เงื่อนไขชำระ), ตารางรายการ, สรุปยอด (subtotal/discount/after-discount/VAT/grand total/
   ตัวหนังสือไทย), ส่วนท้าย (หมายเหตุ + ช่องลายเซ็น 2 ช่องพร้อมวงเล็บชื่อ + พื้นที่ตรายาง) — ตรง
   ตามโครงสร้างตัวอย่างที่ปอนด์แนบทุกส่วน

## Definition of Done — Gate 2 Round 2 (แทนที่ชุดเดิมสำหรับรอบทดสอบถัดไป)

- [ ] Sales/CS เพิ่ม/ดู (ชื่อสินค้า+จำนวน+ราคา อ่านง่าย)/ลบบรรทัดสินค้าใน PO ได้ก่อนกดยืนยัน
- [ ] หน้า stock ค้นหาด้วยชื่อ/รหัสวัตถุดิบได้
- [ ] มีหน้า BOM Management ให้ผู้ใช้ธุรกิจสร้าง/ดู/แก้ไขสูตรของสินค้าสำเร็จรูปได้เองโดยไม่ต้อง
      พึ่ง developer/seed data
- [ ] หน้า trace: ช่องค้นหาเดียวรับ Lot/Batch/PO/Invoice number ได้ทั้งหมด และคืนผลลัพธ์ห่วงโซ่
      เต็มถูกต้องทั้ง 4 กรณี พร้อมคำอธิบาย Lot vs Batch ที่มองเห็นได้ในหน้าผลลัพธ์
- [ ] หน้า production: กรอก/เลือก Lot ที่ถูกต้องแล้วไม่ error อีก และปริมาณวัตถุดิบที่ต้องใช้ถูก
      คำนวณล่วงหน้าจาก BOM × จำนวนที่ผลิต ให้ฝ่ายผลิต review/ยืนยัน (ไม่ต้องพิมพ์ตัวเลขเอง)
- [ ] QC มีฟอร์มบันทึกผลตรวจวัตถุดิบขาเข้า (ECP-017 ยกระดับ Must) ใช้งานได้จริงจากเมนู QC
- [ ] หน้าสร้าง/แก้ไขลูกค้าเก็บเลขผู้เสียภาษี (13 หลัก) และที่อยู่จดทะเบียนได้
- [ ] Finance เปิดดูรายละเอียด invoice ใบเดียวได้
- [ ] Finance สร้าง/พิมพ์เอกสารใบกำกับภาษีที่มีโครงสร้างตรงกับตัวอย่างที่ปอนด์แนบ (หัวเอกสารผู้
      ออก+ลูกค้า+เลขผู้เสียภาษี, รายการสินค้า, subtotal/discount/VAT7%/grand total, จำนวนเงินเป็น
      ตัวอักษรไทย, ช่องลายเซ็น 2 ช่อง) ได้อย่างน้อย 1 invoice ตัวอย่าง โดยปอนด์เทียบกับตัวอย่างจริง
      แล้วยืนยันว่าตรงกัน
- [ ] หน้าจอที่ถูกแก้ไขในข้อ 1–6, 9, 10 ข้างต้น ดูสม่ำเสมอ/เป็นมืออาชีพขึ้นอย่างเห็นได้ชัดเทียบกับ
      รอบ 1 (redesign เต็มรูปทุกหน้าที่เหลือ **ไม่อยู่ในขอบเขตรอบนี้** ดู backlog)
- [ ] ระบบใช้งานได้ดีบน desktop และ tablet (portrait/landscape) สำหรับหน้าจอข้างต้น (full
      phone-width responsive **ไม่อยู่ในขอบเขตรอบนี้** ดู backlog)
- [ ] ปอนด์ทดสอบซ้ำบน localhost แล้วยืนยันว่า feedback ทั้ง 10 ข้อถูกแก้ไข หรือรับทราบและยอมรับ
      รายการที่ถูกเลื่อนเข้า backlog อย่างชัดเจนแล้ว
